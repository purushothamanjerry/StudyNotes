Foundation

What Is the TCP/IP Model?

The TCP/IP model (Transmission Control Protocol / Internet Protocol) is a four-layer framework describing how data is packaged, addressed, transmitted, routed, and received across interconnected networks. Unlike the OSI model — a theoretical reference — TCP/IP is the actual protocol suite running on every internet device today.

📡
Protocol Suite
TCP/IP is not one protocol — it's a family of hundreds (HTTP, DNS, TCP, UDP, IP, ARP, ICMP…) organized into four logical layers working in concert.
🌐
The Internet's Core
Every web request, email, video stream, and API call uses the TCP/IP stack. Understanding it means understanding how the internet fundamentally operates.
🔧
Practical Standard
Developed by DARPA in the 1970s, standardized in RFC 793 (TCP) and RFC 791 (IP). It replaced proprietary protocols and unified global communication permanently.
💡
Real-World Analogy: Think of sending a package internationally. You write contents (Application), seal with address label (Transport), the postal network routes through hubs (Internet), physical trucks/planes carry it (Network Access). Each layer does its specific job without knowing how the others work internally.
History

Origins & Evolution

Before TCP/IP, networks were isolated islands — IBM's SNA only talked to IBM systems, DECnet to DEC. DARPA needed a universal protocol for ARPANET that could survive nuclear attack and scale globally.

1969
ARPANET Created
First packet-switched network connecting 4 universities. Used NCP (Network Control Protocol) — a TCP/IP predecessor with no routing between different networks.
1974
TCP Paper Published
Vint Cerf and Bob Kahn publish "A Protocol for Packet Network Intercommunication" — the conceptual birth of TCP/IP, describing internetworking between disparate networks.
1981
IPv4 (RFC 791) + TCP (RFC 793)
32-bit addressing, 4.3 billion addresses. The suite is formally split: IP handles logical addressing and routing, TCP handles reliable delivery and ordering.
1983
Flag Day — Internet Birthday
January 1, 1983: all ARPANET hosts switch to TCP/IP simultaneously. This date is widely considered the birthday of the modern internet.
1998
IPv6 Standardized (RFC 2460)
128-bit addressing — 340 undecillion addresses — solving IPv4 exhaustion. Dual-stack deployment becomes the norm in the 2010s as IPv4 addresses run out.
Today
TCP/IP Everywhere
Powers 5G, cloud computing, IoT, satellite internet (Starlink), submarine cables. QUIC (HTTP/3) evolves transport but still rides atop IP — the model endures.

Core Design Principles
  • Layered abstraction — layers communicate only with adjacent layers
  • End-to-end principle — intelligence at endpoints, simple network core
  • Best-effort IP — simplifies routers; reliability is TCP's responsibility
  • Interoperability — any OS, hardware, vendor — all speak TCP/IP
  • Robustness — routes around damage (original ARPANET nuclear design goal)

HTTPS / TLS 1.3 Flow

1. ClientHello → TLS version, cipher suites, random bytes
2. ServerHello + Certificate ← chosen cipher, server cert
3. Key Exchange → ECDHE — forward secrecy
4. Finished ↔ both sides derive session keys
5. Encrypted Application Data ↔ AES-256-GCM
🔒
TLS 1.3 reduced handshake from 2 RTTs to 1 RTT and supports 0-RTT session resumption. Mandatory forward secrecy via ECDHE — no RSA key exchange allowed.

Layer 3

Transport Layer — TCP vs UDP

The Transport Layer creates logical end-to-end channels between applications. Port numbers identify which application on a host. Two major protocols: TCP for reliability, UDP for speed.

TCP Slow Start & AIMD

Start cwnd=1–10 MSS. Double each RTT (exponential growth) until ssthresh. Then linear +1 MSS/RTT (Congestion Avoidance). On loss: halve ssthresh, reset or reduce cwnd. On 3 duplicate ACKs: Fast Retransmit without restarting slow start.
TCP Buffer Tuning (Linux)
# BDP = Bandwidth × RTT
# 1Gbps × 50ms = ~6.25 MB needed
net.core.rmem_max = 134217728
net.core.wmem_max = 134217728
net.ipv4.tcp_rmem = 4096 87380 134217728
net.ipv4.tcp_wmem = 4096 65536 134217728
net.ipv4.tcp_congestion_control = bbr
net.core.default_qdisc = fq

TCP CUBIC (Linux default)

▼
Uses a cubic function for window growth. Aggressive on high-BDP links. More stable than Reno in high-speed datacenter environments. Default since Linux 2.6.19. Scales well up to 10 Gbps+ links.

BBR (Bottleneck Bandwidth RTT)

▼
Google's model-based algorithm (2016). Directly measures max bottleneck bandwidth and min RTT instead of reacting to loss. 2–25× faster on lossy links. Used by YouTube, GCP, Cloudflare. Enable: sysctl net.ipv4.tcp_congestion_control=bbr

QUIC Congestion (HTTP/3)

▼
HTTP/3 runs QUIC over UDP. Per-stream flow control eliminates head-of-line blocking. 0-RTT on reconnect — data sent in first packet. Connection ID migration — persists across network changes (WiFi → 5G). Uses NewReno or CUBIC by default, BBRv2 optionally.

TTL — Time To Live
Starts at 64 (Linux) or 128 (Windows). Each router decrements by 1. At TTL=0, packet is dropped and ICMP Type 11 "Time Exceeded" is returned. Prevents infinite loops. Traceroute exploits this by probing with TTL=1,2,3…
Fragmentation
If packet exceeds link MTU (typically 1500 bytes Ethernet), it fragments. DF bit (Don't Fragment) forces Path MTU Discovery instead. Fragments reassembled at destination only. IPv6 eliminated router fragmentation entirely.

Complete Journey — Step by Step

Step 1: DNS Resolution (before TCP)

▼
Browser checks local cache → OS cache → /etc/hosts → stub resolver sends UDP query to 8.8.8.8:53. Recursive resolver walks Root → .com TLD → Authoritative NS → returns A record IP. Cached per TTL. Total: 1–50ms (cached: <1ms).

Step 2: TCP 3-Way Handshake (1.5 RTTs)

▼
Client SYN (seq=x) → Server SYN-ACK (seq=y, ack=x+1) → Client ACK. ISN (Initial Sequence Number) is randomized (RFC 6528) to prevent prediction attacks. Connection ESTABLISHED, ready for TLS.

Step 3: TLS 1.3 Handshake (1 RTT)

▼
ClientHello → ServerHello + Certificate → both derive session keys via ECDHE. All HTTP data is now encrypted with AES-256-GCM. TLS 1.3 mandates forward secrecy — no static RSA key exchange.

Step 4: HTTP Request Encapsulation Down Stack

▼
"GET /" → TLS encrypts → TCP wraps (src:54321, dst:443, seq, flags) → IP wraps (src:192.168.1.10, dst:93.184.216.34, TTL:64, proto:6) → Ethernet wraps (src MAC, dst: Router MAC, EType:0x0800) → bits on wire.

Step 5: NAT & Routing Through Internet

▼
Home router: NAT translates 192.168.1.10:54321 → 203.0.113.5:49152. Each router strips old Ethernet, makes new frame for next hop, decrements TTL. BGP routes across ASes until reaching destination.

Step 6: Response & TCP Teardown

▼
Server sends HTTP 200 OK. ACKs confirm each segment. Teardown: FIN → ACK → FIN → ACK (4 messages). TIME_WAIT (2×MSL = 4 min) ensures stale packets expire. HTTP Keep-Alive reuses connections to avoid repeated handshakes.
Internet Layer Protocol

ARP — Address Resolution Protocol

Before a packet leaves, the OS needs the MAC of the next hop. ARP resolves IP→MAC on local network segments. It's the glue between Layer 2 (MAC) and Layer 3 (IP).

1. Check ARP cache — "Do I know MAC for 192.168.1.1?"
↓ Not found
2. ARP Request: Broadcast ff:ff:ff:ff:ff:ff — "Who has 192.168.1.1?"
↓ All devices receive; only .1.1 replies
3. ARP Reply: Unicast — "192.168.1.1 is at AA:BB:CC:DD:EE:FF"
↓ Cached
4. Cache entry: IP→MAC, TTL ~20 min (Linux default)
ARP Commands
# Linux — modern:
ip neigh show             # ARP/NDP table
ip neigh flush all        # flush cache
arping -A -I eth0 192.168.1.10 # gratuitous ARP
arp-scan --localnet       # discover all hosts

# Windows:
arp -a                    # show ARP cache
arp -d 192.168.1.1        # delete entry

# Wireshark filter:
arp.opcode == 1           # ARP requests only
arp.opcode == 2           # ARP replies only
⚠️
ARP Spoofing: Attacker sends fake ARP replies associating their MAC with gateway IP. All traffic routes through attacker (MITM). Defense: Dynamic ARP Inspection (DAI) on switches, static ARP entries, 802.1X, encrypted protocols (TLS).
Internet Layer

NAT & PAT — Address Translation

NAT allows multiple private devices to share one public IP. It was a critical band-aid for IPv4 exhaustion and remains ubiquitous in home routers, enterprises, and cloud environments.

NAT table maps the 5-tuple. Return packets hit router, NAT reverses translation, forwards to correct internal host. Private IPs never appear on the internet.

iptables NAT (Linux)
# Enable IP forwarding:
echo 1 > /proc/sys/net/ipv4/ip_forward

# Masquerade (dynamic PAT):
iptables -t nat -A POSTROUTING \
  -o eth0 -j MASQUERADE

# DNAT (Port Forward 8080→internal:80):
iptables -t nat -A PREROUTING \
  -p tcp --dport 8080 -j DNAT \
  --to-destination 192.168.1.100:80

# View conntrack table:
conntrack -L

# View NAT rules:
iptables -t nat -L -n -v

OSI vs TCP/IP

The OSI model has 7 theoretical layers; TCP/IP has 4 practical layers. Both are essential knowledge — OSI for conceptual clarity, TCP/IP for real-world implementation.

Why collapse layers? OSI was designed by ISO committee as a universal reference standard. TCP/IP was built by engineers solving real problems. The TCP/IP Application layer merges OSI's App+Presentation+Session because in practice these functions are tightly coupled in applications (TLS handles presentation+session within the app context).
Security Engineering

Security Vulnerabilities & Defenses

Every TCP/IP layer has attack vectors. Understanding them is essential for building secure networks and for security certifications (CISSP, CEH, OSCP).

SYN Flood (DoS/DDoS)

Layer: Transport | Send millions of SYNs, exhaust backlog queue. Defense: SYN Cookies (RFC 4987), rate limiting, anycast CDN absorption, net.ipv4.tcp_syncookies=1

ARP Spoofing / MITM

Layer: Network Access | Fake ARP replies poison cache, intercept traffic. Defense: Dynamic ARP Inspection (DAI), private VLANs, static ARP, 802.1X, encrypt everything with TLS.

IP Spoofing + Amplification

Layer: Internet | Forge source IP, exploit UDP amplification (DNS/NTP). Defense: BCP38 ingress filtering (ISP-level), uRPF (rp_filter=1), rate-limit UDP responses.

BGP Hijacking

Layer: Internet/Routing | Announce more-specific routes to steal traffic (Pakistan Telecom / YouTube 2008). Defense: RPKI (Route Origin Validation), IRR filters, MANRS compliance.

DNS Cache Poisoning

Layer: Application | Inject forged DNS responses, redirect to malicious IPs. Defense: DNSSEC (cryptographic signing), DoH/DoT (encrypted DNS), randomized ports and TXIDs (Kaminsky fix).

TCP Session Hijacking

Layer: Transport | Predict sequence numbers, inject packets into established sessions. Defense: Randomized ISNs (RFC 6528), TLS encryption, TCP MD5 (BGP sessions), RFC 5961 challenge ACK.

Linux Network Security Hardening (sysctl)
# /etc/sysctl.d/99-network-security.conf

# SYN flood protection:
net.ipv4.tcp_syncookies = 1
net.ipv4.tcp_max_syn_backlog = 4096

# Prevent IP spoofing (strict uRPF):
net.ipv4.conf.all.rp_filter = 1
net.ipv4.conf.default.rp_filter = 1

# Ignore ICMP redirects (MITM prevention):
net.ipv4.conf.all.accept_redirects = 0
net.ipv4.conf.all.send_redirects = 0

# Ignore broadcast pings (Smurf attack):
net.ipv4.icmp_echo_ignore_broadcasts = 1

# Ignore bogus ICMP errors:
net.ipv4.icmp_ignore_bogus_error_responses = 1

# Apply immediately:
sysctl -p /etc/sysctl.d/99-network-security.conf
Engineering

Performance Optimization

The key formula: Throughput = Window Size / RTT. Understanding Bandwidth-Delay Product (BDP) is critical for high-speed, high-latency links like satellite or trans-continental connections.

TCP Tuning for High-BDP Links
# BDP = Bandwidth × RTT
# 1 Gbps × 50ms RTT = 6.25 MB buffer

# Increase socket buffers:
net.core.rmem_max = 134217728   # 128MB
net.core.wmem_max = 134217728
net.ipv4.tcp_rmem = 4096 87380 134217728
net.ipv4.tcp_wmem = 4096 65536 134217728

# BBR congestion control:
net.ipv4.tcp_congestion_control = bbr
net.core.default_qdisc = fq

# TCP Fast Open (reduces handshake RTT):
net.ipv4.tcp_fastopen = 3

# Reduce TIME_WAIT sockets:
net.ipv4.tcp_fin_timeout = 30
net.ipv4.tcp_tw_reuse = 1

# Expand ephemeral port range:
net.ipv4.ip_local_port_range = 1024 65535

High Latency (RTT) Issues

▼
Causes: Physical distance, router queuing, congestion. Solutions: CDN edge caching, Anycast, QUIC 0-RTT, TCP Fast Open, HTTP/2 multiplexing, connection reuse (keep-alive), colocate services near users.

Low Throughput

▼
Causes: Small TCP window, congestion, packet loss causing CUBIC backoff, MTU mismatch causing fragmentation, NIC offloading disabled. Solutions: Increase TCP buffers, enable BBR, fix MTU (PMTUD), fix packet loss, jumbo frames in DC (MTU 9000).

Diagnosing Packet Loss

▼
mtr --report -c 100 8.8.8.8 — shows per-hop loss. Wireshark tcp.analysis.retransmission filter. ethtool -S eth0 | grep error for NIC errors. Causes: Oversubscribed links, faulty cables (CRC errors), wireless interference, buffer bloat.

DNS Performance

▼
Use local caching resolver (unbound, bind9). DNS prefetching in browsers. Anycast resolvers (1.1.1.1 typically < 8.8.8.8). Increase TTL for stable records. DoH for encrypted resolution. Monitor with dig +stats example.com.

Troubleshooting Guide

Systematic bottom-up troubleshooting using TCP/IP layers is the fastest path to resolution. Start at Layer 1 and work upward — eliminate possibilities methodically.

The Diagnostic Ladder

L1 Physical: Cable connected? Link light on? ip link show → state UP?
L1 Errors: CRC errors? ethtool -S eth0 | grep -i error
L2 ARP: Gateway reachable? ping 192.168.1.1 — ARP cache populated?
L3 IP: Correct IP/subnet/gateway? ip addr · ip route
L3 Internet: Can ping 8.8.8.8? If yes, routing works. If no, check default gateway or NAT.
L4 DNS: dig google.com — if IP pings work but domain doesn't → DNS issue.
L5 App: Port open? ss -tlnp — firewall? iptables -L

Career Prep

Interview Questions

Curated questions from CCNA-level to Senior Network/Cloud Engineer interviews at major tech companies. Click each to reveal the model answer.

Q: What is the difference between TCP and UDP?

▼
A: TCP is connection-oriented and reliable: 3-way handshake, sequence numbers, ACKs, and retransmission guarantee ordered, error-free delivery. UDP is connectionless and unreliable: no handshake, no ACKs, no retransmission — just send and forget. UDP has 8-byte fixed header vs TCP's 20-60 bytes, so it's faster with less overhead. Use TCP for correctness-critical apps (HTTP, SSH, banking). Use UDP for speed-critical apps where a late packet is useless (DNS, VoIP, live video, gaming).

Q: What is a subnet mask and why do we need it?

▼
A: A subnet mask separates the network portion of an IP address from the host portion. For 192.168.1.10/24, the mask 255.255.255.0 says: first 24 bits = network (192.168.1), last 8 bits = host (.10). Devices use this to determine if a destination is on the same local network (send via ARP directly) or needs to go through the default gateway (router). Without subnet masks, every device would broadcast to everyone.

Q: What happens when you type google.com in a browser?

▼
A: (1) Browser checks DNS cache → OS /etc/hosts → OS DNS cache. (2) If miss: UDP DNS query to resolver (8.8.8.8:53) → walks Root→.com TLD→Authoritative NS → returns A record IP. (3) Browser TCP SYN to IP:443 (3-way handshake). (4) TLS 1.3 handshake (1 RTT) → session keys. (5) HTTP/2 GET / encrypted. (6) Server HTTP 200 OK with HTML. (7) Browser parses, makes sub-requests for CSS/JS/images. (8) Render. Total first byte: typically 100–400ms depending on CDN.

Q: What is the TTL field and what happens when it reaches 0?

▼
A: TTL (Time To Live) is an 8-bit counter initialized to 64 (Linux) or 128 (Windows). Every router decrements TTL by 1. At TTL=0, the router drops the packet and returns ICMP Type 11 "Time Exceeded" to the source. This prevents infinite routing loops. You can identify the OS from ping TTL responses: 64→Linux, 128→Windows, 255→network devices. Traceroute exploits TTL by sending probe packets with TTL=1,2,3... to get a response from each hop.

Q: Explain TCP's 3-way handshake. Why are all 3 steps necessary?

▼
A: SYN (C→S): Client picks random ISN (x), proves it can SEND. SYN-ACK (S→C): Server picks ISN (y), acknowledges x+1 (proves it CAN RECEIVE client's data), sends y (proves it can SEND). ACK (C→S): Client acknowledges y+1, proves it CAN RECEIVE server's data. All 3 steps are needed because TCP is full-duplex — each direction must independently establish send/receive ability. With only 2 steps, server never knows the client can receive. ISN is randomized (RFC 6528) to prevent sequence prediction attacks.

Q: What is the difference between OSPF and BGP?

▼
A: OSPF is an IGP (Interior Gateway Protocol) — runs within one AS. Uses link-state database, floods LSAs, runs Dijkstra's SPF for shortest path. Fast convergence (sub-second with BFD). Metric: cost (bandwidth-based). Used inside enterprise/campus. BGP is an EGP (Exterior Gateway Protocol) — runs between ASes. Path-vector algorithm (carries full AS-PATH to prevent loops). Slow convergence (minutes). Policy-based (communities, MED, LOCAL_PREF). BGP connects ISPs and is "the protocol of the internet" — ~900K routes in the global table.

Q: How does DHCP snooping work and why is it important?

▼
A: DHCP snooping is a Layer 2 switch security feature. Ports are classified as "trusted" (uplink to legitimate DHCP server) or "untrusted" (access ports). On untrusted ports, the switch: drops DHCP Offer/Ack messages (blocks rogue DHCP servers), validates client messages (prevents spoofing), and builds a binding table (IP↔MAC↔Port↔VLAN) used by Dynamic ARP Inspection and IP Source Guard downstream. Critical in multi-tenant environments and public Wi-Fi networks.

Q: What is the Bandwidth-Delay Product and why does it matter?

▼
A: BDP = Bandwidth × RTT. It represents how much data can be "in flight" on a link at maximum utilization. Example: 1 Gbps link with 100ms RTT = 12.5 MB in-flight. TCP's window must be at least this large to fully saturate the link. With the default 64KB window (no scaling), you can only achieve: 64KB / 0.1s = 5.12 Mbps — even on a 1Gbps link! Window Scaling (RFC 1323) enables windows up to 1GB. On satellite links (600ms+ RTT), BDP-aware tuning is essential.

Q: Explain how QUIC improves upon TCP+TLS and its trade-offs.

▼
A: QUIC (RFC 9000) runs over UDP, combining transport + TLS 1.3. Improvements: (1) 1 RTT handshake vs 3 RTTs for TCP+TLS 1.2; (2) 0-RTT resumption — send data in first packet; (3) No head-of-line blocking — each HTTP stream is independent (TCP blocks all streams when one packet is lost); (4) Connection migration — Connection ID persists when switching from WiFi to 5G (TCP would drop and reconnect); (5) Encrypted headers by default. Trade-offs: UDP may be blocked by firewalls, middleboxes need updates, higher CPU overhead, OSID inspection is harder, loss characteristics on UDP differ from TCP.

Q: What happened technically in the Facebook BGP outage of October 2021?

▼
A: A routine maintenance command on Facebook's backbone routers triggered a configuration audit that found an error. The audit caused BGP peering sessions to withdraw all Facebook-originated prefixes from the global routing table. Every internet router stopped advertising paths to Facebook's IP blocks. Facebook's DNS nameservers (authoritative for facebook.com) are inside Facebook's network — now unreachable. DNS resolvers worldwide returned SERVFAIL. Since internal tools (for engineers to fix the issue) also ran inside the now-isolated network, physical on-site access was required. Total outage: ~6 hours. Lesson: Never allow a single automated command to withdraw all routes; require multiple approval steps and test on canary routers.

Q: How does ECMP routing work and what problems can it cause?

▼
A: ECMP (Equal-Cost Multi-Path) load-balances across multiple paths with equal routing cost. Routers hash the 5-tuple (src IP, dst IP, src port, dst port, protocol) to consistently map a TCP session to one path — ensuring packet ordering within a flow. Problems: (1) Asymmetric routing — ACKs may take a different path, breaking stateful firewalls; (2) NAT + ECMP — NAT state exists on only one router, return packets may hit the wrong one; (3) Large flows get one path, small flows get another — poor load distribution (polarization). In spine-leaf DC architectures, ECMP is fundamental with 4–64 equal-cost paths to maximize bandwidth utilization.

Scenario: Users can ping 8.8.8.8 but can't browse websites. What's wrong?

▼
Analysis: IP routing works (can reach internet). Issue is DNS resolution. Steps: (1) Try ping google.com — probably fails with "unknown host". (2) Check /etc/resolv.conf — is nameserver set? (3) Test bypass: dig @8.8.8.8 google.com — if works, local resolver is broken. (4) Check if port 53 blocked: nc -u 8.8.8.8 53. (5) Check daemon: systemctl status systemd-resolved. Fix: update /etc/resolv.conf or restart resolver service, ensure port 53 egress is allowed.

Scenario: Throughput between servers is far below link capacity. Diagnose.

▼
Analysis: Throughput = Window / RTT. (1) Check RTT: ping -c 100 server. (2) Check window scaling: sysctl net.ipv4.tcp_window_scaling. (3) Run parallel streams: iperf3 -c server -P 8 — if parallel is fast, single-flow bottleneck (BDP issue). (4) Check packet loss: Wireshark retransmission filter or mtr --report. (5) Increase buffers per performance tuning. (6) Check NIC offloading: ethtool -k eth0. (7) Check CPU bottleneck (encryption on TLS-heavy workloads).

Scenario: Service unreachable from some locations but not others. Debug.

▼
Analysis: Likely routing/BGP issue. (1) From affected location: can they ping server IP? If yes, L3 works — it's application/firewall. If no, routing issue. (2) Compare traceroutes from affected vs unaffected locations — where do paths diverge? (3) Check global BGP table: bgpview.io or route-views.org — is the prefix being announced? Specific enough? (4) Check for geo-based ACLs or anycast misconfiguration. (5) Use RIPE Atlas probes for distributed testing. (6) Check if the issue is IPv4 vs IPv6 (try both explicitly).

Mini Quiz

Click an option to reveal the answer and explanation. Test yourself before your next interview or certification exam.

1. Which TCP flag combination initiates the 3-way handshake?
✅ B — SYN only. The first handshake packet has only SYN set (seq=x). Server responds with SYN+ACK (seq=y, ack=x+1). Client completes with ACK (ack=y+1). FIN+ACK is used during connection teardown.
2. What is the default TTL on Linux systems?
✅ C — 64. Linux default is 64. Windows default is 128. Cisco IOS is 255. Check on Linux: cat /proc/sys/net/ipv4/ip_default_ttl. You can fingerprint an OS from ping TTL responses!
3. Which protocol does traceroute use by default on Linux?
✅ A — UDP. Linux traceroute sends UDP probes to high-numbered ports (33434+) by default. Windows tracert uses ICMP Echo Requests. Use traceroute -T -p 80 for TCP to bypass firewalls that block UDP.
4. A /28 subnet has how many usable host addresses?
✅ D — 14. /28 = 32-28 = 4 host bits = 2⁴ = 16 total addresses. Subtract network address and broadcast address = 14 usable hosts. Subnet mask: 255.255.255.240. Wildcard: 0.0.0.15.
5. ICMP Type 3, Code 3 means what?
✅ B — Type 3 Code 3 = Destination Port Unreachable. Type 3 = Destination Unreachable family. Code 0=Net, 1=Host, 3=Port. Port Unreachable is what you get when you connect to a closed UDP port. Type 8=Echo Request, Type 11=Time Exceeded (TTL=0, traceroute), Type 5=Redirect.

🏗️
The Layered Model Is Everything
Separation of concerns is what allows the internet to scale to billions of devices. Application doesn't know about routing. IP doesn't know about TCP reliability. Each layer does one thing well.
📦
Encapsulation Is the Core Mechanism
Data flows DOWN the stack (application → bits) adding headers, and UP the stack stripping them. MAC addresses change at every hop; IP addresses stay the same end-to-end (modulo NAT).
🔄
Choose TCP vs UDP Consciously
TCP for correctness (banking, files, APIs). UDP for latency-sensitive apps where a late retransmit is worse than loss (VoIP, gaming, live video). QUIC combines reliability with UDP performance.
🔒
Security Is Retrofitted, Not Built-In
TCP/IP was designed for a trusted academic network. ARP has no auth. IP source is spoofable. BGP has no origin validation by default. Modern internet security patches these gaps layer by layer.
⚡
Performance = Window Size / RTT
TCP throughput is fundamentally limited by BDP. Latency is dominated by physics, queuing, and handshake overhead. BBR, QUIC, TFO, and CDN anycast each attack different latency components.
🌐
The Internet Survives Through Redundancy
BGP reroutes around failures. TCP retransmits lost packets. DNS has multi-level caching. IPv6 was designed to fix IPv4 exhaustion. Every layer anticipates failure and handles it gracefully.
