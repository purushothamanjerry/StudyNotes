Section 01

What is the OSI Model?

Understanding the conceptual framework behind all modern networking.

The Open Systems Interconnection (OSI) model is a conceptual framework developed by the International Organization for Standardization (ISO) in 1984. It defines how different network protocols should interact to allow communication between devices, regardless of underlying architecture or vendor.

💡 The Core Problem OSI Solves
Before OSI, every vendor created proprietary networking systems. IBM's SNA couldn't talk to Digital's DECnet. The OSI model was created to standardize networking so any vendor's equipment could interoperate with any other — the foundation of the modern internet.

Real-World Analogy: International Postal Mail

📝 You write a letter (Layer 7)
Your content — the actual message. Like HTTP request data or an email body.
🌐 You translate it (Layer 6)
Converting to a language the recipient understands — encoding, encryption.
✉️ You address an envelope (Layer 4/5)
Packaging and addressing, ensuring delivery tracking.
🚗 Postal truck delivers it (Layer 3/2)
The infrastructure that physically moves the packet from A to B.
🛣️ Roads and bridges (Layer 1)
The actual physical medium — copper, fiber, radio waves.
⚠️ OSI vs TCP/IP
The OSI model is a reference model — the internet actually uses the TCP/IP model (4 layers). OSI layers 5, 6, and 7 collapse into TCP/IP's "Application" layer. However, engineers still use OSI terminology daily for precision — saying "Layer 4 firewall" vs "Layer 7 WAF" conveys exact meaning.

Layer 7: Application Layer

The Application Layer is closest to the end user. It provides network services directly to applications — not to users. The user interacts with software; the software interacts with this layer.

What it does:

  • Resource sharing and remote file access
  • Email, file transfer, web browsing
  • Directory services (LDAP, Active Directory)
  • Network management (SNMP)
  • Provides interface for apps to use network services

Key Protocols & Ports:

  • HTTP/HTTPS — 80/443, web & REST APIs
  • FTP/SFTP — 21/22, file transfer
  • SMTP — 25/587, email sending
  • DNS — 53 UDP/TCP, name resolution
  • DHCP — 67/68, IP assignment
  • SSH — 22, secure remote shell
  • SNMP — 161/162, device monitoring
  • LDAP — 389/636, directory services
Engineering Detail: L7 ≠ The Application
Layer 7 does NOT include the application itself (e.g., Chrome). It defines the protocols applications use. A WAF (Web Application Firewall) operates at L7 — it inspects HTTP headers, cookies, and body content that lower-layer firewalls cannot see.
6
Presentation

PDU: Data
TLS/SSLJPEGASCII

Layer 6: Presentation Layer

Acts as a translator between application and transport layers. Ensures data sent by one system can be read by another regardless of encoding differences.

Three Core Functions:

  • Translation: ASCII↔EBCDIC, Unicode↔UTF-8
  • Encryption/Decryption: TLS/SSL conceptually lives here
  • Compression: gzip, deflate, JPEG, MP3

Real-world Examples:

  • HTTPS — TLS encryption at this layer
  • JPEG encoding before image transmission
  • HTTP Content-Encoding: gzip
  • Base64 encoding for email attachments
Where Does TLS Really Belong?
TLS spans L4–L7: it runs on top of TCP (L4), encrypts Application data (L7), and handles encoding (L6). For OSI discussion, the encryption function = L6. In TCP/IP model, TLS sits "between" Transport and Application.
5

PDU: Data

Layer 5: Session Layer

Manages dialogues (sessions) between computers — establishes, maintains, synchronizes, and terminates sessions between applications.

Session Management:

  • Establishment: Setting up communication channel
  • Maintenance: Keeping session alive via heartbeats
  • Synchronization: Checkpoints for resumable transfers
  • Termination: Orderly session shutdown

Dialog Modes:

  • Simplex: One-way only (broadcast TV)
  • Half-duplex: Both ways, not simultaneously (walkie-talkie)
  • Full-duplex: Both ways simultaneously (phone call)
Real-World: SIP for VoIP
SIP (Session Initiation Protocol) sets up VoIP calls at L5 — it negotiates the session, codec, and endpoints. Once established, RTP carries the actual audio data. Without L5 session management, there's no concept of "who's calling whom."

Layer 4: Transport Layer

Responsible for end-to-end communication between processes. Segments data, handles error recovery, flow control, and reassembly at destination.

TCP — Transmission Control Protocol

  • Connection-oriented: 3-way handshake before data
  • Reliable delivery: Sequence numbers, ACKs, retransmission
  • Flow control: Receive window prevents buffer overflow
  • Congestion control: Slow start, AIMD, Cubic/BBR
  • Ordered delivery: Reorders segments at receiver

UDP — User Datagram Protocol

  • Connectionless: No handshake, fire-and-forget
  • No reliability: Packets may be lost/reordered
  • Low latency: No handshake overhead
  • Minimal header: 8 bytes vs TCP's 20+
  • Best for: DNS, DHCP, VoIP, gaming, streaming, NTP

UDP Header (8 bytes):

Source Port (16 bits)
Destination Port (16 bits)
Length (16 bits)
Checksum (16 bits)
When to Choose UDP
Use UDP when: (1) Speed matters more than reliability — live video won't retransmit dropped frames; (2) Application handles reliability itself — DNS retries on its own; (3) Broadcast/multicast needed — UDP supports these, TCP doesn't; (4) Packet loss is acceptable — voice jitter is better than silence from TCP retransmit.

QUIC — Quick UDP Internet Connections (HTTP/3)

QUIC (RFC 9000) runs over UDP but provides TCP-like reliability with major improvements:

  • 0-RTT handshake: Resumes connections without round trips
  • Built-in TLS 1.3: Encryption at the transport layer
  • Stream multiplexing: No head-of-line blocking
  • Connection migration: Survives IP changes (WiFi→4G)
QUIC in Production
Google routes ~30% of internet traffic via QUIC/HTTP3. YouTube, Google Search, Gmail all use HTTP/3 by default. Connection establishment is ~2× faster than TLS 1.2 over TCP.

Provides logical addressing and routing. Unlike L2 which works within a single network, L3 routes packets across multiple networks using IP addresses.

Routing Protocols:

  • OSPF — Link-state, Dijkstra, enterprise LANs
  • BGP — Path-vector, internet backbone
  • EIGRP — Cisco proprietary, DUAL algorithm
  • RIP — Distance vector, max 15 hops (legacy)
  • IS-IS — Link-state, ISP core networks

Key Concepts:

  • TTL: Prevents loops, decrements each hop. Default: 64 (Linux), 128 (Windows)
  • Fragmentation: Large packets split at MTU boundary
  • ICMP: ping, traceroute, unreachable messages
  • NAT: Translates private↔public IPs at L3/L4
  • Protocol field: 6=TCP, 17=UDP, 1=ICMP, 89=OSPF

Layer 2: Data Link Layer

Handles node-to-node data transfer on the same network segment. Wraps packets into frames, adds MAC addresses, and detects transmission errors.

Two Sublayers (IEEE 802):

  • LLC (Logical Link Control): Error control, identifies L3 protocol in use
  • MAC (Media Access Control): Hardware addressing, medium access control (CSMA/CD for Ethernet, CSMA/CA for WiFi)

Ethernet Frame:

  • Preamble: 7 bytes, sync (10101010...)
  • SFD: 1 byte, start frame delimiter
  • Dst MAC: 6 bytes
  • Src MAC: 6 bytes
  • EtherType: 2 bytes (0x0800=IPv4)
  • Payload: 46–1500 bytes
  • FCS: 4 bytes, CRC-32

MAC Address Deep Dive:

48 bits (6 bytes) in format XX:XX:XX:XX:XX:XX. First 3 bytes = OUI (vendor). Last 3 bytes = device-specific. Example: 00:1A:2B = Apple Inc.

L2 Devices:

  • Switch: Builds CAM table (MAC→port), forwards frames only to correct port. Separate collision domain per port
  • VLAN (802.1Q): Tags frames with 12-bit VLAN ID. Logically separates networks on same switch
  • STP (802.1D): Blocks redundant links to prevent broadcast storms

PDU: Bit

Layer 1: Physical Layer

Responsible for actual physical transmission of raw bits. Defines electrical voltages, pin layouts, cable specs, and signaling methods.

Transmission Media:

  • Copper UTP: Cat5e (1Gbps/100m), Cat6 (10Gbps/55m), Cat6A (10Gbps/100m), Cat8 (40Gbps/30m)
  • Fiber Optic: SMF single-mode 100km+, MMF multi-mode 550m
  • Wireless: 2.4GHz/5GHz WiFi, mmWave 60GHz

Physical Layer Devices:

  • Hubs: Legacy, broadcasts all signals to all ports (collision domain)
  • Repeaters: Amplify/regenerate signals to extend range
  • Modems: Digital↔analog conversion
  • Transceivers: SFP, SFP+, QSFP28 — convert signals
Binary Signaling: How a bit becomes voltage
1000BASE-T (Gigabit Ethernet) uses PAM-5 (5 voltage levels: -2,-1,0,+1,+2V) over 4 wire pairs. Each symbol encodes 2 bits, achieving 1Gbps. In fiber: photon = 1, no photon = 0. Modern 400G uses 64-QAM coherent modulation.
Section 03

Data Encapsulation

Each layer wraps data with its own header as it travels down — and strips it going up.

When data travels down the OSI model (sending), each layer adds its own header — this is encapsulation. When received, each layer strips its header — this is de-encapsulation.

💡 PDU Names Matter in Interviews
Each layer has a specific PDU name: L7=Data, L4=Segment(TCP)/Datagram(UDP), L3=Packet, L2=Frame, L1=Bit. Saying "a TCP segment is encapsulated in an IP packet, which is encapsulated in an Ethernet frame" signals deep mastery.

ARP Resolution Process

Host wants to send to 192.168.1.1 (router)
Check ARP cache: arp -a — is 192.168.1.1 known? If yes, use cached MAC. Cache TTL: ~20 min (Linux), ~30s–4min (Cisco).
ARP Broadcast: "Who has 192.168.1.1?"
Sends ARP Request: src MAC=own, dst MAC=FF:FF:FF:FF:FF:FF (broadcast). EtherType = 0x0806. All devices on segment receive this frame.
Target responds: "I have it — my MAC is aa:bb:cc:dd:ee:ff"
ARP Reply sent unicast back. Contains router's MAC address. Now the sender can build a valid Ethernet frame.
ARP Cache Updated
IP→MAC mapping cached. Subsequent packets skip ARP. Even though destination IP is on the internet, the Ethernet dst MAC is the router's MAC — it handles L3 forwarding.

Spanning Tree Protocol (L2 Loop Prevention)

Why STP is Critical
Without STP, a loop in L2 switches causes broadcast storms — frames circulate forever consuming 100% bandwidth in milliseconds. STP blocks redundant links and creates a loop-free topology while maintaining redundancy for failover.
Cisco STP States
STP 802.1D Port States:
  BLOCKING    → Receives BPDUs, no forwarding (loop prevention)
  LISTENING   → 15s: selects root bridge, no forwarding
  LEARNING    → 15s: builds MAC table, no forwarding
  FORWARDING  → Normal, forwards frames
  DISABLED    → Admin shutdown

RSTP (802.1w) improvements:
  DISCARDING  → Combines Blocking + Listening
  LEARNING    → Same
  FORWARDING  → Same

Convergence: STP = 30-50s | RSTP = 1-2s
Section 07

TCP Congestion Control

The algorithms that prevent the internet from collapsing under load.

Slow Start & Congestion Avoidance (TCP Reno)

+

Slow Start: cwnd starts at 1 MSS (1460 bytes), doubles every RTT until reaching ssthresh. Exponential growth.

Congestion Avoidance: After ssthresh, cwnd grows by 1 MSS per RTT (linear — AIMD).

On packet loss (timeout): ssthresh = cwnd/2, cwnd = 1, restart slow start.

On triple duplicate ACK (fast retransmit): ssthresh = cwnd/2, cwnd = ssthresh, skip slow start.

TCP Cubic (Linux Default since kernel 2.6.19)

+

Uses cubic function: W(t) = C(t - K)³ + W_max where K = time to grow back to W_max after congestion.

RTT-independent — unlike Reno, high-BDP (bandwidth-delay product) networks like trans-Pacific fiber get full throughput.

Check congestion control
sysctl net.ipv4.tcp_congestion_control
# cubic (default Linux)
cat /proc/sys/net/ipv4/tcp_available_congestion_control
# cubic reno bbr

BBR — Bottleneck Bandwidth & RTT (Google, 2016)

+

BBR models the network path by measuring BtlBw (max bottleneck bandwidth) and RTprop (min propagation delay). Sends at rate that fills the pipe without filling buffers.

Achieves ~95% link utilization vs ~60-70% for Cubic in lossy networks. Used by YouTube (2× throughput improvement reported).

Enable BBR on Linux
echo "net.core.default_qdisc=fq" >> /etc/sysctl.conf
echo "net.ipv4.tcp_congestion_control=bbr" >> /etc/sysctl.conf
sysctl -p
# Verify:
sysctl net.ipv4.tcp_congestion_control

TCP Flow Control: Sliding Window

+

Prevents fast sender overwhelming slow receiver. Receiver advertises rwnd in every ACK.

Sliding Window Example
Receiver buffer: 65535 bytes

Client sends 8000 bytes
Server ACK: window = 57535 (65535 - 8000)

App reads 20000 bytes from buffer:
Server ACK: window = 77535 (buffer freed, expands)

Zero Window: window=0 → sender stops completely
Window Update: rwnd > 0 → sender resumes

Section 11

Security at Each Layer

Attacks, vulnerabilities, and defenses organized by OSI layer.

L2Data LinkARP Spoofing / ARP Poisoning
Attacker sends gratuitous ARP replies mapping their MAC to the gateway IP. All hosts on the segment poison their cache — traffic meant for the gateway flows to the attacker (MITM). Defense: Dynamic ARP Inspection (DAI) on Cisco switches, static ARP for critical hosts, 802.1X port authentication, arpwatch monitoring.
L2Data LinkMAC Flooding / CAM Table Overflow
Floods switch with thousands of fake-MAC frames, filling the CAM table. Switch falls back to hub behavior (broadcasts all frames to all ports) enabling passive sniffing. Defense: Port security (max MACs per port), 802.1X, storm control, private VLANs.
L3NetworkIP Spoofing & DDoS Amplification
Attacker forges source IP to amplify traffic to a victim. DNS amplification: 60B query → 3000B response (50× amplification). NTP monlist: 234B → 468×482B responses. Defense: BCP38/uRPF (drop packets with spoofed source IPs at network edge), rate limiting UDP amplifiers, disabling vulnerable services.
L4TransportSYN Flood Attack
Millions of TCP SYN packets with spoofed IPs. Server allocates state per SYN, exhausting the backlog queue. Legitimate connections time out. Defense: SYN cookies (state in sequence number — no allocation until ACK), SYN rate limiting, TCP Intercept, connection limits per IP.
L7ApplicationSQL Injection / XSS / SSRF
HTTP-level attacks bypassing all lower-layer defenses since traffic is valid HTTPS. SQL injection exploits database queries. XSS injects scripts into responses. SSRF abuses server requests to reach internal services. Defense: WAF, parameterized queries, input validation, CSP headers, mTLS for internal services.

Section 12

Performance Optimization

Tuning each layer for maximum throughput and minimum latency.

L1/L2: Physical & Link Tuning

+
  • Full duplex: Hard-coding one side causes duplex mismatch → 30–50% performance loss via collisions
  • Jumbo frames (MTU 9000): Reduces CPU overhead 30–40% for storage traffic (iSCSI, NFS)
  • LACP bonding: Bond multiple 10G links for 20G/40G server uplinks
  • Cable quality: Cat6A for 10G at full 100m. Fiber eliminates EMI concerns entirely

L3: Routing Optimization

+
  • ECMP: Equal-cost multi-path load-balances across ISP links (BGP or OSPF)
  • IP Fast Reroute (IPFRR): Pre-calculated backup routes activate in <50ms vs OSPF convergence 200ms–2s
  • Path MTU Discovery: Fix PMTUD black holes that cause TCP to never negotiate correct MSS
  • Hardware ASICs: Cisco Silicon One, Juniper Q5 — 100Tbps forwarding in silicon, not CPU

L7: Application Performance

+
  • HTTP/2 multiplexing: Eliminates head-of-line blocking. 2–3× faster page loads vs HTTP/1.1
  • HTTP/3 / QUIC: 0-RTT for returning users, no TCP handshake. Critical for high-latency mobile
  • TLS 1.3: 1-RTT handshake (vs 2-RTT for TLS 1.2). Removes RC4, SHA1, CBC vulnerabilities
  • CDN edge caching: Serve from POP within 20ms. Reduces origin traffic 90%+ for static assets
  • Keep-Alive: Persistent TCP connections for multiple HTTP requests — eliminates per-request handshake
Section 13

Troubleshooting Guide

Systematic bottom-up diagnosis of network problems.

The Golden Rule: Troubleshoot Bottom-Up
Start at Layer 1 (physical) and work upward. You cannot have L3 issues if L2 is broken. Never blame the application until all lower layers are verified healthy.
1
Layer 1 — Is the link up?
Check LEDs: green = link, amber = activity. Run ip link show eth0 → should be "UP". If "DOWN": cable unplugged, faulty SFP, wrong cable type. Run ethtool eth0 → verify "Link detected: yes" and speed/duplex match on both ends.
2
Layer 2 — Right VLAN? ARP working?
Run ip neigh show — can you see the gateway MAC? No entry = wrong VLAN, DHCP issue, or ARP being blocked. On Cisco: show mac address-table — is host MAC visible on correct port?
3
Layer 3 — Can you ping the gateway? The internet?
ping 192.168.1.1 — if fails: wrong default gateway, gateway down, ICMP blocked. ping 8.8.8.8 — if fails but gateway ok: routing issue, NAT broken, ISP problem. traceroute 8.8.8.8 — where does path stop?
4
Layer 4 — Are ports reachable?
nc -zv host 443 — TCP connectivity test. ss -tunapl — what's listening? Check firewall: iptables -L -n -v or nft list ruleset.
5
Layer 7 — Is the application responding?
curl -v https://host/api — full HTTP + TLS detail. dig host A — DNS resolving? Check application logs: nginx/apache error.log, systemd journal, app server logs.

Section 14

Interview Q&A

Click "Show Answer" to reveal. Organized by difficulty.

Beginner
Q: What is the OSI model and why is it important?
▶ Show Answer
The OSI (Open Systems Interconnection) model is a 7-layer conceptual framework standardizing how network systems communicate. It matters because: (1) Enables vendor interoperability; (2) Provides a common engineering language; (3) Isolates problems to specific layers; (4) Guides protocol design. Without OSI, Cisco switches couldn't talk to Juniper routers or Windows servers couldn't reach Linux web servers.
Beginner
Q: What's the difference between TCP and UDP?
▶ Show Answer
TCP: connection-oriented (3-way handshake), reliable (ACKs + retransmission), ordered delivery, flow and congestion control, 20-byte header. Use for: HTTP, SSH, SMTP — anything requiring guaranteed delivery. UDP: connectionless, unreliable, no ordering, no flow control, 8-byte header. Use for: DNS (application retries), VoIP (jitter better than silence), gaming, streaming, NTP. Key insight: UDP's "unreliability" is a feature — it's faster and lets the application decide how to handle loss.
Intermediate
Q: What's the difference between a router, switch, and hub?
▶ Show Answer
Hub (L1): Repeats electrical signals to ALL ports. No intelligence, one collision domain. Obsolete. Switch (L2): Builds CAM table of MAC addresses, forwards frames only to correct port. Separate collision domain per port. Supports VLANs and STP. Router (L3): Routes packets between different IP networks using routing table. Makes decisions based on destination IP. Performs NAT. Separates broadcast domains. L3 Switch: Has both switching and routing capability — faster than separate devices but same function. Key: switch sees MACs, router sees IPs.
Intermediate
Q: Explain the TCP 3-way handshake. What if the server isn't listening?
▶ Show Answer
3-way handshake: (1) Client sends SYN(seq=X) — "I want to connect, my sequence starts at X". (2) Server responds SYN-ACK(seq=Y, ack=X+1) — "OK, my sequence is Y, I acknowledge X". (3) Client sends ACK(ack=Y+1) — "Connection established". If server is NOT listening: immediately responds with RST — you see "Connection refused" instantly. If server SYN backlog is full: packet is silently dropped, client retransmits with exponential backoff (1s, 2s, 4s, 8s...). SYN cookies eliminate the backlog allocation to defend against SYN floods.
Intermediate
Q: What is ARP and what security risks does it have?
▶ Show Answer
ARP maps IP addresses to MAC addresses on local networks. Broadcasts "Who has IP X?" — owner replies with their MAC. ARP is stateless and unauthenticated, creating risks: (1) ARP spoofing: attacker sends unsolicited ARP replies poisoning caches — all traffic for gateway routes to attacker (MITM); (2) ARP flooding: fills switch CAM table forcing broadcast mode; (3) Gratuitous ARP abuse: devices announce their own IP/MAC, can be exploited. Mitigations: Dynamic ARP Inspection (DAI) validates ARP against DHCP snooping table, 802.1X ensures only authenticated ports can send ARP, static ARP for critical servers.
Advanced
Q: You see intermittent timeouts to a remote server. Walk through your diagnosis.
▶ Show Answer
Bottom-up approach: L1: Check interface errors — ethtool eth0, Cisco "show interface" for CRC/input errors → physical/cable issue. L2: Check for duplex mismatch. L3: Run mtr to identify which hop has packet loss/high latency. Loss at hop 5 means problem between 4 and 5. Check ECMP flows — PMTUD issues can cause asymmetric routing. L4: Check TCP retransmit rate: ss -ti. Check if issue is consistent (routing) or intermittent (congestion/ECMP). L7: curl with timing: curl -w "time_total: %{time_total}" — is it connect time or response time? Tools: mtr (path+loss), tcpdump (capture the actual timeout), ss (socket buffer state). Distinguish: network RST (fast failure) vs application timeout (connection established but app slow).
Advanced
Q: Explain BGP and why it's called the "glue of the internet".
▶ Show Answer
BGP (RFC 4271) is the exterior gateway protocol routing between Autonomous Systems (AS) — independent networks with their own IP prefixes. Path-vector protocol: each UPDATE carries the full AS path to prevent loops. BGP peers exchange OPEN, UPDATE, KEEPALIVE, NOTIFICATION messages over TCP port 179. Policy-driven: AS path length, local preference, MED, and communities allow fine-grained routing control. Called internet's "glue" because ~75,000 ASes exchanging ~900,000 IPv4 prefixes rely on it. BGP hijacking (Pakistan Telecom vs YouTube 2008, Rostelecom hijacking Google/AWS 2020) occurs when an AS announces someone else's prefix. RPKI provides cryptographic route origin validation as defense.
Scenario
Q: Your HTTPS site returns 502 errors intermittently. How do you investigate?
▶ Show Answer
1) Check LB logs — 502 = LB received invalid response from upstream server. 2) Determine if all backends or just some — partial = specific server issue (OOM, crash). 3) On affected server: check process (systemctl status), app error logs, CPU/memory (top, free). 4) TCP connectivity: nc -zv backend:8080 — is backend accepting connections? 5) TLS: check certificate expiry (openssl s_client -connect host:443). 6) Time correlation: high load? Check LB connection pool exhaustion, backend concurrency limits. 7) Network: ping/mtr from LB to backend. 8) Resolution: app crash → restart; connection pool exhaustion → tune LB timeout/pool size; OOM → scale or fix memory leak; cert expiry → renew certificate.

Section 18

Key Takeaways

The most important OSI concepts to carry forward.

🏗️ Abstraction is the Point
TCP doesn't know if it runs over Ethernet or WiFi. HTTP doesn't know if TCP or QUIC carries it. This layered abstraction makes the internet scalable to billions of devices.
📦 Overhead is Real
Every packet carries at minimum: Ethernet (14B) + IP (20B) + TCP (20B) = 54 bytes overhead per 1460 bytes of payload. Jumbo frames reduce this ratio for bulk transfers.
🔍 Debug Bottom-Up Always
A flapping network cable (L1) looks like an application outage (L7). Never blame the application until L1–L6 are verified clean.
🛡️ Security is Per-Layer
ARP spoofing = L2. SYN flood = L4. SQL injection = L7. Defenses must match attack layer — a stateful firewall can't stop ARP spoofing.
⚡ OSI is Reference, TCP/IP is Reality
The internet runs 4-layer TCP/IP, not 7-layer OSI. But OSI terminology is the universal language — "L7 WAF" vs "L4 NLB" is instantly understood by every network engineer worldwide.
🔄 The Model Evolves
QUIC blurs L4/L7. VXLAN tunnels L2 over L3. eBPF operates at the kernel networking stack. SD-WAN virtualizes L1–L3. OSI is a framework — real protocols never fit perfectly into neat boxes.
