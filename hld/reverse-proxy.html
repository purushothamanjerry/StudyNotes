Reverse Proxy

The silent guardian between your clients and backend services. Learn what a reverse proxy is, how it differs from a load balancer and forward proxy, what it actually does at each layer, and how Nginx and Traefik are configured in real production systems — with every line of config explained.

Introduction

What is a Reverse Proxy?

A reverse proxy is a server that sits between clients and one or more backend servers. It receives all incoming requests, forwards them to the appropriate backend, then returns the backend's response to the client. From the client's perspective, they are always talking to the proxy — they never know the backend servers exist.

The word "reverse" distinguishes it from a forward proxy (which sits on the client side, forwarding outgoing requests on behalf of clients). A reverse proxy acts on behalf of the server, not the client.

🏢 Analogy — Corporate Receptionist

When you visit a large company, you don't walk directly to any employee's desk. You speak to the receptionist (reverse proxy) at the entrance. The receptionist verifies your identity, notes your purpose, and directs you to the right person or department. You never know whether the person is on Floor 3 or Floor 12 — and if they're away, the receptionist handles it. The employees' actual desks (backend servers) are hidden and protected behind the receptionist's desk.

Your website runs on a Node.js app on port 3000. You want users to access it on port 80 (HTTP) and 443 (HTTPS). You don't expose the Node.js server directly — instead, you put Nginx in front.

Nginx listens on port 80/443, handles SSL, and forwards requests to your Node.js app on port 3000. Users only ever see Nginx. Your app's internal port is hidden. This is a reverse proxy in its simplest form.

A reverse proxy operates at Layer 7 (HTTP) or Layer 4 (TCP). It decouples the client-facing network interface from the backend implementation. This separation enables: independent scaling of the proxy tier vs application tier, transparent backend replacement (deploy new app version without client-side changes), centralised cross-cutting concerns (auth, rate limiting, logging), and protocol translation (HTTP → gRPC, HTTP/1.1 → HTTP/2).

Modern reverse proxies like Envoy and Traefik support service mesh integration via the sidecar pattern — every service pod runs a proxy sidecar that handles all inbound/outbound traffic, enabling mTLS, traffic shaping, and observability without any code changes.

Core Concepts

Forward Proxy vs Reverse Proxy — Critical Distinction

These two are constantly confused. The key difference is whose side the proxy acts on.

// Forward Proxy vs Reverse Proxy — Who Does It Serve?
🔀 Forward Proxy
Serves: Clients (hides client identity)
Who uses it: Client configures it in browser/OS
Server knows: Only the proxy's IP, not client's IP
Examples: Squid, corporate web filter, VPN
Use case: Caching outbound traffic, filtering, anonymity
🔄 Reverse Proxy
Serves: Servers (hides backend infrastructure)
Who uses it: Server operator deploys it
Client knows: Only the proxy's IP, not backend IPs
Examples: Nginx, HAProxy, Traefik, Envoy
Use case: SSL termination, load balancing, auth, caching
Capabilities

What a Reverse Proxy Actually Does

People say "just put Nginx in front of it" — but what does Nginx actually do as a reverse proxy? Here's every responsibility, explained clearly.

🔒
SSL/TLS Termination
Handles HTTPS on behalf of backend servers. Decrypts incoming TLS traffic, forwards plain HTTP internally. Backend servers never deal with certificates — Nginx manages renewal via certbot/Let's Encrypt.
⚖️
Load Balancing
Distributes requests across multiple backend instances using round-robin, least connections, or IP hash. One Nginx handles traffic for 10 app server instances behind it.
📂
Static File Serving
Serves static assets (CSS, JS, images) directly from disk — never hitting your app server. Nginx can serve static files at 100k+ req/sec with microsecond latency. App server only handles dynamic requests.
🗜️
Compression
Gzip/Brotli compress responses before sending to clients. API responses may shrink 60–90%. Backend servers return uncompressed data; Nginx handles all compression transparently.
📋
Header Manipulation
Add, remove, or modify request/response headers. Add X-Real-IP so apps know client IP. Remove X-Powered-By to hide tech stack. Add security headers (HSTS, CSP) to all responses.
🛡️
Security Buffer
Absorbs slow-client attacks (Slowloris). Validates request size limits. Rate limits IPs. Backend servers only see completed, valid requests — protected from many L7 attacks by the proxy layer.
💾
Response Caching
Cache backend responses for a configurable TTL. Identical requests get the cached response without hitting the backend. Nginx proxy cache can hold GB of cached responses on SSD.
🔀
URL Rewriting & Routing
Route /api/v1/* to service A, /api/v2/* to service B, /static/* to S3. Rewrite URLs before forwarding. Redirect HTTP to HTTPS. Handle versioning transparently.
Internal Architecture

How Nginx Works Internally as a Reverse Proxy

// Nginx Reverse Proxy — Full Request Pipeline

Nginx's Event-Driven Architecture

Nginx uses a non-blocking event-driven model. Each worker process (one per CPU core) handles thousands of simultaneous connections using epoll/kqueue — no thread-per-connection overhead. This is why Nginx can handle 10,000+ concurrent connections per worker with minimal memory (~2MB per worker idle).

Compare to Apache's process-per-request model: 1,000 connections = 1,000 processes = ~1GB RAM just for Apache. Nginx handles 1,000 connections in one worker using ~2MB. This "C10K problem" solution is why Nginx replaced Apache as the dominant web server.

These three are often confused because modern tools (Nginx, Traefik) do all three simultaneously. Here's the precise distinction:

Reverse Proxy: Any server that receives requests and forwards them to backends. The broadest category. Every load balancer and API gateway is also a reverse proxy. Primary job: request forwarding + termination.

Load Balancer: A reverse proxy whose primary job is distributing traffic across multiple identical backend instances. Focuses on traffic distribution, health checking, and backend pool management.

API Gateway: A reverse proxy with application-layer intelligence: authentication, authorisation, rate limiting, request transformation, and API-level routing. Understands the business logic of APIs.

In practice: Nginx configured with upstream blocks IS a load balancer. Nginx configured with JWT validation IS an API gateway. Kong IS a reverse proxy AND an API gateway. The labels describe the primary role, not the underlying technology.

Configuration

Production Nginx Reverse Proxy Configurations

Complete Production Nginx Config

NGINX nginx.conf — Full production reverse proxy 
# ─── Global Settings ─────────────────────────────────
user       nginx;
worker_processes auto;          # one per CPU core
worker_rlimit_nofile 65535;     # max open file descriptors
error_log  /var/log/nginx/error.log warn;
pid        /run/nginx.pid;

events {
  worker_connections  4096;  # per worker, so 4 workers × 4096 = 16384 conns
  use                 epoll; # Linux kernel event model
  multi_accept        on;    # accept multiple connections at once
}

http {
  # ─── Upstream (backend server pool) ─────────────────
  upstream app_servers {
    least_conn;                             # route to least busy server
    keepalive     32;                       # persistent connections to backend
    server 127.0.0.1:3000 weight=3;       # higher weight = more traffic
    server 127.0.0.1:3001 weight=2;
    server 127.0.0.1:3002 backup;          # only used if others fail
  }

  # ─── Security Headers ────────────────────────────────
  add_header X-Frame-Options       "SAMEORIGIN"  always;
  add_header X-Content-Type-Options "nosniff"     always;
  add_header X-XSS-Protection      "1; mode=block" always;
  add_header Referrer-Policy       "strict-origin-when-cross-origin" always;
  server_tokens off;               # hide nginx version in headers

  # ─── Compression ─────────────────────────────────────
  gzip              on;
  gzip_vary         on;
  gzip_proxied      any;
  gzip_comp_level   6;   # 1-9, higher = smaller but more CPU
  gzip_types        text/plain text/css application/json application/javascript
                    text/xml application/xml image/svg+xml;
  gzip_min_length   1024; # don't compress tiny responses

  # ─── Proxy Cache ─────────────────────────────────────
  proxy_cache_path /var/cache/nginx
    levels=1:2
    keys_zone=app_cache:10m  # 10MB metadata zone
    max_size=1g               # max 1GB cached responses
    inactive=60m              # evict if not accessed in 60 min
    use_temp_path=off;

  # ─── Rate Limiting ───────────────────────────────────
  limit_req_zone $binary_remote_addr zone=api:10m rate=100r/m;
  limit_req_zone $binary_remote_addr zone=login:1m rate=5r/m;

  # ─── HTTP → HTTPS redirect ───────────────────────────
  server {
    listen 80;
    server_name yourapp.com www.yourapp.com;
    return 301 https://$host$request_uri;
  }

  # ─── Main HTTPS virtual host ─────────────────────────
  server {
    listen 443 ssl http2;
    server_name yourapp.com www.yourapp.com;

    # SSL — managed by certbot/Let's Encrypt
    ssl_certificate      /etc/letsencrypt/live/yourapp.com/fullchain.pem;
    ssl_certificate_key  /etc/letsencrypt/live/yourapp.com/privkey.pem;
    ssl_protocols        TLSv1.2 TLSv1.3;
    ssl_ciphers          ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
    ssl_session_cache    shared:SSL:10m;   # session resumption cache
    ssl_session_timeout  1d;
    ssl_session_tickets  off;
    add_header Strict-Transport-Security "max-age=63072000" always;

    # Max upload size (important for file upload APIs)
    client_max_body_size 50m;

    # ─── Static files — serve directly from disk ─────────
    location /static/ {
      alias     /var/www/static/;
      expires   1y;
      add_header Cache-Control "public, max-age=31536000, immutable";
      access_log off;  # don't log static asset hits (high volume)
    }

    # ─── Login — strict rate limit ─────────────────────
    location /auth/login {
      limit_req zone=login burst=3;
      proxy_pass http://app_servers;
      include /etc/nginx/proxy_params;
    }

    # ─── API — proxy to backend pool ────────────────────
    location /api/ {
      limit_req zone=api burst=20 nodelay;

      proxy_pass          http://app_servers;
      proxy_http_version  1.1;

      # Tell backend who the real client is
      proxy_set_header X-Real-IP        $remote_addr;
      proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
      proxy_set_header X-Forwarded-Proto $scheme;
      proxy_set_header Host              $host;
      proxy_set_header Connection        "";  # enables keepalive

      # Timeouts — prevent slow backends from blocking everything
      proxy_connect_timeout  5s;
      proxy_send_timeout     30s;
      proxy_read_timeout     30s;

      # Buffering — absorb slow clients (anti-Slowloris)
      proxy_buffering         on;
      proxy_buffer_size       4k;
      proxy_buffers           8 16k;

      # Caching for public API endpoints
      proxy_cache         app_cache;
      proxy_cache_valid   200 5m;    # cache 200 responses for 5 minutes
      proxy_cache_methods GET HEAD;
      proxy_cache_bypass  $http_authorization;  # never cache authenticated reqs
      add_header          X-Cache-Status $upstream_cache_status;
    }

    # ─── WebSocket support ──────────────────────────────
    location /ws/ {
      proxy_pass          http://app_servers;
      proxy_http_version  1.1;
      proxy_set_header    Upgrade    $http_upgrade;   # WebSocket upgrade headers
      proxy_set_header    Connection "upgrade";
      proxy_read_timeout  3600s;  # long timeout for persistent WS connections
    }

    # ─── Catch-all — serve SPA index.html ───────────────
    location / {
      try_files $uri $uri/ /index.html;
      root      /var/www/html;
    }
  }
}

Nginx as Multi-Service Reverse Proxy (Microservices)

NGINX nginx-microservices.conf — Route different paths to different services 
http {
  # Each microservice has its own upstream pool
  upstream user_service    { server user-svc:3001; }
  upstream order_service   { server order-svc:3002; }
  upstream payment_service { server payment-svc:3003; }
  upstream search_service  { server search-svc:3004; }

  server {
    listen 443 ssl http2;
    server_name api.yourapp.com;

    # Route by URL path to appropriate microservice
    location /api/users/    { proxy_pass http://user_service;    }
    location /api/orders/   { proxy_pass http://order_service;   }
    location /api/payments/ { proxy_pass http://payment_service; }
    location /api/search/   { proxy_pass http://search_service;  }

    # Route by subdomain pattern
    if ($host ~* ^admin\.) {
      proxy_pass http://admin_service;
    }

    # URL rewrite example: /api/v1/users → /users (strip prefix)
    location /api/v1/ {
      rewrite ^/api/v1/(.*)$ /$1 break;
      proxy_pass http://user_service;
    }

    # Canary deployment: 10% traffic to new version
    split_clients "${remote_addr}${request_uri}" $upstream_variant {
      10%    new_version;
      *      stable_version;
    }
    upstream new_version    { server app-new:3000; }
    upstream stable_version { server app-stable:3000; }
    location /api/feature/ {
      proxy_pass http://$upstream_variant;
    }
  }
}
Advanced Patterns

Advanced Reverse Proxy Patterns

Standard TLS authenticates only the server. mTLS requires both the client (proxy) and server (backend) to present valid certificates. This means even if an attacker gets inside your network, they can't directly hit backend services without the proxy's certificate.

Configuration: The reverse proxy presents its client certificate when connecting to backends. Backends verify the proxy certificate. Any connection without the correct certificate is rejected — even from inside the datacenter.

Service meshes like Istio enforce mTLS automatically for all service-to-service communication by injecting Envoy sidecars that handle certificate management via SPIFFE/SPIRE.

A Slowloris attack sends requests extremely slowly — partial HTTP headers trickled byte-by-byte. Without a reverse proxy, this consumes a backend thread for minutes per connection. With thousands of such connections, the backend is exhausted.

Nginx's proxy buffering solves this: Nginx buffers the complete request before forwarding to the backend. The backend only sees the complete, valid request and immediately finishes. Nginx handles the slow client connection independently. Your backend never sees slow clients.

Run two identical environments (blue = current, green = new). Deploy new version to green. Point the reverse proxy upstream from blue to green — zero-downtime switch. If green has issues, point back to blue in seconds.

Implementation: Two upstream blocks in Nginx. Switch between them using a variable set by your deployment script (or Nginx's runtime API for HAProxy). No application restarts, no DNS changes, instantaneous cutover.

The reverse proxy sees 100% of traffic — making it the ideal place to collect metrics and logs. Configure structured access logs with: latency, upstream response time, cache status, upstream address (which backend handled it), and status codes.

Nginx access log format for structured logging: log_format json_combined escape=json '{"timestamp":"$time_iso8601","method":"$request_method","uri":"$uri","status":$status,"upstream":"$upstream_addr","upstream_time":"$upstream_response_time","request_time":"$request_time"}'

Ship these logs to your ELK stack or Datadog. You get complete API observability across all backend services without modifying any application code.

Industry Usage

How Top Companies Use Reverse Proxies

🎬 Netflix Zuul 2 + Envoy
Netflix's Zuul 2 is their edge reverse proxy — rewritten from blocking to async Netty I/O to handle 100B+ requests/day. Every request entering their system goes through Zuul, which handles dynamic routing, auth, A/B testing, and DDoS mitigation. Internally, each microservice uses Envoy as a sidecar for service-mesh traffic management.
🔍 Google Envoy / GFE
Google's Google Frontend (GFE) is their global reverse proxy — every request to Google services (Search, Gmail, YouTube) goes through GFE before reaching backend servers. It handles SSL termination, routing, load balancing, and DDoS at massive scale. Envoy (open-sourced by Lyft, co-developed with Google) powers their internal service mesh.
☁️ Cloudflare Nginx + Custom
Cloudflare's edge runs heavily modified Nginx as a reverse proxy for millions of websites. They've contributed major patches upstream (lua-nginx-module, ngx_http_substitutions_filter_module). Their Workers product runs V8 workers inside Nginx workers via a custom C extension — executing customer JavaScript in the proxy request pipeline.
🐦 Twitter Nginx + TFE
Twitter Frontend (TFE) is their reverse proxy layer handling ~500k requests/sec. Built on Nginx with heavy custom Lua scripting for request routing, authentication, rate limiting, and feature flags. The Lua scripting allowed engineering teams to modify routing logic without restarting Nginx — critical for a platform where downtime is very visible.
📦 GitHub HAProxy + Nginx
GitHub uses HAProxy for L4 load balancing and Nginx as reverse proxy for application traffic. When GitHub migrated from monolith to services, Nginx path-based routing was the key enabler — gradually moving routes like /api/* to new microservices while keeping legacy routes on the monolith. Zero-risk incremental migration using the proxy layer.
🏦 Stripe Nginx + Envoy
Stripe's API handling is built around Nginx as the edge reverse proxy with comprehensive request logging. Their internal service mesh uses Envoy with tight circuit-breaker configuration — a single slow service cannot cascade failures because Envoy enforces request timeouts and circuit breaks at every hop. Critical for a payment platform where partial failures are worse than clean failures.
Tradeoffs

Advantages, Disadvantages & Best Practices

✅ Advantages
  • Hides backend topology — clients never know server IPs
  • Centralised SSL management — one cert for all services
  • Decouples client interface from backend implementation
  • Handles slow clients before they affect backends
  • Enables zero-downtime deployments (blue-green)
  • Single point for logging, metrics, rate limiting
  • Serves static files without hitting app servers
❌ Disadvantages
  • Additional network hop (adds ~1-5ms)
  • Single point of failure if not deployed in HA
  • Configuration complexity (Nginx config is notoriously tricky)
  • Must handle WebSocket upgrades explicitly
  • Debugging requires checking proxy logs as well as app logs
  • Buffer limits can cause 413 errors for large uploads
  • Proxy cache invalidation adds operational complexity
⚠️
Critical Mistake: Forgetting to set proxy_set_header X-Real-IP $remote_addr. Without this, your application sees only the proxy's IP (127.0.0.1 or internal IP) as the client address. Rate limiting by IP, geo-location features, and audit logs all break silently. Always set real-IP headers and always enable trust proxy in your application framework.
Interview Preparation

Common Interview Questions

What is the difference between a reverse proxy and a load balancer?
A reverse proxy is any server that receives client requests and forwards them to backends — the broadest category. A load balancer is a specific type of reverse proxy whose primary role is distributing traffic across multiple identical backend instances. All load balancers are reverse proxies, but not all reverse proxies are load balancers. Nginx configured with one upstream server is a reverse proxy but not a load balancer. Nginx configured with multiple upstream servers and a balancing algorithm is both. The distinction is about primary purpose, not the underlying technology. In production systems, a reverse proxy typically handles: SSL termination, request routing, static file serving, compression, caching, header manipulation, and load balancing — all simultaneously. The term "reverse proxy" covers the full concept; "load balancer" describes one specific function it performs.
Why do we need a reverse proxy in front of a Node.js application?
Several reasons: (1) Node.js runs on port 3000 (non-privileged). Nginx listens on port 80/443 (privileged ports requiring root), proxies to Node.js. Node.js doesn't need to run as root. (2) SSL termination — Nginx handles HTTPS, Node.js receives plain HTTP. Simpler cert management. (3) Static file serving — Nginx serves CSS/JS/images from disk at 100k+ req/sec without hitting Node.js. (4) Compression — Nginx compresses responses; Node.js doesn't waste CPU on gzip. (5) Slow client protection — Nginx buffers slow HTTP requests; Node.js only sees complete requests. (6) Multiple apps on one server — Nginx routes api.domain.com to port 3000, admin.domain.com to port 3001. (7) Load balancing — distribute across multiple Node.js processes (cluster mode). (8) Zero-downtime deploys — swap the upstream in Nginx without DNS changes.
How does Nginx handle 10,000 concurrent connections with a single worker?
Nginx uses an event-driven, non-blocking I/O architecture. Each worker process runs a single-threaded event loop using epoll (Linux) or kqueue (macOS). When a connection arrives, Nginx registers it with the kernel's event notification mechanism. Instead of blocking a thread waiting for data, the event loop handles thousands of connections concurrently: register connection → handle when data arrives → process → register again. This is the same model as Node.js (libuv). One worker can handle thousands of connections because it never blocks — it just switches between connections as data becomes available. Memory per connection is very small (~a few KB for buffers) vs thread-per-connection models (~8MB stack per thread). Apache's prefork model needs 1000 processes for 1000 connections = 1GB+ RAM. Nginx needs 1 worker for 1000 connections = ~10MB RAM.
What happens when a reverse proxy receives a WebSocket connection request?
WebSockets start as an HTTP request with special Upgrade headers: "Upgrade: websocket" and "Connection: Upgrade". Without specific configuration, most reverse proxies strip these headers and the WebSocket handshake fails — the client gets a 400 Bad Request. To support WebSockets, Nginx must: (1) Pass through the Upgrade header: proxy_set_header Upgrade $http_upgrade. (2) Pass through the Connection header with value "upgrade": proxy_set_header Connection "upgrade". (3) Set proxy_http_version 1.1 (WebSockets require HTTP/1.1). (4) Set a long proxy_read_timeout (e.g., 3600s) because WebSocket connections are persistent — default timeout would close them prematurely. Once the handshake completes (101 Switching Protocols), Nginx acts as a transparent TCP tunnel between client and backend, forwarding WebSocket frames bidirectionally.
Design a reverse proxy architecture for a system with 5 different microservices.
Architecture: Single Nginx instance (or pair for HA via Keepalived) as the edge reverse proxy. Five upstream blocks, one per service. Path-based routing: /api/users/ → user-service, /api/orders/ → order-service, /api/payments/ → payment-service (with stricter rate limiting, 10 req/min), /api/search/ → search-service, /api/notifications/ → notification-service. All services behind the proxy — none have public IPs. SSL terminates at Nginx (one cert for api.yourapp.com, issued by Let's Encrypt, auto-renewed by certbot). Each upstream has health checks (proxy_next_upstream error timeout), keepalive connections (reduce TCP handshake overhead), and timeout settings. Nginx adds X-Real-IP, X-Forwarded-For, X-Request-ID headers for all requests. Access logs in structured JSON shipped to Elasticsearch. For high availability: Active-passive Nginx pair with VRRP/Keepalived floating VIP. In Kubernetes: replace Nginx with an Ingress Controller (Nginx Ingress or Traefik) that reads routing rules from Kubernetes Ingress resources — same concept, cloud-native implementation.
Summary
🎯 What You Now Know
  • ✦ Reverse proxy acts on behalf of servers; forward proxy acts on behalf of clients — completely different roles
  • ✦ Every load balancer is a reverse proxy; not every reverse proxy is a load balancer
  • ✦ Reverse proxy capabilities: SSL termination, load balancing, static serving, compression, caching, header manipulation, DDoS buffering, URL routing
  • ✦ Nginx event-driven model: one worker handles 1000s of connections via epoll — no thread-per-connection overhead
  • ✦ Always set X-Real-IP header and enable trust proxy in apps — otherwise rate limiting and geo features break
  • ✦ WebSocket requires explicit Upgrade/Connection header pass-through and long proxy_read_timeout
  • ✦ Nginx: static config, battle-tested, high performance. Traefik: Docker/K8s native, auto-discovers services, dynamic config
  • ✦ Proxy buffering protects backends from slow-client attacks (Slowloris)
  • ✦ Blue-green and canary deployments are cleanly implemented via upstream switching in reverse proxy config
