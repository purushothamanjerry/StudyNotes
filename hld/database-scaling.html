Hardest Topic
Introduction

Why Database Scaling is the Hardest Problem

Databases are the most difficult component to scale in any distributed system. Compute and memory are stateless and horizontally scalable by nature. Databases are stateful — you can't just spin up 10 new database servers without carefully managing how data is distributed, kept consistent, and accessed.

Every scaling decision involves trade-offs between consistency, availability, and partition tolerance (CAP theorem). Understanding these trade-offs is what separates senior engineers from junior ones.

📚 Analogy — The Library Problem

A small town has one library (your database). As the town grows, more people want books simultaneously. You can: Vertical scaling — build a bigger library with more shelves (faster hardware). Read replicas — photocopy popular books and put them in branch libraries (read replicas). Sharding — move fiction books to Library A and non-fiction to Library B (shard by category). Caching — keep the most popular books in a small shelf at the entrance (Redis). Each approach has costs, complexity, and limits.

When your app starts, one database server handles everything. As users grow, the database gets slower. Queries take longer. The CPU on the DB server runs at 90%. Users see timeouts and errors.

Database scaling is about figuring out: how do I make the database handle more queries faster? The solutions range from "buy a bigger server" to "split the data across 100 servers globally." Each step is more complex than the last.

Database scaling involves multiple orthogonal dimensions: query throughput (reads/writes per second), storage capacity (total data size), connection concurrency (simultaneous connections), and latency (time per query). Each requires different techniques. A system might need read replicas for throughput but sharding for storage, and connection pooling for concurrency — all simultaneously.

The fundamental tension: strong consistency requires coordination between nodes, which adds latency and limits throughput. Relaxing consistency (eventual consistency) enables horizontal scaling but requires careful application-level handling of stale reads and conflict resolution.

Scaling Progression

The Database Scaling Ladder — Step by Step

Most systems follow this progression as they grow. Each step should only be taken when the previous one is insufficient — premature optimization is real.

1
Optimise Queries & Add Indexes
Before adding hardware, fix the queries. A missing index on a WHERE column turns an O(log n) lookup into an O(n) table scan. Adding one index often makes a 50ms query run in 0.5ms — a 100× speedup with zero infrastructure cost. Run EXPLAIN ANALYZE on slow queries. Add composite indexes for common filter combinations. This alone handles most early scaling needs.
2
Vertical Scaling (Scale Up)
Upgrade to a larger server: more CPU cores, more RAM, faster NVMe SSDs. PostgreSQL scales very well vertically — a single server with 64 cores, 512GB RAM, and NVMe storage can handle hundreds of thousands of queries per second. Simple, no application changes required. Limit: a single server can only be so large, and it's still a single point of failure.
3
Connection Pooling (PgBouncer / ProxySQL)
PostgreSQL is expensive to connect to — each connection forks a process (~5–10MB RAM). 1,000 simultaneous connections = 5–10GB RAM just for connections. PgBouncer sits between your app and DB, multiplexing many app connections onto a small pool of real DB connections. With pooling, handle 10,000 app connections using only 50 real DB connections.
4
Caching Layer (Redis)
Cache frequently-read data in Redis. For most apps, 80% of DB reads are for 20% of data (Pareto principle). Cache user profiles, product pages, session data. Turn 50ms DB reads into 0.3ms Redis reads. Reduces DB load by 60–90% for read-heavy applications. This is often the highest-leverage single change you can make.
5
Read Replicas
Create replica copies of your database that receive all writes but can serve reads independently. Route read-heavy queries (reports, dashboards, analytics) to replicas. Primary handles only writes. Typical apps are 80% reads / 20% writes — replicas dramatically reduce primary load. Multiple replicas can be added for more read throughput.
6
Sharding (Horizontal Partitioning)
Split data across multiple database servers (shards) based on a shard key. Shard by user ID: users 1–1M on Shard 1, 1M–2M on Shard 2, etc. Each shard is independent — its own server, its own storage. This is how you scale to billions of rows and millions of writes/second. The most complex option — only when everything else is exhausted.
Read Replicas

Read Replicas — How Replication Works

// Primary-Replica Replication Architecture

PostgreSQL uses Write-Ahead Logging (WAL). Every change (INSERT, UPDATE, DELETE) is first written to the WAL log as a binary record, then applied to the actual data files. This order guarantees durability — if the server crashes, the WAL can replay changes on restart.

For replication: the WAL stream is continuously sent to replica servers. Each replica applies WAL records in order, maintaining an exact copy of the primary's data. This is called streaming replication.

Synchronous vs Asynchronous replication: Async (default) — primary doesn't wait for replica to confirm. Fastest but tiny data loss window on failover. Sync — primary waits for at least one replica to confirm before committing. Zero data loss but adds latency (~network RTT) to every write.

Async replicas are slightly behind the primary (typically 10–200ms, can be seconds under heavy write load). If you write a record and immediately read from a replica, you might get the old value. This is called a stale read.

Solutions:

1. Read-your-writes consistency: After a write, route the next read for that user to the primary (or sticky-route that user to primary for 1 second). Other users can still read from replicas.

2. Application-level tracking: After a write, record the WAL LSN (Log Sequence Number). Before reading from a replica, check its LSN — only route to replicas that have caught up past your write's LSN.

3. Session-level routing: Within a transaction or user session, always read from primary. Use replicas only for background tasks and analytics where stale data is acceptable.

Horizontal Scaling

Database Sharding — Splitting Data Across Servers

Sharding splits your data across multiple independent database servers (shards). Each shard holds a subset of the data and operates completely independently. This is the ultimate horizontal scaling strategy for databases.

// Sharding by User ID — Hash-based Distribution

Sharding Strategies

shard = hash(shardKey) % numberOfShards

Apply a hash function to the shard key (usually user ID or entity ID), then take modulo by number of shards. This gives uniform distribution — no hot spots.

Problem: Adding a shard changes the modulo, so most keys map to different shards — you need to move ~(N-1)/N of all data when adding one shard. Solved with consistent hashing (only ~1/N data moves).

Used by: Cassandra, DynamoDB, Vitess (MySQL sharding layer)

Assign contiguous ranges of shard key values to shards. User IDs 1–1,000,000 → Shard 1; 1,000,001–2,000,000 → Shard 2; etc.

Advantage: Range queries are efficient — "get all users created in Jan 2024" goes to one shard. Easy to add shards (just add a new range).

Problem: Hot spots — if new users all land on the latest shard (because IDs are sequential), one shard gets all writes while others sit idle.

Used by: MongoDB (range-based chunks), HBase, Google Bigtable

Maintain a lookup table (directory) that maps shard keys to shard locations. The router checks this table to find which shard has each key.

Advantage: Maximum flexibility — move specific users to specific shards, gradually migrate data. Can implement custom logic (VIP customers on faster shard).

Problem: The directory itself becomes a bottleneck and single point of failure. Must be highly available and fast.

The Cross-Shard Query Problem

🚨
The Biggest Sharding Pain: Queries that span multiple shards require either: (1) the application to query all shards and merge results (scatter-gather), which is slow and complex, or (2) denormalization — store copies of data on each relevant shard. JOINs across shards are impossible in a traditional sense. Before sharding, carefully model your access patterns and ensure the majority of queries are single-shard operations. The shard key choice is the most important architectural decision when sharding.
Query Optimisation

Indexing — The Highest Leverage Scaling Tool

Before any infrastructure scaling, indexing almost always provides the biggest performance gain per unit of effort. A missing index can make a query 100–10,000× slower than necessary.

The default PostgreSQL index type. A balanced tree where each node contains key values and pointers to children. Lookups, range queries, and sorts are O(log n). For a table with 100 million rows, a B-tree lookup requires at most ~27 node reads (log₂(100M) ≈ 27) vs 100M row reads for a table scan.

Best for: Equality lookups (WHERE user_id = 123), range queries (WHERE created_at > '2024-01-01'), sorting (ORDER BY name).

Downside: Writes are slightly slower (must update the index). Each index adds ~10–30% overhead to INSERT/UPDATE operations on that table.

A single index on multiple columns. CREATE INDEX ON orders (user_id, status, created_at). This index supports queries that filter on any left-prefix of the indexed columns.

Supports: WHERE user_id=123, WHERE user_id=123 AND status='pending', WHERE user_id=123 AND status='pending' AND created_at > '2024-01-01'.

Does NOT support: WHERE status='pending' alone (skips user_id — the leftmost column must be present).

Key rule — Column Order Matters: Put highest cardinality, most-filtered columns first. Columns used in ORDER BY last (enables index sort). The selectivity of the leading column determines how many rows are eliminated first.

Index only the rows matching a WHERE condition. CREATE INDEX ON orders (created_at) WHERE status = 'pending'. If 99% of orders are 'completed' and you only query pending orders, a partial index on pending orders is much smaller and faster than a full index.

Real example: A table with 100M orders, only 10k currently pending. A partial index on pending orders contains only 10k entries vs 100M for a full index — 10,000× smaller, fits entirely in cache, ultra-fast.

The single most important PostgreSQL tool for performance. Prepend EXPLAIN ANALYZE to any query to see exactly how the query planner executes it, where time is spent, and whether indexes are used.

Key things to look for: Seq Scan (sequential table scan — usually means missing index), cost estimates vs actual, rows estimates vs actual (large discrepancies mean stale statistics — run ANALYZE tablename), Nested Loop vs Hash Join (hash joins are usually faster for large datasets).

Connection Management

Connection Pooling — PgBouncer & ProxySQL

Each PostgreSQL connection spawns a backend process consuming ~5-10MB RAM. With 500 app server instances each holding 20 connections, you'd need 10,000 DB connections — 50–100GB RAM just for connection overhead. Connection pooling solves this.

🔄
Session Pooling
One server connection per client session. Client holds the connection for its entire lifetime. Reduces connection spawning overhead but doesn't save connections overall. Best for long-lived connections.
⚡
Transaction Pooling
A server connection is held only during a transaction. After COMMIT, the connection returns to the pool. Allows 10,000 client connections to share 50 server connections if transactions are short. Default PgBouncer mode for web apps.
🚀
Statement Pooling
Most aggressive — connection released after each statement. Maximum connection reuse but disables prepared statements and multi-statement transactions. Rarely used due to application compatibility issues.

[databases]

Table Management

Table Partitioning — Managing Huge Tables

Partitioning splits a single large table into smaller physical sub-tables (partitions) while keeping a single logical table visible to the application. Unlike sharding (across servers), partitioning happens on a single server.

Split a table by time ranges — one partition per month, quarter, or year. Events from January 2024 go to the events_2024_01 partition; February to events_2024_02, etc.

Massive benefit: When you query WHERE created_at > '2024-01-01', PostgreSQL only scans the relevant monthly partitions — partition pruning. A query that once scanned 10 billion rows now only scans 100 million rows in the current month's partition.

Another benefit: Drop old partitions instantly (DROP TABLE events_2022_01) — no slow DELETE of millions of rows, just drop the partition file. Perfect for time-series data with retention policies.

List partitioning: Split by discrete values. E.g., orders partitioned by country: orders_us, orders_eu, orders_asia. Queries filtering by country only scan the relevant partition.

Hash partitioning: Distribute rows evenly across N partitions using a hash of the partition key. Good for even distribution without specific range/list semantics. PostgreSQL native hash partitioning since v11.

Subpartitioning: Partition by date, then hash-subpartition within each month for additional parallelism. E.g., events_2024_01 → 8 hash subpartitions processed in parallel.

SQL partitioning.sql — PostgreSQL date-range partitioning 
-- Create partitioned table (PostgreSQL 10+)
CREATE TABLE events (
  id           BIGSERIAL,
  user_id      BIGINT       NOT NULL,
  event_type   VARCHAR(50) NOT NULL,
  payload      JSONB,
  created_at   TIMESTAMPTZ  NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (created_at);

-- Create monthly partitions
CREATE TABLE events_2024_01 PARTITION OF events
  FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

CREATE TABLE events_2024_02 PARTITION OF events
  FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');

-- Each partition gets its own indexes
CREATE INDEX ON events_2024_01 (user_id, created_at);
CREATE INDEX ON events_2024_02 (user_id, created_at);

-- Query looks the same — partition pruning is automatic
SELECT * FROM events
WHERE user_id = 12345
  AND created_at >= '2024-01-15'
  AND created_at <  '2024-02-01';
-- ^ Only scans events_2024_01 partition, not the full table

-- Automate partition creation with pg_partman extension
-- Or use a cron job to create next month's partition:
CREATE TABLE events_2024_03 PARTITION OF events
  FOR VALUES FROM ('2024-03-01') TO ('2024-04-01');

-- Drop old data instantly (no DELETE overhead)
DROP TABLE events_2022_01;  -- Drops the partition file, instant
NoSQL Trade-offs

SQL vs NoSQL — When to Choose Each

NoSQL databases trade SQL features (ACID, joins, schemas) for horizontal scalability, flexibility, and specialised performance profiles. Choosing NoSQL isn't about "it's faster" — it's about matching the data model to the use case.

DatabaseTypeBest ForTradeoff
PostgreSQLRelational (SQL)Complex queries, ACID transactions, financial dataHarder to scale writes horizontally
MongoDBDocumentFlexible schemas, hierarchical data, rapid iterationNo joins, eventual consistency by default
CassandraWide-ColumnTime-series, write-heavy, global distributionQuery design must match access patterns upfront
DynamoDBKey-Value/DocumentServerless, predictable latency, auto-scalingVendor lock-in, expensive for complex queries
RedisIn-MemoryCaching, sessions, leaderboards, pub/subData limited by RAM, not a primary DB
ElasticsearchSearch EngineFull-text search, log analytics, aggregationsNot for primary storage, eventual consistency
ClickHouseColumnar OLAPAnalytics, aggregations on billions of rowsNot for OLTP (individual row reads/writes)
Neo4jGraphSocial graphs, recommendation engines, fraud detectionDoesn't scale as well for non-graph queries

In a distributed system, you can only guarantee 2 of these 3 properties simultaneously:

C — Consistency: Every read gets the most recent write (or an error). All nodes see the same data at the same time.

A — Availability: Every request gets a response (not an error), even if it might be stale.

P — Partition Tolerance: The system continues operating even when network splits occur between nodes. In real distributed systems, P is non-negotiable — networks fail. So the real choice is C vs A during a partition.

CP systems (prefer consistency): HBase, Zookeeper, MongoDB with majority reads. If nodes can't communicate, some return errors rather than stale data.

AP systems (prefer availability): Cassandra, CouchDB, DynamoDB with eventual consistency. Nodes always respond, even if data might be slightly stale.

Real-world nuance: Most modern databases offer tunable consistency — you can choose per-query how consistent you need the result to be. "Strong consistency for payment queries, eventual consistency for view counts."

OLTP (Online Transaction Processing): Many small, fast read/write operations. INSERT one order, UPDATE one user, GET one product. PostgreSQL, MySQL are optimised for this.

OLAP (Online Analytical Processing): Few large queries scanning millions of rows. "What's the total revenue by country last quarter?" "How many users converted this week?" ClickHouse, Redshift, BigQuery are optimised for this — columnar storage reads only the columns needed, skipping irrelevant ones.

Why separate them: OLAP queries (full table scans) would destroy OLTP performance if run on the same database. The solution: ETL (Extract, Transform, Load) data from OLTP DB → data warehouse nightly (or real-time via CDC). Run analytics on the warehouse; transactional queries on the OLTP DB.

Read/Write Splitting in Node.js

JAVASCRIPT db/pool.js — Read/write splitting with Knex 
const knex = require('knex');

// Write pool → primary DB (via PgBouncer)
const writePool = knex({
  client: 'pg',
  connection: {
    host:     process.env.DB_PRIMARY_HOST,
    port:     6432,              // PgBouncer port
    database: 'app_db',
    user:     process.env.DB_USER,
    password: process.env.DB_PASS,
    ssl:      { rejectUnauthorized: true },
  },
  pool: { min: 2, max: 10 },   // Small pool — PgBouncer multiplexes
});

// Read pool → replica DB (round-robin across replicas)
const readPool = knex({
  client: 'pg',
  connection: {
    host:     process.env.DB_REPLICA_HOST,  // HAProxy/RDS proxy in front of replicas
    port:     6432,
    database: 'app_db_read',
    user:     process.env.DB_USER_READONLY,
    password: process.env.DB_PASS_READONLY,
  },
  pool: { min: 2, max: 20 },   // Larger pool for reads
});

// ─── DB abstraction layer ────────────────────────────
const db = {
  // Reads → replica (async, eventually consistent)
  read:  (tableName) => readPool(tableName),

  // Writes → primary
  write: (tableName) => writePool(tableName),

  // Critical reads (post-write) → primary to avoid stale reads
  readStrong: (tableName) => writePool(tableName),

  // Transactions always go to primary
  transaction: (callback) => writePool.transaction(callback),
};

// ─── Usage examples ──────────────────────────────────
// List users — can tolerate slight staleness → read replica
async function listUsers(page, limit) {
  return db.read('users')
    .select('id', 'name', 'email')
    .orderBy('created_at', 'desc')
    .limit(limit).offset(page * limit);
}

// Create user — write to primary, then read from primary
async function createUser(data) {
  const [user] = await db.write('users').insert(data).returning('*');
  // Use readStrong (primary) immediately after write — avoid stale read
  return db.readStrong('users').where({ id: user.id }).first();
}

// Transfer funds — ACID transaction → primary only
async function transferFunds(fromId, toId, amount) {
  return db.transaction(async (trx) => {
    const from = await trx('accounts').where({ id: fromId }).first();
    if (from.balance < amount) throw new Error('Insufficient funds');
    await trx('accounts').where({ id: fromId }).decrement('balance', amount);
    await trx('accounts').where({ id: toId }).increment('balance', amount);
    await trx('transactions').insert({ fromId, toId, amount, timestamp: new Date() });
  });
}

PostgreSQL Indexing Cheatsheet

SQL indexes.sql — Production indexing patterns 
-- ─── Basic Index ──────────────────────────────────────
-- Missing index on foreign key (very common issue)
CREATE INDEX CONCURRENTLY idx_orders_user_id
  ON orders (user_id);
-- CONCURRENTLY = no table lock, safe for production

-- ─── Composite Index (order matters!) ─────────────────
-- Supports: WHERE user_id=X, WHERE user_id=X AND status=Y
-- Does NOT support: WHERE status=Y alone
CREATE INDEX CONCURRENTLY idx_orders_user_status
  ON orders (user_id, status, created_at DESC);

-- ─── Partial Index ────────────────────────────────────
-- Only index pending orders (tiny fraction of total)
CREATE INDEX CONCURRENTLY idx_orders_pending
  ON orders (created_at, user_id)
  WHERE status = 'pending';

-- ─── Expression Index ─────────────────────────────────
-- Case-insensitive email lookup
CREATE UNIQUE INDEX idx_users_email_lower
  ON users (LOWER(email));
-- Query: WHERE LOWER(email) = LOWER($1)  ← uses index

-- ─── JSONB Index ──────────────────────────────────────
-- Index a specific key inside a JSONB column
CREATE INDEX idx_orders_payload_country
  ON orders ((payload ->> 'country'));
-- Query: WHERE payload ->> 'country' = 'US'  ← uses index

-- GIN index for any-key JSONB containment queries
CREATE INDEX idx_orders_payload_gin
  ON orders USING GIN (payload);
-- Query: WHERE payload @> '{"status": "paid"}'  ← uses GIN

-- ─── EXPLAIN ANALYZE ──────────────────────────────────
-- Always run this before deploying a slow query fix
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT * FROM orders
WHERE user_id = 12345
  AND status = 'pending'
ORDER BY created_at DESC
LIMIT 20;
-- Look for: "Index Scan" (good) vs "Seq Scan" (usually bad)
-- BUFFERS shows cache hits vs disk reads
Industry Usage

How Top Companies Scale Their Databases

📘 Facebook/Meta MySQL + Vitess/TAO
Facebook built TAO, a custom distributed graph store optimised for social graph reads (who are your friends' friends). For MySQL: Vitess (originally from YouTube) provides automatic horizontal sharding of MySQL with a proxy layer. They also use RocksDB (LSM-tree) as the MySQL storage engine for write-heavy workloads — 10× better write throughput than InnoDB on SSDs.
🛒 Amazon DynamoDB + Aurora
Amazon uses DynamoDB for the Shopping Cart service (predictable single-digit millisecond latency, auto-scaling). Aurora for relational data. The famous DynamoDB paper (2022) explains their partition-based, consistent hashing approach. Key: data modelled around access patterns, not relational normalisation. A DynamoDB table may have dozens of different entity types co-located.
🎬 Netflix Cassandra + CockroachDB
Netflix uses Cassandra for viewing history (billions of rows, 99.999% uptime, multi-region). Key design: user_id as partition key ensures all of one user's data is on one node — fast reads. They also migrated parts of their billing to CockroachDB (globally distributed PostgreSQL-compatible) for ACID transactions with geo-distribution.
🐦 Twitter/X Manhattan + MySQL
Twitter built Manhattan, their own distributed key-value store, for tweets and timelines (horizontal scale, multi-DC). MySQL with custom sharding handles user account data. The challenge: "whale" accounts (celebrities with 100M+ followers) create hot partitions in any hash-based scheme — special handling required for uneven key distribution.
🚗 Uber Docstore + Schemaless
Uber built Docstore (MySQL-based document store with automatic sharding) and Schemaless (append-only data store for trip events). For geospatial indexing of driver locations, they use a custom H3 hexagonal grid system for efficient nearest-driver queries across shards.
📊 Cloudflare D1 + Workers KV
Cloudflare runs Workers KV, a globally distributed key-value store with eventual consistency (writes replicate to all 300 PoPs). For consistent data: D1 (SQLite at the edge). Key insight: SQLite on NVMe at the edge is often faster than a remote PostgreSQL connection — they replicate small SQLite databases to edge nodes nearest each user.

Interview Preparation

Common Interview Questions

How would you scale a database that's getting 100,000 reads/second?
Step 1: Check if reads are hitting indexes — missing indexes often cause queries to scan millions of rows unnecessarily. Add appropriate indexes. Step 2: Add Redis caching for the most frequently-read data (user profiles, product pages). For most apps, 80% of reads are for 20% of data — cache the hot 20%. This typically reduces DB reads by 60-80%. Step 3: Add read replicas. Route all SELECT queries to replicas, writes to primary. With 3 replicas, you've quadrupled your read capacity. Step 4: If still insufficient, consider CDN caching for public data. Step 5: For analytics/reporting queries, route to a dedicated analytics replica or data warehouse. 100k reads/second is comfortably handled by read replicas + caching without sharding.
What is database sharding and what are its downsides?
Sharding splits data across multiple independent database servers based on a shard key. Each shard holds a subset of data. Enables horizontal scaling of both reads and writes — just add more shards. Downsides: (1) Cross-shard queries — JOINs across shards are impossible in traditional SQL. Must use scatter-gather (query all shards, merge) or denormalize data. (2) Resharding complexity — adding shards means redistributing data, which is expensive and risky. Consistent hashing minimizes but doesn't eliminate data movement. (3) Hot spots — if shard key is poorly chosen (e.g., timestamp for events), all new data goes to one shard. (4) Distributed transactions — ACID transactions across shards require 2-phase commit or Saga patterns. (5) Operational complexity — many databases instead of one to monitor, backup, maintain. Use sharding only when replicas + caching are insufficient (typically >1B rows or very high write throughput).
Explain the CAP theorem and how it applies to database choice.
CAP: in a distributed system, you can only guarantee 2 of: Consistency (every read gets the latest write), Availability (every request gets a non-error response), Partition Tolerance (system works despite network failures). Network partitions happen in real systems, so P is non-negotiable. The real choice is C vs A during a partition. CP databases (HBase, MongoDB in strong mode): prefer consistency. During a partition, some requests fail rather than returning stale data. Use for: financial transactions, inventory (must be correct). AP databases (Cassandra, DynamoDB): prefer availability. During a partition, requests succeed but data may be stale (eventually consistent). Use for: social feeds, view counts, recommendations. Practical nuance: most modern databases offer tunable consistency per-query. The choice isn't binary — you can use Cassandra for most things but enforce strong consistency for specific critical operations using QUORUM read/write levels.
How do read replicas work and what are their limitations?
Read replicas receive a continuous stream of write operations from the primary (WAL in PostgreSQL, binlog in MySQL) and apply them asynchronously. They're identical copies of the primary that only accept reads. Limitations: (1) Replication lag — replicas are asynchronously updated, so reads may return data 10ms-1s behind the primary. This is "eventual consistency." (2) Write amplification — writes go to primary only, no write scaling. (3) Not for write-heavy workloads — if your bottleneck is writes, replicas don't help. (4) Failover complexity — promoting a replica to primary requires application configuration changes and may cause brief downtime. Solutions for lag: read-your-writes consistency (route reads after writes to primary), track replication LSN and only use replicas that have caught up, or use synchronous replication (adds latency to writes but zero lag on replicas).
Design the database architecture for a social media platform with 500M users.
User data (profiles, auth): PostgreSQL with read replicas. User IDs are the natural shard key if sharding becomes necessary. Add Redis caching for profile reads (cache-aside, 5min TTL). Social graph (followers/following): Graph database (Neo4j) for traversal queries, or denormalised in Cassandra for simple "list followers" lookups. At 500M users each following 200 people = 100B relationship rows — needs horizontal scaling. Posts/Timeline: Cassandra with user_id as partition key, created_at as clustering key. Optimised for "get latest 20 posts by user X" — single partition read. Fan-out: write to followers' timeline caches (Redis Sorted Sets, scored by timestamp). For celebrities (10M+ followers): fan-out on read instead. Media (images/videos): Object storage (S3/GCS), not in the database. URLs stored in DB. Served via CDN. Analytics: Kafka → ClickHouse (real-time) or BigQuery (batch) for engagement metrics — never run analytics on production OLTP databases.
Summary
🎯 What You Now Know
  • ✦ Always optimise queries and indexes first — often 100× improvement with zero infrastructure cost
  • ✦ Vertical scaling is simple and effective until you hit hardware limits
  • ✦ Connection pooling (PgBouncer) handles concurrency — 10k app connections on 50 real DB connections
  • ✦ Read replicas scale read throughput; caching (Redis) handles hot data — together handle most read scaling
  • ✦ Sharding is the nuclear option for write scaling — high complexity, use only when necessary
  • ✦ Partitioning manages huge tables on one server — date-range partitioning enables partition pruning and instant old-data deletion
  • ✦ CAP theorem: choose between consistency and availability during network partitions — tune per query
  • ✦ OLTP vs OLAP: never run analytics on production databases — use a data warehouse (ClickHouse, BigQuery, Redshift)
  • ✦ Replication lag is real — implement read-your-writes consistency for post-write reads
