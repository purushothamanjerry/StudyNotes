Rate Limiting

The traffic cop of your API. Learn every rate limiting algorithm — Fixed Window, Sliding Window, Token Bucket, Leaky Bucket — and how to implement them in a distributed system with Redis so your API survives abuse, DDoS, and traffic spikes without going down.

Introduction

What is Rate Limiting?

Rate limiting is the practice of controlling how many requests a client can make to a server within a defined time window. It prevents any single user, bot, or service from overwhelming your system with too many requests.

Without rate limiting, a single bad actor (or even a misbehaving client) can send millions of requests per second and take your entire service down for everyone else.

🚦 Analogy — Highway On-Ramp Metering

During rush hour, highway on-ramps use metering signals — traffic lights that let one car through every 5 seconds. Without metering, every car floods onto the highway at once, causing gridlock. The metering doesn't stop you from driving — it spreads your entry over time so the highway keeps flowing. Rate limiting does the same for API traffic: it doesn't block legitimate users permanently, it controls the rate of access to keep the system healthy.

Imagine Stripe's API. They allow each developer to make 100 API calls per second. If someone's script goes crazy and sends 10,000 calls per second, Stripe's servers would slow down for every other developer. Rate limiting detects this and returns 429 Too Many Requests after the 100th call, protecting everyone else.

It's like a bouncer at a club — you can enter, but only one person every few seconds. The club doesn't get overcrowded, and everyone inside has a good experience.

Rate limiting in distributed systems is hard because requests are spread across multiple API servers. A counter stored on one server doesn't reflect global traffic. The solution is a centralized counter store (Redis) with atomic operations.

The key challenges: atomicity (check-and-increment must be one operation), consistency (all nodes read the same counter), low overhead (rate limit check must be <1ms or you've added more latency than the call saves), and sliding precision (fixed windows have boundary-burst problems).

Problem Statement

Why Rate Limiting is Non-Negotiable

💀
DDoS / Brute Force
Attackers send millions of requests to exhaust server resources or brute-force login endpoints. Rate limiting on /login (5 attempts per minute per IP) makes brute-force computationally infeasible.
🐛
Buggy Clients
A developer accidentally deploys a while-loop that calls your API 10,000x/second. Without rate limiting, this takes down production for all users. With it, only that one client is throttled.
💰
Cost Control
Cloud compute, DB queries, and third-party API calls all cost money. Uncontrolled traffic = unbounded costs. Rate limiting enforces usage tiers (free: 100 req/min, paid: 10,000 req/min).
⚖️
Fair Usage
One power user shouldn't degrade service for thousands of regular users. Rate limiting ensures equitable resource distribution — a shared resource is protected from greedy consumers.
⚠️
Real Incident: In 2020, a major fintech company's payment API went down for 4 hours because a single merchant's system had a bug that fired 50,000 payment confirmation requests per second. No rate limiting was in place. The fix: Stripe-style per-API-key rate limits (not just per-IP) would have isolated the blast radius to that one merchant.
Core Algorithms

The 4 Rate Limiting Algorithms

Each algorithm has different performance characteristics, burst behavior, and complexity. Choosing the right one depends on your use case.

// Fixed Window vs Sliding Window — The Boundary Burst Problem
1. Fixed Window Counter
Space: O(1)  |  Time: O(1)
Divide time into fixed windows (e.g., each minute). Maintain a counter per window. Reset to 0 at the start of each new window. If counter ≥ limit, reject.
✓ Extremely simple to implement
✓ Minimal memory (one counter per client)
✓ Fast (single Redis INCR)
✗ Boundary burst: 2× limit possible at window edges
✗ Resets abruptly — feels unfair to users
2. Sliding Window Log
Space: O(requests)  |  Time: O(log N)
Store exact timestamp of each request in a sorted set. On each request, remove timestamps older than the window. Count remaining — if ≥ limit, reject.
✓ Perfectly accurate — no boundary burst
✓ Smooth rate enforcement
✓ Easy to implement "per-rolling-minute"
✗ Memory grows with request count
✗ Slower (ZREMRANGEBYSCORE + ZCARD)
✗ Storing all timestamps is wasteful
3. Sliding Window Counter
Space: O(1)  |  Time: O(1)
Hybrid approach. Keep counters for current and previous window. Approximate current rate as: prev_count × (1 - elapsed/window) + curr_count
✓ O(1) space and time (like fixed window)
✓ ~0.003% error rate — near perfect
✓ Eliminates boundary burst problem
✓ Used by Cloudflare, Nginx
✗ Slight approximation (not exact)
✗ Slightly more complex to implement
4. Token Bucket
Space: O(1)  |  Time: O(1)
Bucket holds up to N tokens. Tokens refill at rate R per second. Each request consumes 1 token. If bucket is empty, request is rejected. Allows controlled bursting.
✓ Allows burst traffic up to bucket size
✓ Smooth average rate enforcement
✓ Most realistic for real-world traffic
✓ Used by AWS, Stripe, GitHub
✗ Requires atomic check+decrement (Lua script)
✗ Need to track last refill timestamp
5. Leaky Bucket
Space: O(queue)  |  Time: O(1)
Requests enter a queue (the bucket). Requests are processed at a fixed output rate (the leak). If queue is full, new requests are dropped. Smooths all traffic to a constant rate.
✓ Perfectly smooth output rate
✓ Great for downstream service protection
✓ Natural queuing of requests
✗ Adds latency (requests wait in queue)
✗ Queue memory overhead
✗ Burst traffic gets queued, not served fast
6. Concurrent Request Limit
Space: O(1)  |  Time: O(1)
Limit how many simultaneous in-flight requests a client can have, regardless of time. Use Redis INCR on entry, DECR on completion. If count ≥ limit, reject immediately.
✓ Prevents slow-request resource exhaustion
✓ Protects against connection pool exhaustion
✓ Complements time-based rate limits
✗ Doesn't prevent high-frequency fast requests
✗ Must decrement on error too (cleanup critical)

Deep Dive

Distributed Rate Limiting with Redis

In a single-server system, a simple in-memory counter works. But with 10 API servers behind a load balancer, each server has its own counter — they're independent. A user could send 10× the limit by splitting requests across servers.

The solution: centralized counter in Redis. All servers read/write the same counter atomically.

// Distributed Rate Limiting Architecture

The Race Condition Problem — Why Lua Scripts Are Essential

The naive approach — check counter, then increment — has a race condition:

🚨
Race Condition: Server A reads counter = 99 (limit = 100). Server B reads counter = 99. Both think they're the last allowed request. Both increment to 100. Both allow their request. Result: 101 requests allowed instead of 100. In high-traffic scenarios with hundreds of concurrent requests, this can bypass limits by 2-10×.

The fix: use Redis Lua scripts which execute atomically (no other command runs between the check and increment), or use Redis transactions (MULTI/EXEC with WATCH).

Implementation

Production Implementations

1. Fixed Window Counter (Simplest)

JAVASCRIPT fixed-window.js 
const redis = require('ioredis');
const client = new redis();

/**
 * Fixed Window Rate Limiter
 * Allows `limit` requests per `windowSeconds`
 * Key resets to 0 at the start of each window
 */
async function fixedWindowRateLimit(identifier, limit, windowSeconds) {
  // Key = identifier + current window (floor to window boundary)
  const window = Math.floor(Date.now() / 1000 / windowSeconds);
  const key = `rl:fw:${identifier}:${window}`;

  // Pipeline: INCR + EXPIRE in one round-trip
  const pipeline = client.pipeline();
  pipeline.incr(key);
  pipeline.expire(key, windowSeconds * 2); // 2× window for safety
  const [[, count]] = await pipeline.exec();

  const remaining = Math.max(0, limit - count);
  const resetAt = (window + 1) * windowSeconds;

  return {
    allowed:   count <= limit,
    count,
    remaining,
    limit,
    resetAt,   // Unix timestamp when window resets
  };
}

// Express middleware
function fixedWindowMiddleware(limit = 100, windowSeconds = 60) {
  return async (req, res, next) => {
    // Identify by user ID (authenticated) or IP (anonymous)
    const identifier = req.user?.id || req.ip;
    const result = await fixedWindowRateLimit(identifier, limit, windowSeconds);

    // Always set rate limit headers (RFC 6585 standard)
    res.set({
      'X-RateLimit-Limit':     limit,
      'X-RateLimit-Remaining': result.remaining,
      'X-RateLimit-Reset':     result.resetAt,
    });

    if (!result.allowed) {
      return res.status(429).json({
        error:      'Too Many Requests',
        retryAfter: result.resetAt - Math.floor(Date.now() / 1000),
        message:    `Rate limit exceeded. Try again in ${result.resetAt - Math.floor(Date.now()/1000)}s`
      });
    }
    next();
  };
}

2. Sliding Window Log (Most Accurate)

JAVASCRIPT sliding-window-log.js 
/**
 * Sliding Window Log using Redis Sorted Set
 * - Score = timestamp (microseconds)
 * - Member = unique request ID (timestamp + random)
 * - Remove entries older than the window
 * - Count remaining = current rate
 */
async function slidingWindowLog(identifier, limit, windowMs) {
  const now     = Date.now();
  const windowStart = now - windowMs;
  const key     = `rl:swl:${identifier}`;
  const reqId   = `${now}-${Math.random()}`;

  // Lua script for atomicity
  const script = `
    local key          = KEYS[1]
    local now          = tonumber(ARGV[1])
    local windowStart  = tonumber(ARGV[2])
    local limit        = tonumber(ARGV[3])
    local reqId        = ARGV[4]
    local windowMs     = tonumber(ARGV[5])

    -- Remove timestamps outside the sliding window
    redis.call('ZREMRANGEBYSCORE', key, '-inf', windowStart)

    -- Count current requests in window
    local count = redis.call('ZCARD', key)

    if count < limit then
      -- Add this request's timestamp
      redis.call('ZADD', key, now, reqId)
      -- Expire key after one window (cleanup)
      redis.call('PEXPIRE', key, windowMs * 2)
      return {1, count + 1}   -- allowed, new count
    else
      return {0, count}        -- rejected, current count
    end
  `;

  const [allowed, count] = await client.eval(
    script, 1, key,
    now, windowStart, limit, reqId, windowMs
  );

  return {
    allowed: allowed === 1,
    count,
    remaining: Math.max(0, limit - count),
    limit
  };
}

3. Token Bucket with Lua Script (Best for APIs)

LUA + JAVASCRIPT token-bucket.js — Atomic via Redis Lua 
/**
 * Token Bucket Rate Limiter
 * - capacity: max tokens (burst size)
 * - refillRate: tokens added per second
 * - Allows bursts up to capacity, then smooth rate
 */
const TOKEN_BUCKET_SCRIPT = `
  local key        = KEYS[1]
  local capacity   = tonumber(ARGV[1])
  local refillRate = tonumber(ARGV[2])   -- tokens per second
  local now        = tonumber(ARGV[3])   -- current time (ms)
  local requested  = tonumber(ARGV[4])   -- tokens needed (usually 1)

  -- Get current state: {tokens, last_refill_time}
  local bucket = redis.call('HMGET', key, 'tokens', 'last_refill')
  local tokens = tonumber(bucket[1]) or capacity
  local lastRefill = tonumber(bucket[2]) or now

  -- Calculate tokens to add based on elapsed time
  local elapsed = math.max(0, now - lastRefill) / 1000  -- seconds
  local newTokens = math.min(capacity, tokens + elapsed * refillRate)

  if newTokens >= requested then
    -- Consume token(s)
    newTokens = newTokens - requested
    redis.call('HMSET', key, 'tokens', newTokens, 'last_refill', now)
    redis.call('PEXPIRE', key, math.ceil(capacity / refillRate) * 2000)
    return {1, math.floor(newTokens), 0}  -- allowed, remaining, retry_after
  else
    -- Not enough tokens — calculate when enough will be available
    local waitMs = math.ceil((requested - newTokens) / refillRate * 1000)
    redis.call('HMSET', key, 'tokens', newTokens, 'last_refill', now)
    redis.call('PEXPIRE', key, math.ceil(capacity / refillRate) * 2000)
    return {0, 0, waitMs}  -- rejected, 0, retry_after_ms
  end
`;

// Load script once (EVALSHA for efficiency)
let tokenBucketSha;
async function loadScript() {
  tokenBucketSha = await client.script('LOAD', TOKEN_BUCKET_SCRIPT);
}

async function tokenBucketLimit(identifier, capacity, refillRate) {
  const key = `rl:tb:${identifier}`;
  const now = Date.now();

  const [allowed, remaining, retryAfterMs] = await client.evalsha(
    tokenBucketSha, 1, key,
    capacity, refillRate, now, 1
  );

  return {
    allowed:      allowed === 1,
    remaining,
    retryAfter:   Math.ceil(retryAfterMs / 1000),
    capacity,
    refillRate
  };
}

4. Express Middleware — Production Grade

JAVASCRIPT rate-limit-middleware.js — Full production middleware 
/**
 * Production rate limiting middleware with:
 * - Per-user AND per-IP limits
 * - Different limits per route/tier
 * - Proper response headers
 * - Graceful degradation if Redis is down
 */
function createRateLimiter({ limit, windowMs, tier = 'default' }) {
  return async (req, res, next) => {
    let result;
    try {
      // Priority: authenticated user > API key > IP
      const identifier = req.user?.id
        || req.headers['x-api-key']
        || req.ip;
      const key = `${tier}:${identifier}`;
      result = await slidingWindowCounter(key, limit, windowMs);
    } catch (err) {
      // Redis down → FAIL OPEN (allow request, log error)
      // Change to FAIL CLOSED for security-critical endpoints
      console.error('Rate limiter Redis error:', err);
      return next(); // Degrade gracefully
    }

    // Standard rate limit headers (RateLimit header spec)
    res.set({
      'RateLimit-Limit':     limit,
      'RateLimit-Remaining': result.remaining,
      'RateLimit-Reset':     result.resetAt,
      'RateLimit-Policy':    `${limit};w=${windowMs/1000}`,
    });

    if (!result.allowed) {
      res.set('Retry-After', result.retryAfter);
      return res.status(429).json({
        error: 'Too Many Requests',
        retryAfter: result.retryAfter,
        docs: 'https://docs.yourapi.com/rate-limits'
      });
    }
    next();
  };
}

// Different limits per endpoint type
const app = express();

// Login: strict (prevent brute force)
app.post('/auth/login',
  createRateLimiter({ limit: 5,   windowMs: 60_000,  tier: 'login'   }),
  loginHandler
);

// General API: tier-based
app.use('/api',
  createRateLimiter({ limit: 1000, windowMs: 60_000,  tier: 'api'     }),
);

// Search: expensive, strict
app.get('/api/search',
  createRateLimiter({ limit: 30,   windowMs: 60_000,  tier: 'search'  }),
  searchHandler
);

// Webhook endpoint: per-sending-service
app.post('/webhooks',
  createRateLimiter({ limit: 10000, windowMs: 60_000, tier: 'webhook' }),
  webhookHandler
);

Response Headers — What to Return

HTTP/1.1 429 Too Many Requests
RateLimit-Limit: 100 // Max requests per window
RateLimit-Remaining: 0 // Requests left in current window
RateLimit-Reset: 1704067260 // Unix timestamp when window resets
Retry-After: 47 // Seconds until retry is safe (required by 429)
RateLimit-Policy: 100;w=60 // 100 requests per 60 second window
// On successful requests, always return these headers too:
RateLimit-Remaining: 53 // So clients can self-throttle
✅
Best Practice: Always return rate limit headers on every response, not just on 429. This lets well-behaved clients self-throttle before hitting the limit. Stripe, GitHub, and Twitter all do this — it reduces 429 errors dramatically.

Nginx Rate Limiting Config

NGINX nginx.conf — Built-in rate limiting 
http {
  # Define rate limit zones (shared memory)
  # Zone name, key (binary IP = smaller), memory size, rate
  limit_req_zone $binary_remote_addr zone=general:10m  rate=100r/m;
  limit_req_zone $binary_remote_addr zone=login:1m    rate=5r/m;
  limit_req_zone $http_x_api_key    zone=api_key:10m rate=1000r/m;

  # Connection limit (concurrent, not rate)
  limit_conn_zone $binary_remote_addr zone=addr:10m;

  server {
    location /api/ {
      # burst=20: queue up to 20 extra requests
      # nodelay: serve burst immediately (don't add delay)
      limit_req zone=general burst=20 nodelay;
      limit_req_status 429;
      limit_conn addr 20; # max 20 concurrent connections per IP
    }

    location /auth/login {
      # Strict: 5/min, no burst allowed
      limit_req zone=login burst=2;
      limit_req_status 429;

      # Custom error page for rate limiting
      error_page 429 /errors/429.json;
    }

    location /api/premium/ {
      # Use API key as identifier (per-key limits)
      limit_req zone=api_key burst=100 nodelay;
      limit_req_status 429;
    }
  }
}
Advanced Patterns

Advanced Rate Limiting Patterns

Different customers get different limits based on their subscription tier. Implemented by looking up the user's tier from their JWT claims or a fast Redis lookup on their API key.

Free
100 req/min
1k req/day
Pro
1000 req/min
100k req/day
Enterprise
10k req/min
Custom

Implementation: decode JWT to get tier claim → lookup tier config → apply tier's limit. The tier config can be stored in Redis (tier:pro:limit = 1000) so it can be updated without deploys.

Instead of static limits, adjust them based on current system health. If CPU > 80% or DB latency > 500ms, automatically tighten limits for lower-tier users while maintaining service for premium users.

Implementation: a background job monitors system metrics and updates rate limit configs in Redis every 5 seconds. The rate limiting middleware reads the current limit from Redis instead of hardcoded values — instant propagation across all gateway instances.

Used by Cloudflare's adaptive rate limiting and AWS API Gateway's throttling under DDoS conditions.

If your API is deployed across US, EU, and Asia data centers, a client could bypass per-region limits by routing requests through different regions. True global rate limiting requires synchronizing counters across DCs.

Options:

1. Centralized Redis: All DCs write to one Redis cluster. Simple but adds latency for distant DCs (EU API calling US Redis adds 100ms).

2. Approximate Global: Each DC tracks local usage. Share counters asynchronously every second. Allows slight over-limit (2-5%) but zero cross-DC latency. Used by Cloudflare.

3. CRDT Counters: Conflict-free Replicated Data Types — counters that can be incremented independently and merged eventually. Redis Enterprise supports this.

Rate Limiting: Rejects excess requests immediately with 429. Client must retry later. Simple, stateless.

Backpressure: Queues excess requests and processes them when capacity is available. Client waits but eventually gets served. More complex — requires a queue (Redis List, SQS) and a processing loop.

Use rate limiting for public APIs (reject fast, protect service). Use backpressure for internal services where it's better to slow down than lose work (batch processing pipelines, email sending queues).

Industry Usage

How Top Companies Implement Rate Limiting

💳 Stripe Token Bucket
Stripe limits: 100 read req/sec, 100 write req/sec per API key. Uses token bucket to allow short bursts. Returns X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset on every response. Their 429 responses include a Retry-After header and their SDKs auto-retry with exponential backoff.
🐙 GitHub Sliding Window
GitHub's REST API: 5,000 requests/hour for authenticated users, 60/hour for unauthenticated. GraphQL API: 5,000 "points" per hour (queries cost different amounts). Their rate limit is per-API-key, not per-IP. They specifically designed point-based limits for GraphQL because a single query can return vastly different amounts of data.
🐦 Twitter/X Per-endpoint limits
Twitter has extremely granular per-endpoint limits: GET /tweets/:id = 300 req/15min, POST /tweets = 300 req/3hour, GET /search/recent = 180 req/15min. Different endpoint types have completely different windows. This prevents abuse of expensive endpoints without limiting cheap reads.
☁️ Cloudflare Sliding Window Counter
Cloudflare's rate limiting product protects millions of websites. They use a sliding window counter approximation (previous + current window weighted) which is O(1) space/time but eliminates the boundary burst problem. Implemented in their edge nodes with local Redis per PoP and gossip protocol for global coordination. Processes 50M+ rate limit decisions per second.
🛒 Shopify Leaky Bucket
Shopify's GraphQL Admin API uses a leaky bucket model: bucket capacity of 1000 "cost points", leaks 50 points/second. Complex queries cost more points. This smooths traffic and prevents merchants' apps from hammering the API during high-volume events (flash sales). Explicitly documented so developers can calculate request costs upfront.
🤖 OpenAI Token + Request limits
OpenAI has two separate rate limits: requests per minute AND tokens per minute. Both must be under the limit. This dual-layer approach prevents both high-frequency short requests AND low-frequency huge requests from overloading their GPU infrastructure. Tier-based: Free=3 RPM/200 TPM, Tier 5=10k RPM/800k TPM.
Tradeoffs

Tradeoffs & Design Decisions

✅ Benefits of Rate Limiting
  • Protects against DDoS and brute force attacks
  • Ensures fair resource distribution across users
  • Enables tiered monetization (free/paid plans)
  • Prevents runaway bugs from taking down production
  • Reduces infrastructure costs from abuse
  • SLA enforcement for downstream services
❌ Challenges & Costs
  • Every request hits Redis — adds ~0.5-2ms latency
  • Shared IP limits penalize users behind NAT/proxies
  • Distributed systems need careful atomic implementation
  • Must handle Redis failures gracefully (fail-open vs fail-closed)
  • Hard to tune limits without affecting legitimate users
  • Clock skew across servers can cause inconsistencies

Fail-Open vs Fail-Closed — Critical Design Choice

🔓
Fail-Open (Allow)
If Redis is down, allow all requests. Prioritizes availability over rate limiting. Good for non-security-critical APIs. Risk: attackers may target your Redis to disable limiting.
🔒
Fail-Closed (Block)
If Redis is down, reject all requests with 503. Prioritizes security over availability. Good for auth endpoints, payment APIs. Risk: Redis outage = total API outage.
📱
Fail-Local
If Redis is down, fall back to per-server in-memory limiting. Each server enforces limit/N (where N = server count). Approximate but functional. Best balance for most production systems.
Engineering Wisdom

Common Mistakes & Best Practices

A corporate office with 5,000 employees might all share a single NAT IP. A strict IP-based rate limit would throttle entire companies based on one employee's usage.

Fix: Always prefer user ID (from auth token) over IP for authenticated endpoints. Use IP only for unauthenticated endpoints (login, registration). Consider user-agent + IP combination for additional granularity on public endpoints.

Without app.set('trust proxy', 1) in Express, req.ip returns the load balancer's private IP for every request. All users share one rate limit bucket → everyone hits limits together.

Fix: Set trust proxy, and use req.ip (which then correctly returns the X-Forwarded-For client IP). In Nginx: rate limit on $http_x_forwarded_for instead of $remote_addr.

For concurrent request limiting (INCR on start, DECR on end), if the server crashes or throws an unhandled exception, the DECR never runs. Over time, users get permanently stuck at the limit even when making no requests.

Fix: Always use try/finally to guarantee decrement: try { INCR; process() } finally { DECR }. Also set a TTL on the counter key as a safety net for crash recovery.

When a client receives 429, they should not immediately retry — that makes the problem worse. The server is already overloaded, and instant retries just add more pressure.

Correct pattern: Read the Retry-After header and wait that long. If not present: exponential backoff with jitter: wait = min(cap, base × 2^attempt) + random(0, base). E.g., 1s, 2s, 4s, 8s, 16s, max 60s. AWS SDKs, Stripe SDKs, and Axios retry plugins implement this correctly.

Interview Preparation

Common Interview Questions

Design a rate limiter for an API that handles 1 million requests per second.
At 1M RPS: (1) Use sliding window counter algorithm — O(1) space/time, no boundary bursts. (2) Redis Cluster for the counter store — shard by user ID hash. (3) Use Lua scripts for atomic check+increment. (4) Local in-memory cache per API server — cache rate limit decisions for 100ms. If a user is well under the limit, don't hit Redis on every request. If near the limit, check Redis every request. This "local fast path" reduces Redis load by 80-90%. (5) Fail-local (in-memory fallback) if Redis is unavailable. (6) Each Redis node handles ~200k INCR/s comfortably — at 1M RPS you'd need 5+ Redis nodes in cluster. Response target: <1ms overhead for rate limit check.
What's the difference between Token Bucket and Leaky Bucket?
Token Bucket: tokens accumulate over time up to capacity. Each request consumes tokens. If tokens available → serve immediately. Allows bursts (consume many tokens at once). Rate of consumption is variable, rate of token addition is fixed. Leaky Bucket: requests enter a queue (the bucket). Processed at a fixed rate (the leak). Output is always smooth. Bursts are queued, not served immediately. If queue is full, requests are dropped. Key difference: Token Bucket allows bursting (serve N requests instantly if N tokens available). Leaky Bucket smooths output to a constant rate regardless of input. Use Token Bucket for APIs that should allow occasional bursts (user browsing, mobile apps). Use Leaky Bucket to protect downstream services that can't handle traffic spikes.
How do you handle rate limiting in a multi-region deployment?
Three approaches with different tradeoffs: (1) Global centralized Redis: accurate but adds cross-region latency (100ms for distant regions). Unacceptable for most APIs. (2) Per-region limits: each region has limit/N. Simple, zero added latency. But an attacker using a VPN can hit N× the global limit. Good enough for most cases. (3) Approximate global with gossip: each region increments local counter + publishes deltas to other regions asynchronously (~1s lag). Global count = sum of all regional counts. Allow slight over-limit (< 5%) in exchange for zero cross-region latency. This is how Cloudflare does it at scale. For security-critical endpoints (auth, payment), use approach 1 and accept the latency, or deploy a centralized auth service in each region with synchronization.
Why is the Fixed Window algorithm flawed and how does Sliding Window fix it?
Fixed Window flaw: the boundary burst problem. With a 100 req/min limit using fixed windows (minute boundaries), a user can send 100 requests at 11:59:59 and 100 more requests at 12:00:01 — that's 200 requests in 2 seconds, which is 2× the intended rate, while both windows only show 100 requests. Sliding Window Log fix: instead of resetting at minute boundaries, track the exact timestamp of each request. Count only requests within the last 60 seconds. At 12:00:01, the 100 requests from 11:59 are all within the last 60 seconds — they count. The user is already at 100 and gets throttled. No boundary burst. Sliding Window Counter approximation: prev_window_count × (1 - elapsed_in_current_window / window_size) + current_window_count. Near-perfect accuracy (0.003% error) with O(1) space.
How would you implement rate limiting without Redis (interview follow-up)?
Single server: use in-memory Map<userId, {count, resetAt}> with a setInterval cleanup job to remove expired entries. Thread-safe with a mutex (or use Node.js single-threaded nature). Multi-server without Redis: (1) Sticky sessions (IP hash load balancing) — same user always goes to same server, which has their counter. Breaks on server failure. (2) Each server tracks local count + gossip to peers every second — approximate. (3) Use database (PostgreSQL) with atomic UPDATE ... RETURNING — slower (~10ms) but works across servers. Not suitable for high-traffic. (4) Use Hazelcast, Memcached, or any distributed cache instead of Redis. Redis is just the most common choice because it has atomic INCR, Lua scripting, and sub-millisecond latency built in.
Summary
🎯 What You Now Know
  • ✦ Rate limiting controls request frequency to protect APIs from abuse, DDoS, and overload
  • ✦ Fixed Window: simple, fast, but has boundary burst flaw — 2× the limit possible at window edges
  • ✦ Sliding Window Log: perfect accuracy, but O(n) memory — store each request timestamp in Redis ZSet
  • ✦ Sliding Window Counter: best of both — O(1) space with ~99.997% accuracy (Cloudflare's choice)
  • ✦ Token Bucket: allows controlled bursting — great for real-world API clients (Stripe, AWS, GitHub)
  • ✦ Leaky Bucket: smooths output to constant rate — great for protecting downstream services
  • ✦ Distributed: centralize counters in Redis with Lua scripts for atomicity — no race conditions
  • ✦ Always return RateLimit-* headers so clients can self-throttle before hitting 429
  • ✦ Key decisions: per-user vs per-IP, fail-open vs fail-closed, tiered limits, global vs per-region
