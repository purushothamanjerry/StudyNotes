Microservices

The architectural style that powers Netflix, Uber, Amazon and every modern tech giant. Learn why monoliths break at scale, how to decompose them into services, how services communicate, and the real organisational and technical trade-offs that nobody tells you about.

Beginner → Advanced

Architecture Core
Introduction

What are Microservices?

Microservices is an architectural style where a large application is built as a collection of small, independently deployable services. Each service owns a single business capability, runs in its own process, communicates over a network, and can be developed, deployed, and scaled independently.

It is the opposite of a monolith — instead of one big codebase that does everything, you have many focused services each doing one thing well.

🏙️ Analogy — City vs Single Building

A monolith is like a single skyscraper that houses every business in the city — shops, hospitals, schools, offices, all in one building. Adding a new floor is risky and affects everyone. If the power goes out, everyone is affected. Microservices is like a city — each building is separate, has its own power, its own rules. A fire in one building doesn't burn the whole city. You can demolish and rebuild one building without touching others. Each can grow independently. The trade-off: now you need roads (APIs), traffic management (service mesh), and city planning (architecture governance).

Imagine Amazon.com as a monolith — one huge application handling search, checkout, payments, recommendations, user accounts, inventory, and shipping. When the shopping cart feature needs an update, you re-deploy the entire application. If the recommendation engine has a memory leak, the checkout page crashes too.

With microservices: each function is a separate service with its own team and codebase. The Checkout team deploys their service 20 times a day without touching Search or Recommendations. A bug in Recommendations doesn't take down the checkout page.

Microservices implement Conway's Law deliberately — the architecture mirrors the team structure. Each service is owned by a two-pizza team (6-8 people) who have full autonomy over technology choices, deployment cadence, and internal implementation as long as they honour the published API contract.

The key principles: bounded context (from Domain-Driven Design) defines service boundaries around business capabilities; database-per-service ensures true isolation; smart endpoints, dumb pipes means business logic lives in services, not in the communication layer; and design for failure treats every remote call as potentially failing.

Design Principles

How to Decompose a Monolith — Service Boundaries

The hardest question in microservices: how do you decide where to draw the service boundary? Cut too fine and you have a distributed monolith. Cut too coarse and services are still tightly coupled.

This is the most reliable decomposition strategy. Identify the business capabilities your system provides, then build one service per capability.

For an e-commerce platform: User Management, Product Catalog, Shopping Cart, Order Management, Payment Processing, Inventory Management, Shipping, Notifications, Search, Reviews. Each becomes a service.

Bounded Context (DDD): Each service has its own model of "what an order is." The Order Service cares about order status and line items. The Shipping Service only cares about delivery address and weight. They have different internal models of the same concept — and that's correct.

Conway's Law: "Organizations which design systems are constrained to produce designs which are copies of the communication structures of those organizations."

Amazon enforced this with the "two-pizza team" rule — if a team can't be fed by two pizzas (6-8 people), it's too large. Each team owns and is fully responsible for their service(s), including on-call, deployments, and architecture decisions.

The inverse Conway Maneuver: restructure your teams to match the microservice architecture you want to achieve. The architecture will follow.

Separate things that change often from things that change rarely. If the Recommendation Engine algorithm changes 10 times per day (A/B tests) but the User Profile rarely changes, keep them separate — you don't want 10 deployments per day to risk the User service.

Group services by their rate of change and risk profile. Payments changes rarely and is high-risk → isolated, heavily tested, slow deployment cadence. Search changes frequently, lower risk → fast deployment, experimentation-friendly.

A service is too small when: it can't be independently deployed (it always deploys with another service), it has no meaningful business capability on its own, or managing it (CI/CD, monitoring, on-call) costs more than the value it provides.

Nano-services anti-pattern: A service that just wraps a single function call. The network overhead, service mesh complexity, and operational burden dwarf any benefit.

Good rule of thumb: A service should be something a team of 4-8 people can fully understand, develop, and own. It should have a clear business purpose. If you can't explain its responsibility in one sentence, consider merging it.

Inter-Service Communication

How Microservices Talk to Each Other

When you decompose a monolith, what used to be a function call orderService.getOrder(id) becomes a network call. Network calls can fail, time out, or be slow. This is the biggest source of complexity in microservices.

// Communication Patterns — Sync vs Async

Service-to-Service Communication Patterns

The most common choice for external APIs and many internal service calls. Human-readable, universally supported, easy to debug with curl/Postman.

Best practice: Use versioned URLs (/v1/orders), return consistent error shapes, include correlation IDs in every request. Use HTTP/2 for keep-alive connections between services to avoid TCP handshake overhead on every call.

When to prefer REST: External APIs, heterogeneous services (different languages), when developer experience matters, when you need human-readable payloads for debugging.

Google's RPC framework. Uses Protocol Buffers (binary serialization) and HTTP/2 for multiplexed streaming. 5-10× faster and 3-5× smaller payloads than REST/JSON for internal calls.

Key advantages: Strongly typed contracts (proto files), bidirectional streaming, client/server code generation in any language, built-in load balancing and retries.

When to prefer gRPC: Internal service-to-service calls where performance matters, polyglot systems where you want type safety across languages, streaming use cases. Used by Netflix, Uber, Google internally.

A service mesh injects a sidecar proxy (Envoy) alongside every service container. All inter-service traffic goes through this proxy. The mesh handles: mTLS encryption, load balancing, retries, circuit breaking, distributed tracing — without any code changes in the service itself.

How it works: Order Service → Envoy sidecar → encrypted traffic → Envoy sidecar → Payment Service. The services just call each other normally; the sidecars handle all the reliability infrastructure.

When to use: Large microservice deployments (50+ services) where you need consistent observability, security, and traffic management without instrumenting every service individually.

Data Architecture

Database-per-Service — The Most Controversial Rule

The most counter-intuitive microservices rule: each service must own its own database. No two services share a database. This is the rule most teams break first — and the one that causes the most pain when broken.

💥
Why Shared DB Breaks Microservices
If Services A and B share a DB, they share schema. A schema change by Team A breaks Team B. Now you need cross-team coordination for every DB change. You've rebuilt the monolith's coupling, just in a different place.
🏛️
Polyglot Persistence
Each service picks the right DB for its data. User Service: PostgreSQL (relational). Search Service: Elasticsearch (full-text). Session Service: Redis (in-memory). Cart Service: DynamoDB (key-value). This freedom is a superpower.
🔄
The Join Problem
Without a shared DB, you can't do SQL JOINs across services. Solution: compose data in the API Gateway / BFF layer, or use event sourcing to maintain read-model denormalized views that join data for queries.
📋
CQRS Pattern
Command Query Responsibility Segregation: separate read and write models. Write to the authoritative service's DB; maintain denormalized read views in other services updated via events. Query the read view, not the source.
⚠️
The Saga Pattern for Distributed Transactions: Without a shared DB you can't use ACID transactions across services. Instead use Sagas — a sequence of local transactions, each publishing an event. If one step fails, compensating transactions undo the previous steps. Example: Order saga: reserve inventory → charge payment → confirm order. If payment fails: release inventory reservation. Covered in detail in Topic 19.
Deployment Patterns

Deployment Strategies for Microservices

1
Containerisation with Docker
Each microservice packages its code, runtime, dependencies, and config into a Docker container. Containers are immutable — the same image runs identically in dev, staging, and production. Enables the "build once, run anywhere" promise.
2
Orchestration with Kubernetes
Kubernetes manages container deployment, scaling, health checking, and rolling updates across a cluster of machines. Declares desired state: "Run 5 instances of Order Service, restart if unhealthy, expose on port 3000." K8s makes it so.
3
Blue-Green Deployment
Run two identical environments (Blue = current, Green = new). Deploy new version to Green, run smoke tests, then switch the load balancer from Blue to Green in seconds. Instant rollback: flip the load balancer back to Blue. Zero downtime.
4
Canary Deployment
Route 5% of traffic to the new version, 95% to old. Monitor error rates and latency. If metrics look good, gradually increase to 10%, 25%, 50%, 100%. If metrics worsen, instantly route 100% back to old version. Used by Netflix, Google.
5
CI/CD Pipeline per Service
Each service has its own pipeline: code commit → unit tests → integration tests → build Docker image → push to registry → deploy to staging → smoke tests → deploy to production. Teams deploy independently without coordinating with others.

Inter-Service HTTP Call with Circuit Breaker

JAVASCRIPT lib/service-client.js — Resilient inter-service call 
const axios      = require('axios');
const CircuitBreaker = require('opossum');

/**
 * Resilient HTTP client for inter-service calls
 * Features: timeout, retry, circuit breaker, correlation ID
 */
function createServiceClient(baseURL, options = {}) {
  const http = axios.create({
    baseURL,
    timeout: options.timeout || 5000,   // 5s timeout — ALWAYS SET THIS
    headers: { 'Content-Type': 'application/json' },
  });

  // Add correlation ID to every outbound request
  http.interceptors.request.use((config) => {
    config.headers['X-Correlation-ID'] =
      global.currentCorrelationId || require('uuid').v4();
    return config;
  });

  // Circuit Breaker wraps the HTTP function
  const breaker = new CircuitBreaker(
    async (method, path, data) => http[method](path, data),
    {
      timeout:           5000,   // Call timeout
      errorThresholdPercentage: 50,  // Open after 50% failures
      resetTimeout:      10000,  // Try again after 10s
      volumeThreshold:   5,      // Min requests before tripping
    }
  );

  // Fallback: return null when circuit is open
  breaker.fallback((method, path) => {
    console.warn(`Circuit open for ${baseURL}${path}`);
    return null;
  });

  breaker.on('open',     () => console.error(`⚡ Circuit OPEN: ${baseURL}`));
  breaker.on('halfOpen', () => console.info(`🔁 Circuit HALF-OPEN: ${baseURL}`));
  breaker.on('close',    () => console.info(`✅ Circuit CLOSED: ${baseURL}`));

  return {
    get:    (path) => breaker.fire('get', path),
    post:   (path, data) => breaker.fire('post', path, data),
    put:    (path, data) => breaker.fire('put', path, data),
    delete: (path) => breaker.fire('delete', path),
  };
}

// Usage in Order Service
const userSvc    = createServiceClient(process.env.USER_SVC_URL);
const paymentSvc = createServiceClient(process.env.PAYMENT_SVC_URL);

async function createOrder(userId, items) {
  // Get user info — returns null if User Service is down
  const user = await userSvc.get(`/users/${userId}`);
  if (!user) throw new Error('User service unavailable');

  const order = await db.createOrder({ userId, items });
  await kafka.publish('order.created', { orderId: order.id, userId, items });
  return order;
}
Industry Usage

How Top Companies Use Microservices

🎬 Netflix 1000+ services
Netflix pioneered microservices starting in 2008 after a major DB corruption incident. Today they run 1,000+ microservices globally. Key innovations: Eureka (service discovery), Hystrix (circuit breaker), Zuul (API gateway), Chaos Monkey (deliberate failure injection). Their engineering blog is a masterclass in microservice patterns at scale.
🛒 Amazon Two-pizza teams
Jeff Bezos issued the "API mandate" around 2002: all teams must expose their data through service interfaces. No back-door DB sharing. This forced microservice decomposition across Amazon. The result: AWS was born — they productised their internal service infrastructure. Every AWS service started as an internal Amazon microservice.
🚗 Uber DOMA Architecture
Uber went from 1 monolith to 2,200+ microservices. Then suffered from microservices chaos (too many services, complex dependencies). They introduced DOMA (Domain-Oriented Microservice Architecture) — grouping related services into domains with well-defined boundaries. Now ~40 domains each containing 10-50 services, with strict API contracts between domains.
🛍️ Shopify Modular Monolith First
Shopify famously stayed with a modular monolith for years while most companies rushed to microservices. They introduced strict module boundaries within the Rails monolith before extracting services. Lesson: first build the right modular boundaries, then extract to microservices when the pain is real and boundaries are proven.
🐦 Twitter SOA Migration
Twitter migrated from monolith ("The Monorail") to microservices in 2012. The Timeline Service was the most complex — fan-out on write for 300M users required a dedicated caching service (FlockDB for social graph, Gizzard for distributed storage). Their approach: decompose by bottleneck, not by dogma.
🏦 Monzo 1500+ Go services
Monzo (UK neobank) started with microservices from day one — 1,500+ Go microservices on Kubernetes running the entire bank. Each service handles one banking function (accounts, payments, notifications, cards). Their entire core banking platform is Kubernetes-native, deployed hundreds of times per day without downtime.
Tradeoffs

The Full Truth — Advantages & Real Costs

✅ Real Advantages
  • Independent deployability — deploy any service any time
  • Independent scalability — scale only the bottleneck
  • Fault isolation — one service crash doesn't cascade
  • Technology freedom — best tool per job
  • Team autonomy — each team fully owns their service
  • Faster onboarding — small focused codebases
  • Easier to adopt new technologies incrementally
❌ Real Costs Nobody Mentions
  • Distributed systems complexity — network failures everywhere
  • No ACID transactions across services — eventual consistency
  • Much harder debugging — trace requests across 10 services
  • Operational overhead — run/monitor dozens of services
  • Data consistency between services is your problem
  • Testing is harder — need contract tests, integration tests
  • Latency added by every network hop between services

The "Microservices Premium" — Don't Pay It Early

🚨
Martin Fowler's Warning — "Microservice Premium": Microservices add significant overhead. For small teams or early-stage products, the overhead exceeds the benefits. Start with a well-structured monolith (or modular monolith) with clear module boundaries. Extract to microservices only when you have: (1) organisational scale (multiple teams), (2) deployment frequency conflicts, (3) scaling needs for specific components. Premature microservices is one of the most common startup mistakes.

Engineering Wisdom

Common Mistakes & Best Practices

The most common mistake — two services reading/writing the same tables. Looks convenient but couples services at the data layer. Any schema change requires coordinating both teams. A performance issue in one service's queries degrades the other.

Fix: Database-per-service, strictly. If Service A needs data from Service B's domain, call Service B's API, subscribe to Service B's events, or maintain a denormalised read model updated via events.

Service A calls Service B with no timeout. Service B gets slow (DB issue). Service A's threads pile up waiting. Service A runs out of thread pool → Service A is now down. One slow downstream service has cascaded into a total outage.

Fix: Always set timeouts on every HTTP call to other services. Add circuit breakers (Hystrix, Opossum, Resilience4j) that open when error rates exceed a threshold, preventing the cascade. This is covered in depth in Topic 18.

Teams go overboard and create services for every single function — a "GetUser" service, a "ValidateEmail" service, a "FormatCurrency" service. Now a single user request requires 15 network hops across 15 services. Latency is terrible, debugging is impossible.

Fix: Services should map to business capabilities, not technical functions. If two functions always change together and are never deployed separately, they belong in the same service. The test: can this service be developed, deployed, and operated independently by its own team?

In a monolith, if you change a function signature, the compiler tells you immediately. In microservices, if Service A changes its API and Service B depends on it, you don't find out until runtime — possibly in production.

Consumer-driven contract testing (Pact): Service B (consumer) defines what it expects from Service A's (provider) API. This contract is shared with Service A's team. Service A's CI/CD runs the contract tests on every build. If Service A breaks the contract, the build fails — before deployment. Catches breaking changes at build time, not at 3am production incident time.

Without distributed tracing, debugging a slow request that touched 8 services is nearly impossible. "Which of the 8 services is slow? What database query is taking 2 seconds?"

OpenTelemetry + Jaeger/Zipkin: Generate a trace ID at the edge (API Gateway). Pass it as a header through every service call. Every service logs its spans (start time, duration, service name) tagged with the trace ID. Visualise the entire request journey in Jaeger — see exactly which service and which operation added latency. Essential for microservices at any serious scale.

Interview Preparation

Common Interview Questions

What is the difference between microservices and a monolith? When would you choose each?
A monolith is a single deployable unit where all components share the same process and database. Microservices decompose this into independently deployable services each owning their data. Choose monolith when: small team (<15 engineers), early-stage product where domain boundaries aren't clear yet, no dedicated DevOps. Choose microservices when: multiple teams needing independent deployment, specific components needing independent scaling, stable well-understood domain boundaries. The mistake is treating microservices as inherently superior — they add massive operational overhead and distributed systems complexity. Martin Fowler calls this the "microservice premium." Start with a well-structured monolith and extract services when you feel real pain at module boundaries.
How do microservices handle distributed transactions? What is the Saga pattern?
Microservices can't use traditional ACID transactions across services because each has its own database. Instead, use Sagas — a sequence of local transactions coordinated through events or orchestration. Choreography-based Saga: each service publishes events when done, the next service reacts. Order Service publishes ORDER_CREATED → Payment Service charges card and publishes PAYMENT_SUCCESS → Shipping Service schedules delivery. If Payment fails: publishes PAYMENT_FAILED → Order Service cancels the order (compensating transaction). Orchestration-based Saga: a Saga orchestrator tells each service what to do step by step. If a step fails, the orchestrator calls compensating transactions on already-completed steps. Key trade-off: eventual consistency — the system is temporarily inconsistent but eventually reaches a consistent state. Covered in detail in Topic 19.
How do you handle service discovery in microservices?
Service discovery solves: "Where is the Payment Service running right now?" Services are dynamically scaled and deployed — IPs change constantly. Two patterns: Client-side discovery: each service queries a service registry (Consul, Eureka) to get healthy instances and load-balances itself. Server-side discovery: a load balancer (AWS ALB, Kubernetes Service) handles discovery — services just call a static DNS name and the LB routes to healthy instances. In Kubernetes, every Service gets a stable DNS name (payment-service.default.svc.cluster.local) that resolves to healthy pods — built-in server-side discovery. Covered in depth in Topic 10.
How do you prevent cascading failures in microservices?
Three main patterns: (1) Timeout — every inter-service call must have a timeout. If Payment Service doesn't respond in 5s, return an error. Don't wait forever. (2) Circuit Breaker — after N% failures within a time window, "open" the circuit: stop calling the failing service and return fallback immediately. After a reset timeout, send one probe request. If it succeeds, close the circuit. Library: Resilience4j, Hystrix, Opossum. (3) Bulkhead — isolate failures by using separate thread pools or connection limits per downstream service. If Payment Service is slow and exhausts its dedicated thread pool, it doesn't affect the thread pool serving Inventory Service calls. Also: implement fallbacks (return cached data, default values, or graceful degradation) when a service is unavailable rather than propagating errors up the chain. Detailed in Topic 18.
Design a microservices architecture for an e-commerce platform.
Services: User Service (auth, profiles — PostgreSQL), Product Catalog (search, listings — Elasticsearch + PostgreSQL), Shopping Cart (session-based — Redis), Order Service (order lifecycle — PostgreSQL), Payment Service (Stripe integration — MySQL), Inventory Service (stock management — PostgreSQL), Notification Service (email/SMS — Redis + SNS), Shipping Service (carrier APIs — PostgreSQL). Communication: API Gateway routes all client requests, validates JWT. Sync (REST/gRPC) for user-facing calls (get product, place order). Async (Kafka) for events (order.created → payment, inventory, notification, shipping all subscribe). Database-per-service: no shared databases. Consistency via Saga: Order saga coordinates payment + inventory reservation atomically using compensating transactions. Deployment: each service in Docker container, Kubernetes for orchestration with HPA for auto-scaling. Observability: OpenTelemetry for distributed tracing, Prometheus for metrics, ELK for logs.
Summary
🎯 What You Now Know
  • ✦ Microservices = collection of small, independently deployable services each owning a business capability
  • ✦ Decompose by bounded context (DDD), team ownership (Conway's Law), or change frequency
  • ✦ Database-per-service: never share databases — it recreates monolith coupling at the data layer
  • ✦ Communication: REST for simplicity, gRPC for performance, events (Kafka) for async fan-out
  • ✦ Always: timeouts + circuit breakers on every inter-service call to prevent cascade failures
  • ✦ Distributed transactions → Saga Pattern (choreography or orchestration)
  • ✦ Deployment: Docker + Kubernetes + HPA + rolling updates + canary deployments
  • ✦ Observability: distributed tracing (OpenTelemetry) is non-negotiable — correlation IDs through every hop
  • ✦ Don't start with microservices — start with a modular monolith and extract when boundaries are proven
