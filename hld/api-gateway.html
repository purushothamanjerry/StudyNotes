API Gateway

The single front door to your entire microservices backend. Learn how API gateways handle authentication, rate limiting, routing, transformation, and observability — so your services don't have to.

Beginner → Intermediate
~40 min read
Production Ready
Interview Critical
Introduction

What is an API Gateway?

An API Gateway is a server that acts as the single entry point for all client requests in a microservices architecture. It receives requests, applies cross-cutting concerns (auth, rate limiting, logging), and routes them to the appropriate downstream microservice.

Think of it as the receptionist at a large company HQ — every visitor checks in here, gets verified, gets a visitor badge, and is then directed to the right department. No visitor wanders directly into engineering or finance without going through reception first.

🏢 Real-World Analogy — Airport Security

An airport has one security checkpoint all passengers must pass through, regardless of which gate they're heading to. The checkpoint: verifies your identity (authentication), checks your ticket (authorization), scans your luggage (request inspection), and directs you to the right terminal (routing). You don't go through security again at each gate. That checkpoint is your API Gateway.

Imagine you have 10 microservices: User Service, Order Service, Payment Service, etc. Without an API Gateway, clients need to know the address of every single service. They also have to handle authentication themselves for each service.

With an API Gateway: clients send ALL requests to one address (e.g. api.yourapp.com). The gateway checks auth once, then routes to the right service. Simple, clean, and centralized.

An API Gateway implements the Backend for Frontend (BFF) pattern at its core. It handles cross-cutting concerns as middleware pipelines — each request passes through a chain of plugins: JWT validation → rate limit check → request transformation → service routing → response transformation → logging.

Modern gateways like Kong, AWS API GW, and Envoy support plugin architectures (written in Lua, Go, or Wasm) for extensibility. They maintain an upstream connection pool (like an L7 load balancer) and support circuit breaking, retries, and timeouts per route.

Problem Statement

Why Does an API Gateway Exist?

Before API Gateways, microservice architectures created a "cross-cutting concern explosion". Every team had to implement the same boilerplate in every service:

🔐
Auth Duplication
Every service must validate JWT tokens, check user roles, and decode session data. 20 services = 20 copies of the same auth code, each potentially different and buggy.
🌐
Client Complexity
Mobile apps need to know 20 different service endpoints. Manage 20 different TLS certs. Handle 20 different error formats. This is unmaintainable at scale.
📊
Scattered Observability
No single place to see total API traffic, error rates, or latency. Logs are in 20 different places. Debugging a user's request journey is nearly impossible.
🔄
Protocol Mismatch
Clients speak HTTP/REST. Internal services may use gRPC, GraphQL, or AMQP. The gateway translates between external and internal protocols transparently.
💡
Key Insight: An API Gateway solves the DRY principle at the infrastructure level. Instead of implementing auth, rate limiting, and logging in every service, you implement it once at the gateway. Services stay lean and focused on business logic.
Architecture

Architecture & Full Request Flow

// API Gateway — Complete Request Pipeline

Request Flow — Every Request Goes Through This Pipeline

1
SSL Termination + Parsing
HTTPS connection terminates at the gateway. The request is parsed: method, path, headers, body. The path determines which routing rule applies.
2
Authentication — Who are you?
The gateway validates the identity token: JWT signature check, OAuth introspection, or API key lookup. If invalid → immediate 401 Unauthorized response, request never reaches services.
3
Authorization — Can you do this?
Even if you're authenticated, do you have permission? RBAC check: does this user's role include permission for DELETE /api/users/:id? Reject with 403 Forbidden if not.
4
Rate Limiting — Throttle if needed
Check rate limit counters in Redis. If the user/API key has exceeded their quota (e.g. 1000 req/min), return 429 Too Many Requests with Retry-After header.
5
Request Transformation
Rewrite headers, inject user context (user ID, roles from token), strip sensitive headers, transform request body format if needed (REST → gRPC). Add correlation/trace IDs.
6
Route to Microservice
Forward the request to the correct upstream service based on routing rules. The gateway load-balances across multiple instances of that service.
7
Response Transformation + Logging
Receive response from service. Optionally transform it (strip internal fields, format errors). Log request/response metadata with trace ID. Return to client.

The gateway decodes the JWT from Authorization: Bearer <token>, verifies the signature using the public key (RS256) or shared secret (HS256), and checks expiry. On success, it extracts claims (user ID, roles) and injects them as headers for downstream services:

X-User-ID: 123, X-User-Role: admin, X-User-Email: user@app.com

Services trust these injected headers from the gateway — they don't need to decode JWT themselves. This is safe because services are only accessible internally through the gateway.

Token caching: The gateway caches validated tokens in Redis for ~60 seconds to avoid re-validating every single request (especially important for JWTs that require a JWKS endpoint fetch).

Fixed Window: Allow 100 requests per minute. Counter resets at the top of each minute. Simple but allows "burst at boundary" — 100 req at 11:59 + 100 req at 12:00 = 200 req in 2 seconds.

Sliding Window Log: Store timestamps of all requests in Redis. Count how many fall within last 60 seconds. More accurate but memory-intensive.

Token Bucket: Tokens refill at a constant rate. Each request consumes one token. Burst allowed up to bucket capacity. Used by AWS, Stripe.

Leaky Bucket: Requests queued and processed at a constant output rate. Smooths traffic bursts. Used in network QoS.

Path-based routing: /api/users/* → User Service, /api/orders/* → Order Service. Most common pattern.

Header-based routing: Route based on X-API-Version: v2 header. Allows versioning without URL changes. Accept: application/vnd.api.v2+json → v2 service.

Weight-based (Canary): Send 5% of traffic to new service version, 95% to stable. Roll forward or back based on error rates. Used for zero-risk deployments.

A/B testing: Route user IDs ending in 0-4 to variant A, 5-9 to variant B. Deterministic assignment by hashing user ID.

External clients send standard HTTP/REST JSON. Internal services run gRPC (Protocol Buffers over HTTP/2) for performance. The gateway translates between them automatically using grpc-gateway or Envoy's gRPC-JSON transcoding.

Client sends: POST /api/users {"name": "Alice"}
Gateway translates to: UserService.CreateUser(CreateUserRequest{name: "Alice"}) via gRPC.

This lets your internal architecture use the fastest protocols while keeping the external API clean and standard.

          minute: 100

          redis_port: 6379

  - name: payment-service
    url: http://payment-service:3003
    routes:
      - name: payment-routes
        paths: [/api/payments]
        methods: [POST]
    plugins:
      - name: jwt
      - name: rate-limiting
        config:
          minute: 10   # Much stricter for payments
          policy: redis

Node.js — Building a Lightweight Custom Gateway

JAVASCRIPT gateway.js — Custom Express API Gateway 
const express     = require('express');
const { createProxyMiddleware } = require('http-proxy-middleware');
const jwt         = require('jsonwebtoken');
const rateLimit   = require('express-rate-limit');
const RedisStore   = require('rate-limit-redis');
const redis        = require('ioredis');
const { v4: uuidv4 } = require('uuid');

const app         = express();
const redisClient = new redis({ host: 'redis', port: 6379 });

// ─── 1. Correlation ID Middleware ───────────────────────
app.use((req, res, next) => {
  req.correlationId = req.headers['x-correlation-id'] || uuidv4();
  res.setHeader('X-Correlation-ID', req.correlationId);
  next();
});

// ─── 2. JWT Authentication Middleware ───────────────────
const authenticate = async (req, res, next) => {
  const authHeader = req.headers['authorization'];
  if (!authHeader?.startsWith('Bearer '))
    return res.status(401).json({ error: 'Missing token' });

  const token = authHeader.slice(7);

  // Check Redis cache first (avoid repeated JWT decoding)
  const cached = await redisClient.get(`token:${token.slice(-16)}`);
  if (cached) {
    req.user = JSON.parse(cached);
    return next();
  }

  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.user = { id: decoded.sub, role: decoded.role, email: decoded.email };

    // Cache for 60 seconds
    await redisClient.setex(
      `token:${token.slice(-16)}`, 60,
      JSON.stringify(req.user)
    );
    next();
  } catch (err) {
    res.status(401).json({ error: 'Invalid or expired token' });
  }
};

// ─── 3. Rate Limiting (Redis-backed, per user) ──────────
const apiLimiter = rateLimit({
  windowMs: 60 * 1000,
  max: 100,
  standardHeaders: true,
  legacyHeaders: false,
  store: new RedisStore({ client: redisClient }),
  keyGenerator: (req) => req.user?.id || req.ip,   // per-user limits
  handler: (req, res) =>
    res.status(429).json({
      error: 'Too many requests',
      retryAfter: res.getHeader('RateLimit-Reset')
    })
});

// ─── 4. Inject User Context into upstream headers ───────
const injectUserContext = (req, res, next) => {
  req.headers['x-user-id']    = req.user?.id;
  req.headers['x-user-role']  = req.user?.role;
  req.headers['x-user-email'] = req.user?.email;
  // Remove raw JWT so services don't re-validate (trust gateway)
  delete req.headers['authorization'];
  next();
};

// ─── 5. Route Definitions ───────────────────────────────
const routes = [
  { prefix: '/api/users',    target: 'http://user-service:3001'    },
  { prefix: '/api/orders',   target: 'http://order-service:3002'   },
  { prefix: '/api/payments', target: 'http://payment-service:3003' },
  { prefix: '/api/notify',   target: 'http://notif-service:3004'   },
];

routes.forEach(({ prefix, target }) => {
  app.use(
    prefix,
    authenticate,
    apiLimiter,
    injectUserContext,
    createProxyMiddleware({
      target,
      changeOrigin: true,
      on: {
        error: (err, req, res) => {
          console.error(`[${req.correlationId}] Upstream error: ${err.message}`);
          res.status(502).json({ error: 'Bad Gateway', id: req.correlationId });
        }
      }
    })
  );
});

app.listen(8080, () => console.log('🚪 API Gateway running on :8080'));

Design Patterns

API Gateway Patterns

Backend for Frontend (BFF) Pattern

Instead of one generic API Gateway, create purpose-built gateways per client type. The mobile app has different data needs than the web app — different response shapes, different auth flows, different caching TTLs.

// BFF Pattern — One Gateway Per Client Type
✅
When to use BFF: Use BFF when your mobile app, web app, and partner API have significantly different data shapes and auth requirements. Netflix uses BFF for their device-specific APIs (TV, mobile, browser all have different gateway logic).

Industry Usage

How Top Companies Use API Gateways

🎬 Netflix Zuul → Zuul 2
Netflix built Zuul, their open-source API Gateway. Zuul 2 uses Netty for non-blocking I/O, handling 100+ billion requests per day. They use it for dynamic routing, stress testing (inject failures), AB testing, and authentication. Each filter in Zuul is a separate microservice that can be pushed live without gateway restart.
🚗 Uber NGINX + custom
Uber processes 2M+ trips per day through their gateway. They use it to solve the "1000 client types" problem — drivers, riders, restaurants, couriers all have different API requirements. Their gateway handles protocol translation, auth, and dynamic service discovery through Consul.
💳 Stripe Custom + Envoy
Stripe's gateway is famous for API versioning — they maintain every API version ever released. The gateway detects the version requested (header or URL), transforms the request to the current internal API, and transforms the response back to the requested version format. Zero breaking changes, ever.
📦 Amazon AWS API Gateway
Amazon uses their own API GW product internally and for millions of customers. Handles 100s of billions of API calls per month. Features usage plans, API keys, Lambda authorizers, and WAF integration. Their gateway charges per request + per GB transferred, which drove the architectural choice to minimize response payload sizes.
🔍 LinkedIn Pegasus
LinkedIn built Pegasus, a REST-to-Avro/Protobuf gateway. External REST requests get translated to their internal Pegasus protocol. Handles schema evolution (they can change internal data models without breaking external APIs) for 900M+ member profiles.
🏦 PayPal Apigee / Custom
PayPal's gateway enforces PCI-DSS compliance at the border — ensuring card data is tokenized before reaching any service. The gateway also handles international compliance: EU requests go through GDPR-compliant data paths, US requests go through a different pipeline. Regulatory routing via gateway.
Tradeoffs

Advantages, Disadvantages & Tradeoffs

✅ Advantages
  • Single entry point simplifies client code enormously
  • Cross-cutting concerns (auth, logging, rate limiting) in one place
  • Services stay lean — no auth boilerplate in each service
  • Easy API versioning and backward compatibility
  • Centralized observability — one place to see all traffic
  • Protocol translation enables technology freedom internally
  • Security: backend services never exposed to internet
❌ Disadvantages
  • Single point of failure (if not deployed in HA)
  • Additional network hop adds latency (~2-10ms)
  • Can become a development bottleneck ("gateway team" becomes a blocker)
  • Increases complexity — another system to operate and monitor
  • Risk of "fat gateway" anti-pattern (business logic creeping in)
  • Hard to test locally with all plugins enabled
  • Vendor lock-in with managed services (AWS API GW)
🚨
Anti-Pattern Warning: The Fat Gateway
Never put business logic in the API Gateway. The gateway should only handle infrastructure concerns (auth, rate limiting, routing). If you find yourself writing code like "if user is premium then fetch recommendations" in the gateway, that's a sign you need a BFF service layer, not gateway logic. Gateways are infrastructure, not application code.
Engineering Wisdom

Common Mistakes & Best Practices

Calling your Auth Service for every single request adds 10-50ms latency on every API call and creates a dependency: if Auth Service is slow or down, your entire API goes down.

Fix: Use stateless JWT tokens (self-contained, verify with public key locally) + cache validated tokens in Redis for 60 seconds. No network call needed per request.

A single generic gateway for mobile, web, and partner APIs means you're always making tradeoffs. Mobile needs compact payloads, web needs full data, partners need strict rate limits. One gateway tries to serve all three and serves none well.

Fix: Implement BFF — separate gateway or gateway configuration per client type. Netflix's device-specific gateways are a good model.

Generate a unique X-Correlation-ID (UUID) at the gateway for every incoming request. Pass it to all downstream services in their request headers. All services include it in their logs.

Now you can search your Elasticsearch/Splunk logs for a single UUID and see the complete journey of a user's request across all 10 microservices. Invaluable for debugging production issues.

If a downstream service (e.g. Recommendation Service) is returning errors >50% of the time, the circuit breaker "opens" — the gateway immediately returns a cached fallback response instead of hammering the failing service. This prevents cascade failures.

Configure per-route: open after 50% error rate in 10 seconds, try one request every 30 seconds to check recovery. Libraries: Hystrix, Resilience4j, Envoy circuit breaker.

Version your APIs from day one. Three approaches: URL versioning (/v1/users, /v2/users), Header versioning (Accept: application/vnd.api.v2+json), or Query param (?version=2).

URL versioning is most visible and cacheable. The gateway routes /v1/* to v1 services and /v2/* to v2 services. Maintain both in parallel during migration. Deprecate with a sunset header: Sunset: Sat, 31 Dec 2025 00:00:00 GMT.

Security

Security at the API Gateway

🛡️
Input Validation
Validate request size limits (reject bodies > 1MB for most APIs), content-type headers, URL parameter lengths. Block oversized requests before they reach services.
🚫
OWASP Top 10
Use WAF integration to block SQL injection, XSS, path traversal, and SSRF attacks at the gateway. Services should never trust raw user input — but the gateway is the first defense.
🔑
API Key Management
Never hardcode API keys. Use the gateway's key management (Kong credentials, AWS API Keys). Keys should be rotatable, scoped (read-only vs write), and revocable instantly without code deploys.
📋
Security Headers
Add X-Content-Type-Options: nosniff, X-Frame-Options: DENY, HSTS, Content-Security-Policy at the gateway. One place for all security headers across all services.
Interview Preparation

Common Interview Questions

What is the difference between an API Gateway and a Load Balancer?
A Load Balancer distributes traffic across a pool of identical servers — it knows about servers, not APIs. An API Gateway is application-aware — it understands HTTP methods, routes, auth tokens, and rate limits. An LB operates at L4/L7 for distribution; an API Gateway operates at L7 for intelligence. In production, you often have both: LB distributes traffic to API Gateway instances, and the gateway routes to microservices. Load balancer = traffic distribution. API Gateway = API management.
How would you handle API versioning in an API Gateway?
Three strategies: (1) URL-based: /v1/users and /v2/users — gateway routes to different service versions. Easy to test, cache, and understand. (2) Header-based: Accept: application/vnd.api.v2+json — cleaner URLs but harder to test in browser. (3) Query param: ?version=2 — easy to test but pollutes URLs. Best practice: URL versioning + maintain v1 and v2 simultaneously during transition. Use Sunset header to communicate deprecation. Gateway routes based on path prefix with weighted routing for gradual migration.
How does an API Gateway handle authentication vs authorization?
Authentication (AuthN) = Who are you? — Gateway validates JWT signature + expiry, or calls OAuth2 introspection, or checks API key. This is stateless (JWT) or involves a Redis cache. Authorization (AuthZ) = Can you do this? — Once identity is confirmed, the gateway checks RBAC rules: does this role have permission for this method+path? E.g., POST /api/admin/* requires admin role. Downstream services get X-User-ID and X-User-Role headers injected by the gateway and trust them — they never re-validate the JWT.
Design an API Gateway for a system with 1 million requests per minute.
(1) Deploy multiple gateway instances behind an L4 Load Balancer. (2) Use Redis Cluster for distributed rate limiting counters shared across all gateway instances. (3) Use stateless JWT auth to avoid network calls per request — validate locally with public key. (4) Use keepalive connections to upstream services (connection pooling) to avoid TCP handshake overhead. (5) Enable horizontal autoscaling (Kubernetes HPA) triggered by CPU/RPS metrics. (6) Use gzip compression for response bodies. (7) Cache JWT public keys (JWKS) locally with 5-minute TTL. At 1M RPM (~16k RPS), a single Nginx gateway instance handles this — but run at least 3 for HA.
What is the BFF (Backend for Frontend) pattern and when would you use it?
BFF creates a dedicated API Gateway per client type (mobile, web, partner). Each BFF is optimized for its client: mobile BFF returns compact payloads (bandwidth constrained), web BFF returns full data for SSR, partner BFF enforces strict API contracts and rate limits. Use BFF when: (1) clients have meaningfully different data requirements, (2) you want to avoid over-fetching/under-fetching, (3) clients have different auth requirements (mobile uses device certs, web uses session cookies, partners use API keys). Netflix, Uber, and Amazon all use BFF. Trade-off: more gateways to maintain, but teams can own their own BFF independently.
Production Ops

Production Challenges & Debugging

When every request goes through one gateway, it becomes a scalability bottleneck. Symptoms: gateway CPU pegged at 90%+, p99 latency spikes to 500ms+, connection timeouts.

Solutions: (1) Deploy multiple gateway instances + LB in front, (2) Move heavy plugins (complex transformations) to async processing, (3) Cache JWT validation results in Redis, (4) Use keepalive connection pools to services, (5) Profile which plugins consume the most CPU per request — usually it's JWT crypto.

Gateway shows 200 OK received from service, but the actual response body is an error. This happens when services return HTTP 200 with an error in the JSON body (bad practice but common legacy behavior).

Debug steps: (1) Check gateway access logs for upstream response code + body, (2) Use the Correlation-ID to find matching service logs, (3) Enable gateway debug mode to log full request/response headers and bodies (temporarily), (4) Use curl directly to the service bypassing the gateway to isolate the issue.

Common production issue: users complain about 429 errors but they claim they're not making many requests. Root cause is usually: (1) trust proxy not set — all users appear to come from one LB IP, sharing one rate limit pool. (2) Multiple services retry on failure, multiplying the actual request count. (3) Frontend JavaScript loops/polling not accounting for backoff.

Fix: Rate limit on User ID (from JWT), not IP address. Return X-RateLimit-Remaining and Retry-After headers so clients can self-throttle.
