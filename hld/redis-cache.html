Redis Cache

The in-memory data structure store that makes your application blazing fast. Learn how Redis works internally, every data type, caching patterns, persistence strategies, clustering, and why it's the backbone of every high-scale backend.

Introduction

What is Redis?

Redis (Remote Dictionary Server) is an open-source, in-memory data structure store that can be used as a database, cache, message broker, and queue. It stores data in RAM instead of disk, making reads and writes 100–1000x faster than a traditional relational database.

Redis is single-threaded for command execution (no locking contention), uses non-blocking I/O for network handling, and achieves 100,000–1,000,000+ operations per second on a single node.

📚 Simple Analogy — The Desk vs Filing Cabinet

Your database is like a filing cabinet in a storage room. Fast to search, holds everything, but you have to walk there every time. Redis is your desk — the things you need most often are right in front of you, instantly accessible. You don't re-read a 100-page report every time someone asks about the summary. You keep the summary on your desk. When the summary changes, you update it. That's caching.

Every time a user loads their profile, your app queries the database: "SELECT * FROM users WHERE id=123". This takes 20–50ms. With 10,000 users loading profiles per second, that's 10,000 database queries per second — the DB gets crushed.

With Redis: first request hits the DB (50ms), stores the result in Redis with a key user:123. Every subsequent request for the same user returns from Redis in under 1ms. Database load drops 99%.

Redis uses an event loop (similar to Node.js) — a single thread processes commands sequentially using epoll/kqueue for I/O multiplexing. No locks, no thread context switches. This is why it achieves higher throughput than multi-threaded alternatives despite being single-threaded.

Memory allocation uses jemalloc for efficient fragmentation management. Data structures are implemented as compact encoding switching — a Hash with few fields uses ziplist (contiguous memory) and upgrades to hashtable when it grows. Redis 7+ uses listpack replacing ziplist for even better cache locality.

Problem Statement

Why Redis Exists — The Database Bottleneck

Without caching, every user interaction hits your database. Here's what happens at scale:

🐢
Slow DB Queries
Complex JOINs, aggregations, and full-table scans take 50–500ms. With thousands of users, this becomes the primary bottleneck. A single trending post can trigger millions of DB hits.
💸
Expensive Computation
Leaderboards, recommendation scores, follower counts — computed data that takes seconds to calculate but rarely changes. Recalculating on every request wastes CPU and time.
🔥
Hot Data Problem
80% of traffic hits 20% of your data (Pareto principle). A viral tweet, a trending product — the same rows are read millions of times per second. DB can't handle this without caching.
🔗
Session & State Storage
HTTP is stateless. Sessions stored in DB create DB traffic on every authenticated request. Redis gives sub-millisecond session lookup with automatic expiry — perfect for auth tokens.

Why Single-Threaded is Actually Fast

Redis's single-threaded command execution seems counterintuitive, but here's why it works:

1
No locking overhead
Multi-threaded systems spend enormous time on mutex locks, condition variables, and thread synchronization. Redis avoids all of this — no lock contention, no deadlocks, no context switches between threads.
2
Memory operations are nanoseconds
RAM access is so fast (50–100ns) that even sequential processing handles millions of operations per second. The bottleneck is network I/O, not CPU — and that's handled asynchronously.
3
Multiplexed async I/O
Redis uses epoll (Linux) or kqueue (macOS) to handle thousands of concurrent client connections without blocking. Network I/O is async — the event loop processes all connections in one thread.
4
Redis 6+ I/O threads
Redis 6 introduced threaded I/O for reading/writing network data (while keeping command execution single-threaded). This removes the last bottleneck — now Redis can saturate a 10Gbps network interface.
Core Concepts

Redis Data Types — The Real Power

Redis isn't just a key-value store. Its rich data structures are what make it uniquely powerful for different use cases.

🔤
STRING
SET key value EX 3600
Cache JSON responses, session tokens, counters (INCR), feature flags, rate limit counters. Max 512MB per value.
#️⃣
HASH
HSET user:123 name Alice age 28
User profiles, product objects, session data with multiple fields. More memory-efficient than storing serialized JSON strings.
📋
LIST
LPUSH queue task1 / BRPOP queue 0
Task queues, activity feeds, recent items list, message queues. Supports blocking pops for consumer pattern.
🎯
SET
SADD online:users uid1 uid2
Unique visitors, online users, tags, friends lists. Set operations: SUNION, SINTER, SDIFF for social graph queries.
🏆
SORTED SET (ZSet)
ZADD leaderboard 9500 "alice"
Leaderboards, priority queues, range queries by score. Twitter uses ZSets for timelines. Each member has a float score for ordering.
🌊
STREAM
XADD events * action login uid 123
Event sourcing, append-only logs, IoT sensor data, audit trails. Consumer groups for distributed processing. Added in Redis 5.0.
📊
BITMAP
SETBIT active:2024-01-15 userId 1
Track daily active users with 1 bit per user. 100M users = 12.5MB. Fast bitwise operations for analytics: BITCOUNT, BITOP.
🔬
HYPERLOGLOG
PFADD visitors uid1 uid2 uid3
Approximate unique count with 0.81% error, using only 12KB regardless of cardinality. Unique page views, unique IPs — perfect for analytics.
📍
GEO
GEOADD drivers lon lat "driverID"
Uber/Lyft driver locations, find nearby restaurants, geofencing. GEODIST, GEOSEARCH within radius. Internally uses a sorted set with geohash scores.
💡
Interview Tip: When asked about Redis in system design, always mention which data type you'd use and why. "I'd use a Redis Sorted Set for the leaderboard because ZADD is O(log N) and ZRANGE lets me paginate the top-N efficiently" — this shows depth of knowledge.
Caching Patterns

Caching Patterns — Which One to Use?

// Cache-Aside (Lazy Loading) — Most Common Pattern

Flow: App checks Redis first. Cache HIT → return data. Cache MISS → query DB, store result in Redis with TTL, return data.

Pros: Only caches data that's actually requested. Cache failure doesn't break the app (falls back to DB). Resilient.

Cons: First request always slow (cache cold start). Risk of stale data between DB update and TTL expiry.

Used by: Most web applications, REST APIs. Default pattern for user profiles, product pages.

Flow: Every write goes to Redis AND the database synchronously. Reads always hit Redis first.

Pros: Cache is always up-to-date. No stale data problem. Cache never misses (once populated).

Cons: Every write has double latency (Redis + DB). Cache fills with data that may never be read (write-heavy, read-light data wasted in cache).

Used by: User settings, critical data where staleness is unacceptable.

Flow: Write to Redis immediately (fast), queue a background job to write to DB asynchronously later.

Pros: Extremely fast writes (just Redis). Can batch multiple writes into one DB operation.

Cons: Risk of data loss if Redis crashes before DB write completes. Complexity in handling eventual consistency.

Used by: View counters, like counts (exact count not critical), analytics event batching.

Flow: App always queries Redis. Redis itself (via a cache library) fetches from DB on miss and stores the result. App never directly calls the DB.

Pros: App code is simpler — just talk to Redis. Cache miss logic handled by the cache layer.

Cons: Cache must support read-through plugins. First request still slow on cold start. Less control over TTL per key.

Used by: NCache, Amazon ElastiCache with read-through providers.

Cache Eviction Policies

When Redis runs out of memory, it uses eviction policies to free space:

PolicyDescriptionBest For
allkeys-lruEvict least recently used keys first (across all keys)General purpose caching — most recommended
volatile-lruLRU only on keys with TTL setMix of persistent + cached data
allkeys-lfuEvict least frequently used (Redis 4+)Hot/cold data with skewed access patterns
volatile-ttlEvict keys with shortest TTL remainingWhen TTL already encodes priority
allkeys-randomEvict random keysAlmost never — unpredictable
noevictionReturn OOM error on new writesDatabases where data loss is unacceptable
Configuration & Code

Production Code Examples

Node.js — Cache-Aside Implementation

JAVASCRIPT cache.service.js 
const Redis = require('ioredis');

// Production Redis client with retry logic
const redis = new Redis({
  host: process.env.REDIS_HOST || 'localhost',
  port: 6379,
  password: process.env.REDIS_PASSWORD,
  // Retry with exponential backoff
  retryStrategy: (times) => Math.min(times * 50, 2000),
  maxRetriesPerRequest: 3,
  enableReadyCheck: true,
  lazyConnect: true,
});

redis.on('error', (err) => console.error('Redis error:', err));

// ─── Cache-Aside Helper ──────────────────────────────
async function cacheAside(key, ttlSeconds, fetchFn) {
  try {
    // 1. Try Redis first
    const cached = await redis.get(key);
    if (cached) {
      console.log(`Cache HIT: ${key}`);
      return JSON.parse(cached);
    }
    console.log(`Cache MISS: ${key}`);
  } catch (err) {
    // Redis failure → degrade gracefully, still serve from DB
    console.error('Redis unavailable, falling back to DB', err);
  }

  // 2. Cache miss — fetch from source (DB, API, etc.)
  const data = await fetchFn();

  // 3. Store in Redis with TTL (fire and forget — don't await)
  redis.setex(key, ttlSeconds, JSON.stringify(data))
    .catch(err => console.error('Failed to cache:', err));

  return data;
}

// ─── Usage ───────────────────────────────────────────
// Get user — cached for 5 minutes
async function getUser(userId) {
  return cacheAside(
    `user:${userId}`,
    300,  // 5 min TTL
    () => db.query('SELECT * FROM users WHERE id = $1', [userId])
  );
}

// Update user — invalidate cache on write
async function updateUser(userId, data) {
  await db.query('UPDATE users SET ... WHERE id = $1', [userId]);
  // Delete cache key so next read fetches fresh data
  await redis.del(`user:${userId}`);
}

// ─── Cache user's full profile as Hash ──────────────
async function cacheUserHash(userId, profile) {
  const key = `profile:${userId}`;
  // HSET stores as individual fields — can update single field
  await redis.hset(key, {
    name:    profile.name,
    email:   profile.email,
    plan:    profile.plan,
    credits: profile.credits,
  });
  await redis.expire(key, 600);  // 10 min TTL
}

// Increment view count — atomic, no race conditions
async function incrementViews(postId) {
  return redis.incr(`views:${postId}`);
  // INCR is atomic in Redis — no race condition even with
  // thousands of concurrent requests
}

Redis — Leaderboard with Sorted Sets

JAVASCRIPT leaderboard.js 
// ─── Game Leaderboard using Redis Sorted Sets ────────
const LEADERBOARD_KEY = 'leaderboard:global';

// Update player score — O(log N)
async function updateScore(playerId, newScore) {
  await redis.zadd(LEADERBOARD_KEY, newScore, playerId);
}

// Get top 10 players with scores — O(log N + M)
async function getTopPlayers(limit = 10) {
  // ZREVRANGE: highest scores first, WITHSCORES returns pairs
  const results = await redis.zrevrange(
    LEADERBOARD_KEY, 0, limit - 1, 'WITHSCORES'
  );
  // Transform [playerId, score, playerId, score...] into objects
  const leaderboard = [];
  for (let i = 0; i < results.length; i += 2) {
    leaderboard.push({
      rank: i / 2 + 1,
      playerId: results[i],
      score: parseFloat(results[i + 1])
    });
  }
  return leaderboard;
}

// Get player's rank — O(log N)
async function getPlayerRank(playerId) {
  // ZREVRANK: 0-indexed rank from top (0 = #1)
  const rank = await redis.zrevrank(LEADERBOARD_KEY, playerId);
  return rank !== null ? rank + 1 : null;
}

// Get players around a given player (for context)
async function getNeighbors(playerId, range = 2) {
  const rank = await redis.zrevrank(LEADERBOARD_KEY, playerId);
  const start = Math.max(0, rank - range);
  const end = rank + range;
  return redis.zrevrange(LEADERBOARD_KEY, start, end, 'WITHSCORES');
}

Durability

Persistence — RDB vs AOF

Redis is in-memory, but it offers two persistence mechanisms so data survives restarts:

💾 RDB (Redis Database Backup)
  • Point-in-time snapshots saved to disk periodically
  • Compact binary format — small file size
  • Fast restart (load snapshot on boot)
  • Great for backups and disaster recovery
  • Configurable intervals (every N seconds if M keys changed)
📋 AOF (Append Only File)
  • Logs every write command to disk
  • Can replay on restart to reconstruct state
  • Better durability (can fsync every second or every write)
  • Larger file size than RDB
  • Slower restarts (must replay all commands)
✅
Production Best Practice: Use both RDB + AOF together. AOF for durability (max 1 second of data loss), RDB for fast restarts and backups. Redis 7.2+ supports RDB-AOF hybrid persistence which combines the best of both.
High Availability

Redis Replication, Sentinel & Cluster

One master handles all writes. One or more replicas receive asynchronous copies of all writes. Read traffic can be distributed across replicas.

Config: replicaof 10.0.1.1 6379 in replica's redis.conf. Or runtime: REPLICAOF 10.0.1.1 6379.

Replication lag: Replicas are eventually consistent. For most use cases (cache reads), this is fine. For financial systems, always read from master.

Redis Sentinel is a high-availability solution that: (1) Monitors master and replicas, (2) Notifies when something is wrong, (3) Promotes a replica to master if the master fails.

Requires minimum 3 Sentinel instances (to form quorum for failover decisions). Failover typically completes in 30–60 seconds.

Client must connect to Sentinel first: Sentinel tells the client which IP is the current master. ioredis and Jedis support Sentinel discovery natively.

Redis Cluster shards data across multiple master nodes using 16384 hash slots. Each key is assigned to a slot: slot = CRC16(key) % 16384.

Example with 3 masters: Node A owns slots 0–5460, Node B owns 5461–10922, Node C owns 10923–16383. Each master has replicas for HA.

Minimum 6 nodes (3 masters + 3 replicas) for a production cluster. Supports automatic re-sharding when adding/removing nodes. Multi-key commands require all keys to be in the same slot (use hash tags: {user}.profile and {user}.settings both hash to same slot).

Industry Usage

How Top Companies Use Redis

🐦 Twitter/X Timeline Cache
Twitter caches user timelines in Redis using Sorted Sets — tweet IDs scored by timestamp. When you open Twitter, your timeline is served entirely from Redis. They use a "timeline fanout" system where each tweet is pushed to all followers' Redis timeline caches. Handles billions of timeline reads per day at sub-5ms latency.
🛒 Amazon ElastiCache Redis
Amazon uses Redis (ElastiCache) for shopping cart storage, session management, and product recommendation caching. During Prime Day, their Redis cluster handles millions of cart reads per second. They also use Redis Cluster for horizontal scaling across availability zones.
🎮 Discord Presence & Rate Limits
Discord uses Redis to track which users are online (presence). With 500M registered users, they use Redis Sets per server to track online members. Also uses Redis for rate limiting API calls. They switched from Redis to Cassandra for message storage but kept Redis for real-time features.
🚗 Uber Geospatial + Sessions
Uber uses Redis GEO commands to track driver locations in real time. When you request a ride, Redis GEO finds all drivers within 5km of your location. Also uses Redis for session storage and surge pricing calculations. Their Redis deployment handles 40M+ location updates per day.
📸 Instagram Feed + Counters
Instagram uses Redis for user feed caching, like/follower counts (using Redis atomic INCR), and session tokens. They keep recent activity feeds in Redis Lists. Instagram's "Social Graph" service uses Redis Sets to compute mutual friends and follow suggestions.
🏦 GitHub Job Queues + Cache
GitHub runs Resque (Ruby job queue) on Redis, processing millions of background jobs per day (CI notifications, webhook deliveries, code analysis). Also uses Redis for caching repository metadata, diff computations, and rate limiting API calls to 5000 req/hour per token.
Production Pitfalls

Common Problems & How to Solve Them

Problem: A popular cached item expires. Simultaneously, 10,000 requests come in, all get cache miss, all hit the DB at once — DB is overloaded.

Solutions:

1. Mutex Lock: First thread to get a cache miss acquires a Redis lock (SET lock:key 1 NX EX 10), fetches from DB, populates cache, releases lock. Other threads wait and retry.

2. Probabilistic Early Expiration (XFetch): Before TTL expires, randomly decide to refresh based on how close to expiry. Spreads the refresh load over time.

3. Stale-while-revalidate: Serve stale cache while asynchronously refreshing in the background.

Problem: Many cache keys expire at the same time (e.g., all set with TTL=3600 at startup). Massive DB load spike every hour.

Solutions:

1. TTL Jitter: Add randomness to TTL: TTL = base + Math.random() * jitter. If base=3600, jitter=300, TTLs range from 3600–3900s. Expiries spread over 5 minutes instead of all at once.

2. Different TTLs per data category: User profiles=5min, product data=30min, config=1hour. Natural spread.

Problem: Requests for data that doesn't exist in DB (e.g., user ID -1). Every request is a cache miss and a DB query (finding nothing). Attackers can exploit this to overload your DB with random IDs.

Solutions:

1. Cache null values: When DB returns no result, cache NULL with a short TTL (30–60s). Next requests hit Redis and get null — no DB query.

2. Bloom Filter: Before checking Redis, check a Bloom Filter (uses Redis module RedisBloom) — probabilistic data structure that tells if a key "definitely does not exist" or "probably exists." Near-zero false negatives. Block invalid requests before they reach Redis or DB.

Problem: A single Redis key holds too much data (e.g., a Set with 10 million members, or a String with 500MB JSON). When Redis tries to serialize, delete, or expire this key, it blocks the single-threaded event loop for seconds — all other commands are frozen.

Solutions: (1) Split big keys into smaller chunks: followers:userId:shard1, followers:userId:shard2. (2) Use UNLINK instead of DEL — UNLINK deletes asynchronously without blocking. (3) Scan with HSCAN/SSCAN in batches instead of HGETALL/SMEMBERS which return everything at once.

Tradeoffs

Advantages, Disadvantages & Tradeoffs

✅ Advantages
  • Sub-millisecond latency — 100–10,000x faster than disk DBs
  • Rich data structures for complex use cases
  • Atomic operations eliminate race conditions
  • TTL/expiry built-in — automatic cache invalidation
  • Pub/Sub messaging built-in
  • Lua scripting for complex atomic operations
  • Horizontal scaling via Redis Cluster
❌ Disadvantages
  • Data limited by RAM — expensive at large scale
  • Not a primary database (no complex queries, no joins)
  • Single-threaded = one slow command blocks everything
  • Cluster multi-key operations have limitations
  • Eventual consistency with replicas
  • Complexity of cache invalidation (hardest problem in CS)
  • Memory fragmentation can waste 20–30% of RAM
Interview Preparation

Common Interview Questions

Why is Redis single-threaded but still so fast?
Redis processes commands in a single thread but uses non-blocking I/O (epoll/kqueue) to handle thousands of concurrent connections. The bottleneck isn't CPU (memory ops are nanoseconds) — it's network I/O, which is handled asynchronously. By avoiding thread context switches, mutex locks, and cache coherency overhead, Redis achieves 100k-1M ops/sec. Redis 6+ adds I/O threads for network reading/writing while keeping command execution single-threaded.
How would you implement a distributed rate limiter using Redis?
Use Redis INCR + EXPIRE for a fixed window counter: INCR the key for the current window (e.g., "ratelimit:{userId}:{minute}"), set EXPIRE on first increment. If count exceeds limit, reject. For a sliding window: store request timestamps in a Sorted Set, remove entries older than the window with ZREMRANGEBYSCORE, count remaining with ZCARD. For token bucket: store tokens as a float, use a Lua script to atomically check/decrement tokens and calculate refill based on elapsed time since last request. Redis Lua scripts execute atomically — no race conditions.
Explain the difference between Redis persistence options (RDB vs AOF).
RDB (Redis Database) takes periodic point-in-time snapshots of the entire dataset to a binary file. Fast to restore, small file size, but can lose up to minutes of data between snapshots. AOF (Append Only File) logs every write command — replays them on restart. Can be configured to fsync every second (max 1 second data loss) or every write (no data loss but slower). Best practice: use both. AOF for durability, RDB for fast backups. Redis 7.2+ has RDB-AOF hybrid mode which starts recovery from the latest RDB then replays only the AOF delta.
How does Redis Cluster work and what are its limitations?
Redis Cluster shards data across multiple master nodes using 16384 hash slots. Keys are mapped to slots via CRC16(key) % 16384. Each master owns a range of slots. Minimum 3 masters + 3 replicas. Supports automatic resharding and failover. Limitations: (1) Multi-key commands (MSET, MGET, SUNION) only work if all keys are in the same slot — use hash tags {tag} to force co-location. (2) Lua scripts can only reference keys in one slot. (3) Pub/Sub is per-node, not cluster-wide. (4) SCAN iterates per-node. (5) Cross-slot transactions not supported.
How would you design Twitter's timeline using Redis?
Use a Redis Sorted Set per user: key = "timeline:{userId}", member = tweetId, score = timestamp. On tweet post: ZADD all followers' timeline sorted sets with the new tweetId (fanout on write). On timeline read: ZREVRANGE "timeline:{userId}" 0 99 (latest 100 tweets), then multi-get tweet data from Redis or DB. Trim timelines to 800 entries (ZREMRANGEBYRANK). For celebrities (10M+ followers), use fanout on read — too slow to push to 10M timelines synchronously. Hybrid: push to regular users, pull for celebrities when timeline is requested.
Summary
🎯 What You Now Know
  • ✦ Redis is an in-memory data store — 100–1000x faster than disk DBs, sub-millisecond latency
  • ✦ Single-threaded + async I/O = no locking overhead → 100k-1M ops/sec on one node
  • ✦ Rich data types: String, Hash, List, Set, Sorted Set, Stream, Bitmap, HyperLogLog, Geo
  • ✦ Caching patterns: Cache-Aside (most common), Write-Through, Write-Behind, Read-Through
  • ✦ Eviction: allkeys-lru is the go-to policy for caches
  • ✦ Persistence: use RDB + AOF together for durability + fast recovery
  • ✦ HA: Master-Replica → Sentinel (auto-failover) → Cluster (horizontal scale)
  • ✦ Pitfalls: Cache stampede (mutex/jitter), avalanche (TTL jitter), penetration (null caching/bloom filter), big key (UNLINK/SCAN)
