Content Delivery Network

The global cache that makes your app fast everywhere on Earth. Learn how CDNs work internally — PoPs, edge caching, cache invalidation, origin pull vs push, dynamic acceleration, and how Cloudflare, Akamai, and AWS CloudFront serve billions of requests per second at the speed of light.

Introduction

What is a CDN?

A Content Delivery Network (CDN) is a globally distributed network of proxy servers (called Points of Presence or PoPs) that cache your content close to your users. Instead of every user hitting your origin server in one datacenter, they fetch content from the nearest CDN edge node — potentially 100× closer geographically.

The fundamental insight: latency is dominated by physical distance. A user in Tokyo hitting a server in Virginia travels ~14,000 km — that's ~140ms of round-trip latency just from the speed of light, before any processing. A CDN edge in Tokyo reduces that to ~5ms.

🏪 Analogy — Supermarket vs Local Corner Shop

Your origin server is like a large central warehouse — it has everything, but it's far away. Every time someone wants milk, they drive 2 hours to the warehouse. A CDN PoP is like a local corner shop — it stocks the most popular items (cached content). Most people buy milk from the corner shop (edge node) in 2 minutes. Only when the shop runs out of a specific item does someone go to the warehouse (origin). The warehouse still controls what's stocked — it just doesn't serve every individual customer.

When you visit a website, your browser downloads files — HTML, CSS, JavaScript, images, videos. Without a CDN, all of these come from one server, say, in New York. If you're in Mumbai, every file travels halfway around the world. That's slow.

A CDN puts copies of those files in servers all around the world. Now when you visit from Mumbai, the files come from a server in Mumbai (or Singapore). Fast. Netflix uses this so your movie streams smoothly even though the files originally live in Amazon's US data centres.

CDNs operate at multiple protocol layers. At the DNS level, Anycast routing announces the same IP address from multiple locations simultaneously — BGP automatically routes users to the topologically closest PoP. At the transport layer, PoPs maintain persistent TCP connections to origin and terminate TLS at the edge, amortising handshake costs. At the application layer, edge nodes cache HTTP responses, apply transformation logic (image resizing, minification), and perform DDoS mitigation before traffic reaches origin.

Modern CDNs like Cloudflare also offer edge compute (Workers/Lambda@Edge) — running JavaScript at 300+ PoPs with sub-millisecond cold starts, enabling personalisation, A/B testing, and auth at the edge without hitting origin.

Problem Statement

Why CDNs Are Essential at Scale

🌍
Geographic Latency
Speed of light limits: US↔Europe = ~80ms, US↔Asia = ~140ms. Every additional round trip (TCP handshake, TLS, HTTP) multiplies this. A page with 20 assets needs 20+ RTTs from a single origin — easily 3+ seconds for global users.
💥
Origin Server Overload
A viral post, product launch, or news event creates a sudden traffic spike. Without a CDN, every request hits your origin. With CDN, 95%+ of requests are served from edge cache — your origin only handles the remaining cache misses.
💰
Bandwidth Costs
Egress bandwidth from cloud providers (AWS, GCP) costs $0.05–$0.08/GB. CDN bandwidth costs $0.005–$0.01/GB — 5–10× cheaper. For a platform serving 1 PB/month, CDN saves $40,000–$70,000 monthly versus direct origin serving.
🛡️
DDoS Protection
CDN PoPs absorb attack traffic across their massive network capacity. Cloudflare has 321 Tbps of global DDoS mitigation capacity. Attack traffic is filtered at the edge before it ever reaches your origin — your origin IP is hidden behind the CDN.
Internal Architecture

How a CDN Works — Step by Step

// CDN Request Flow — Cache HIT vs Cache MISS
1
DNS Resolution via Anycast
User's browser resolves cdn.yourapp.com. The CDN's DNS responds with the same IP address (Anycast) that is announced from 300+ locations globally. BGP routing automatically directs the user to the topologically nearest PoP — usually the one with fewest network hops, not necessarily geographic distance.
2
TLS Handshake at Edge
The TLS handshake (normally 2 RTTs = 2 × latency) happens at the nearby edge PoP instead of the distant origin. For a Mumbai user, TLS completes in 10ms (local PoP) instead of 280ms (US origin). The PoP maintains a persistent, pre-warmed TLS connection to the origin for cache miss fetches.
3
Cache Lookup at Edge
The edge node checks its local cache (usually SSD + RAM) for the requested URL. Cache key is typically the full URL (scheme + host + path + relevant query params). A cache HIT returns the cached response immediately — no origin contact needed. This is typically 99%+ of requests for static assets.
4
Cache MISS → Origin Pull
On a cache miss, the edge node fetches from the origin over an optimised private network backbone (CDN providers have their own intercontinental fiber). The response is cached at the edge with TTL from the Cache-Control header, then returned to the user. Subsequent requests for the same asset get the cached version.
5
Cache Headers Control Behaviour
Your origin controls CDN caching via HTTP headers. Cache-Control: public, max-age=31536000 caches for 1 year. Cache-Control: no-cache checks origin on every request (but can still serve stale if origin returns 304 Not Modified). Surrogate-Control or CDN-Cache-Control can set CDN TTLs independently from browser TTLs.

Cache Invalidation — The Hardest Problem

Phil Karlton famously said: "There are only two hard things in Computer Science: cache invalidation and naming things." CDN cache invalidation is exactly this problem at global scale.

Set max-age appropriately and let content expire naturally. When TTL expires, the edge re-fetches from origin. Simple but means potentially stale content for up to TTL duration after an update.

Best practice: Use short TTLs for content that changes (5–60 minutes) and long TTLs for truly immutable content (content-hashed files). Don't use the same TTL for everything.

stale-while-revalidate: Serve stale content immediately while fetching a fresh version in the background. User gets instant response (no wait); they'll get fresh content on the next request. Excellent for pages that update every few minutes.

All major CDNs provide an API to instantly purge cached content across all PoPs globally. This typically propagates in 1–5 seconds worldwide.

Purge by URL: DELETE https://api.cloudflare.com/zones/{zone}/purge_cache with specific URLs or patterns. Use this in your deployment pipeline — after every deploy, purge the HTML files and any updated JS/CSS chunks.

Purge by cache tag (Surrogate-Key): Tag cached responses with logical identifiers. Surrogate-Key: product:123 category:electronics. Purge all content tagged product:123 when that product updates. Much more efficient than URL-by-URL purges for content-rich sites.

The cleanest solution: include a hash of the file content in the URL. app.a3f4c8b9.js, styles.7d2e1f45.css. The file content never changes for a given URL, so you can safely set max-age=31536000, immutable — cache forever.

When you update the code, webpack/Vite generates a new hash → new URL → CDN fetches and caches the new version. Old version stays cached (harmless — nobody requests it anymore). Zero cache invalidation needed. This is what every modern build tool (Next.js, Vite, webpack) does for production builds.

Only cache for HTML and API endpoints needs active management — they don't have content-hashed URLs.

Delivery Models

Origin Pull vs Push — How Content Gets to the Edge

📥 Origin Pull (Lazy Caching)
  • Edge fetches content from origin on first request (cache miss)
  • Zero setup — CDN automatically caches what users request
  • Only popular content uses edge storage
  • Origin is the source of truth
  • Used for: websites, APIs, most web assets
  • Cache warms automatically as users request content
📤 Push (Pre-loading Cache)
  • You proactively push content to CDN edge nodes
  • No origin hit on first request — always a cache hit
  • Higher storage usage — pushing everything everywhere
  • You manage what's on the CDN explicitly
  • Used for: software downloads, large media files, game patches
  • Requires CDN API integration for upload workflows
✅
Real World: Most websites use Origin Pull — it's simpler and CDN providers like Cloudflare, Fastly, and CloudFront all default to this. Push is used by Steam (game patches), Microsoft (Windows updates), and video platforms (pre-stage movie releases before premiere night).
Advanced Features

Modern CDN Capabilities Beyond Caching

Modern CDNs let you run JavaScript/WASM directly at the edge PoP, without hitting origin. Cloudflare Workers, AWS Lambda@Edge, Fastly Compute@Edge.

Use cases: A/B testing (split traffic at edge based on cookie), auth token validation (verify JWT without hitting auth service), personalisation (inject user-specific headers before serving cached HTML), bot detection, request transformation, geo-based redirects.

Why it's powerful: Instead of origin → app server → personalise → respond (200ms), it's edge → run worker (1ms) → serve cached/personalised response. Dramatic latency reduction for personalised content.

CDNs can resize, format-convert, and compress images on-the-fly based on request parameters or Accept headers. Upload one high-resolution image to origin; CDN serves the right size and format to each device.

Example: Mobile user with Accept: image/avif, image/webp gets an AVIF-converted, 400px-wide version. Desktop gets WebP at 1200px. Same origin URL, different CDN-transformed responses. Services: Cloudflare Images, Imgix, Cloudinary, AWS CloudFront + Lambda@Edge.

CDN terminates HTTPS at the edge PoP, managing TLS certificates automatically (Let's Encrypt or their own CA). This eliminates the TLS handshake round-trip to your origin for users.

The CDN-to-origin connection can use HTTP/1.1 (simpler for origin) while the CDN serves users over HTTP/2 or HTTP/3 (QUIC) — automatically upgrading the protocol at the edge. Users get modern protocol performance; origin doesn't need to support it.

0-RTT TLS resumption: Returning users reconnect with 0 round trips for TLS. The CDN stores session tickets and presents them, eliminating the handshake entirely for repeat visitors.

Even for uncacheable dynamic API responses, CDNs provide significant performance improvement through:

TCP connection multiplexing: The CDN maintains persistent multiplexed HTTP/2 connections to your origin. Users share this warm connection pool — no per-user TCP handshake to origin.

Private backbone routing: CDN providers own intercontinental fiber networks. Your user's request travels on Cloudflare's/AWS's private network from PoP to origin, avoiding public internet routing inefficiencies and congestion.

Result: Even non-cached API calls can be 20–50% faster through a CDN vs direct connection to origin due to TCP optimisation and private network routing.

Configuration

Real Production Configurations

Nginx Origin — Cache-Control Headers

NGINX nginx.conf — Serving assets with proper cache headers 
server {
  listen 80;
  server_name origin.yourapp.com;
  root /var/www/html;

  # ─── Hashed static assets — cache forever ─────────────
  # These files have content hash in filename (app.a3f4c8.js)
  location ~* \.(js|css)$ {
    expires 1y;
    add_header Cache-Control "public, max-age=31536000, immutable";
    add_header Vary "Accept-Encoding";
    gzip_static on;
    brotli_static on;
  }

  # ─── Images — cache 30 days ───────────────────────────
  location ~* \.(png|jpg|jpeg|gif|svg|webp|avif|ico)$ {
    expires 30d;
    add_header Cache-Control "public, max-age=2592000";
    add_header Vary "Accept"; # For WebP/AVIF content negotiation
  }

  # ─── HTML — short TTL, stale-while-revalidate ─────────
  location / {
    try_files $uri $uri/ /index.html;
    add_header Cache-Control
      "public, max-age=300, stale-while-revalidate=3600, stale-if-error=86400";
    # Cache for 5 min, serve stale for 1hr while revalidating,
    # serve stale for 24hr if origin is down
  }

  # ─── API responses — no CDN caching ──────────────────
  location /api/ {
    proxy_pass http://backend:3000;
    add_header Cache-Control "private, no-store";
    # CDN will not cache — goes to origin every time
  }

  # ─── Public API responses — cache at CDN ─────────────
  location /api/public/ {
    proxy_pass http://backend:3000;
    # s-maxage = CDN TTL (5 min); max-age = browser TTL (1 min)
    add_header Cache-Control "public, max-age=60, s-maxage=300";
    # Surrogate-Key for tag-based invalidation
    add_header Surrogate-Key "public-api v1";
  }
}

Cloudflare Cache Invalidation via API

JAVASCRIPT deploy/purge-cdn.js — Post-deploy cache purge 
const CF_ZONE_ID  = process.env.CF_ZONE_ID;
const CF_API_TOKEN = process.env.CF_API_TOKEN;
const BASE_URL    = 'https://yourapp.com';

/**
 * Purge specific URLs from Cloudflare cache after deployment.
 * Call this from your CI/CD pipeline after a successful deploy.
 */
async function purgeCDNCache() {
  // 1. Purge HTML pages (these don't have content hashes)
  const urlsToPurge = [
    `${BASE_URL}/`,
    `${BASE_URL}/about`,
    `${BASE_URL}/pricing`,
    // Add all routes that serve HTML
  ];

  const response = await fetch(
    `https://api.cloudflare.com/client/v4/zones/${CF_ZONE_ID}/purge_cache`,
    {
      method: 'POST',
      headers: {
        'Authorization': `Bearer ${CF_API_TOKEN}`,
        'Content-Type':  'application/json',
      },
      body: JSON.stringify({ files: urlsToPurge }),
    }
  );

  const data = await response.json();
  if (!data.success) throw new Error(`CDN purge failed: ${JSON.stringify(data.errors)}`);
  console.log(`✅ Purged ${urlsToPurge.length} URLs from Cloudflare CDN`);
  return data;
}

// 2. Tag-based purge (requires Cloudflare Enterprise or Cache Rules)
async function purgeByTag(tags) {
  return fetch(
    `https://api.cloudflare.com/client/v4/zones/${CF_ZONE_ID}/purge_cache`,
    {
      method: 'POST',
      headers: { 'Authorization': `Bearer ${CF_API_TOKEN}`, 'Content-Type': 'application/json' },
      body: JSON.stringify({ tags }),
    }
  );
  // tags: ['product:123', 'category:electronics']
  // Purges all URLs tagged with any of these in Surrogate-Key header
}

// Run in CI/CD after deploy
purgeCDNCache()
  .then(() => console.log('Cache purge complete'))
  .catch(err => {
    console.error('Cache purge failed (non-fatal):', err);
    // Don't fail the deploy for a cache purge error
    // Old content will expire via TTL anyway
  });

Cloudflare Worker — JWT Auth at Edge

JAVASCRIPT worker.js — Auth + personalisation at Cloudflare Edge 
/**
 * Cloudflare Worker: runs at the CDN edge, not on origin.
 * Validates JWT and injects user context before serving cached content.
 * Zero latency added for authenticated requests — auth happens at edge.
 */
export default {
  async fetch(request, env) {
    const url = new URL(request.url);

    // 1. Allow public assets through without auth check
    if (url.pathname.startsWith('/static/') ||
        url.pathname.startsWith('/api/public/')) {
      return fetch(request);  // pass through to origin/cache
    }

    // 2. Extract and validate JWT at edge (no origin round-trip)
    const token = request.headers.get('Authorization')?.split(' ')[1];
    if (!token) {
      return new Response('Unauthorized', { status: 401 });
    }

    try {
      // Verify JWT using Web Crypto API (available in Workers)
      const payload = await verifyJWT(token, env.JWT_PUBLIC_KEY);

      // 3. Inject user context as headers (origin/cached service reads these)
      const modifiedRequest = new Request(request, {
        headers: {
          ...Object.fromEntries(request.headers),
          'X-User-ID':    payload.sub,
          'X-User-Role':  payload.role,
          'X-User-Country': request.cf?.country || 'unknown',
        }
      });

      // 4. A/B test routing at edge
      const variant = parseInt(payload.sub) % 2 === 0 ? 'A' : 'B';
      if (url.pathname === '/') {
        url.pathname = variant === 'A' ? /v2/' : '/';
      }

      return fetch(new Request(url.toString(), modifiedRequest));
    } catch (err) {
      return new Response('Invalid token', { status: 401 });
    }
  }
};
Industry Usage

How Top Companies Use CDNs

🎬 Netflix Open Connect
Netflix built their own CDN called Open Connect, partnering directly with ISPs to place dedicated Netflix appliances (OCA — Open Connect Appliances) inside ISP networks. Over 17,000 OCAs in 158 countries serve 95%+ of Netflix traffic directly from the ISP's network — zero public internet hops. Each appliance holds popular titles pre-staged. New season drops: pre-push to all appliances before release.
🌐 Cloudflare Own Network
Cloudflare operates the world's largest CDN — 321+ Tbps capacity across 300+ cities. Their key innovation: every PoP runs every product — CDN, DDoS protection, Zero Trust, Workers, DNS. No separate networks for different products. This lets them announce every customer's IP from all 300+ locations simultaneously via Anycast, routing traffic optimally worldwide.
🛒 Amazon CloudFront
AWS CloudFront deeply integrates with S3 (origin), Lambda@Edge, and WAF. Amazon.com's product images, JS/CSS assets, and video content are all served through CloudFront. During Prime Day, CloudFront serves tens of millions of requests per second. Their Origin Shield feature adds a regional caching tier between edge PoPs and origin, reducing origin load by 60–80%.
🐦 Twitter/X Akamai + own
Twitter uses Akamai for media content (images, videos) and has their own CDN for tweet API responses with short TTLs. During major events (elections, sports, disasters), tweet traffic spikes 10–20× — CDN caching of public timelines absorbs the load. The Search API results are cached at CDN edge with 15-second TTL for trending topics.
📦 GitHub Fastly
GitHub uses Fastly as their CDN for serving git objects, release assets, and Pages sites. They leverage Fastly's Surrogate-Key (cache tagging) to instantly invalidate all pages related to a repository when a push happens. A push to main purges cache for that repo's pages within milliseconds — ensuring visitors always see fresh content.
🎮 Steam Akamai + own
Valve's Steam uses CDN for game downloads (petabytes daily), with a unique twist: they use content addressing (chunks identified by SHA-1 hash) so if you already have a file chunk from another game, the CDN knows and doesn't re-download it. Game updates only download changed chunks. Akamai handles global distribution; Valve has dedicated edge nodes in large cities.
Tradeoffs

Advantages, Disadvantages & When NOT to Use

✅ Advantages
  • Dramatically lower latency for global users
  • Reduces origin server load by 80–99%
  • Cheaper bandwidth (CDN egress vs cloud egress)
  • Built-in DDoS protection and WAF capabilities
  • Automatic HTTPS certificate management
  • HTTP/2 and HTTP/3 without origin changes
  • High availability — CDN stays up even if origin is down (stale content)
  • Edge compute for personalisation without origin latency
❌ Disadvantages
  • Cache invalidation is complex — stale content bugs
  • Debugging harder — is issue at edge or origin?
  • Vendor lock-in for advanced features (Workers, Lambda@Edge)
  • Private/personalised content cannot be cached at edge
  • Additional cost (though usually cheaper than origin bandwidth)
  • Cache poisoning attacks if cache keys poorly configured
  • Geographic compliance issues (data residency laws)

Common CDN Mistakes

The most dangerous mistake: an API that returns user-specific data (name, account details, private content) accidentally returns Cache-Control: public. The CDN caches the first user's response and serves it to ALL subsequent users requesting the same URL. User A sees User B's data.

Fix: Always set Cache-Control: private, no-store for authenticated/personalised endpoints. Audit all your API routes. Use CDN-specific cache headers (CDN-Cache-Control: no-store) to control CDN behaviour independently from browser cache.

If you serve both gzip and uncompressed responses based on the Accept-Encoding request header, but don't add Vary: Accept-Encoding to your response, the CDN caches one version and serves it to everyone. A user without gzip support gets a compressed response they can't decode.

Fix: Always add Vary: Accept-Encoding for compressed responses. The CDN stores separate cache entries for each Accept-Encoding value. Similarly, use Vary: Accept for content negotiation (WebP vs JPEG).

Some developers append ?v=random or ?t=timestamp to bypass the CDN cache during debugging. This is fine temporarily but terrible as a production pattern — every request is a cache miss, you lose all CDN benefits and hammer your origin.

Correct approach: For static assets, use content-hashed filenames (webpack/Vite handles this). For HTML/API, use CDN purge APIs after deployment. For debugging, use the CDN dashboard's "bypass cache" feature or set Cache-Control: no-cache temporarily in dev environments.

Interview Preparation

Common Interview Questions

How does a CDN work and what are PoPs?
A CDN (Content Delivery Network) is a geographically distributed network of proxy servers that cache content close to users. PoPs (Points of Presence) are the individual edge server locations — data centres deployed in cities worldwide (Cloudflare has 300+, Akamai has 4000+). When a user requests a resource: (1) DNS resolves via Anycast — the same IP is advertised from all PoPs globally, BGP routes users to the nearest PoP. (2) The edge PoP checks its local cache. Cache HIT: serve immediately from edge (~5ms). Cache MISS: fetch from origin over CDN's private backbone, cache the response, then serve it. (3) Subsequent requests for the same resource hit the cache. The key benefit: instead of Sydney users hitting a US server (150ms RTT), they hit the Sydney PoP (2ms RTT).
How do you handle CDN cache invalidation?
Three strategies depending on content type: (1) Content-hashed URLs for static assets (JS/CSS/images). webpack/Vite generates filenames like app.a3f4c8.js. The content never changes for a given URL, so cache forever (max-age=31536000, immutable). When code changes, a new hash/new URL is generated — automatic invalidation by URL change. (2) TTL-based expiry for HTML and API responses. Set appropriate max-age (5 minutes for HTML). Use stale-while-revalidate to serve stale immediately while refreshing in background. (3) Active purge via CDN API for immediate invalidation. Call Cloudflare/CloudFront purge API from your deployment pipeline. Can purge by specific URL, URL pattern, or cache tag (Surrogate-Key). Tags allow "purge all content about product:123" atomically. Priority: always use (1) for static assets. Use (2) + (3) together for HTML — short TTL as safety net plus active purge on deploy.
When should you NOT cache a response at the CDN?
Never cache at CDN: (1) User-authenticated responses — profile data, account details, private content. Any response whose content differs per user must use Cache-Control: private, no-store. If accidentally cached, User A's data could be served to User B. (2) POST/PUT/DELETE responses — non-idempotent methods should not be cached. (3) Responses with Set-Cookie headers containing session tokens. (4) Financial/sensitive data — payment status, balances, personal health info. (5) Short-lived tokens and nonces. Cache but with very short TTL: search results (30s), product inventory (1min), pricing (5min). Can cache with moderate TTL: product descriptions (1hr), public blog posts (24hr). Always cache long: versioned static assets (1 year), public media files.
Design a CDN strategy for a video streaming platform.
Video streaming CDN strategy: (1) Origin storage: videos stored in S3/GCS as HLS/DASH segments (.ts files, .mpd manifests). (2) CDN layer: CloudFront or Akamai in front of S3. Video segments (.ts): Cache-Control: public, max-age=31536000 — immutable once uploaded. Manifests (.m3u8/.mpd): Cache-Control: public, max-age=3600 for VOD, max-age=3 for live streams. (3) Pre-staging popular content: before a major release, use CDN push APIs to pre-populate all PoPs (like Netflix Open Connect). (4) Multi-bitrate: same video in 360p/720p/1080p/4K. Each stored as separate file paths, cached independently. Player selects quality based on bandwidth. (5) Geo-restrictions: CDN enforces country blocks using signed URLs with geo-restriction policies, or edge Workers check geo header. (6) Token-based access: signed URLs with expiry so only paying subscribers can download segments. CDN validates the signature at edge — no origin call needed.
What is the difference between a CDN and a reverse proxy?
A reverse proxy is a server that sits in front of one or more origin servers and forwards client requests to them — it's typically deployed in your own infrastructure in one location. A CDN is a globally distributed network of reverse proxies (edge nodes) deployed in 100s of locations worldwide. Key differences: (1) Scale and distribution — CDN has hundreds of PoPs globally; a single reverse proxy is in one datacenter. (2) Primary purpose — CDN prioritizes geographic proximity and caching for latency; reverse proxy handles routing, load balancing, SSL termination in one location. (3) Ownership — CDN is a managed service (Cloudflare, Akamai); reverse proxy is your own infrastructure (Nginx, HAProxy). In practice: you often use both together — Nginx/Traefik as reverse proxy within your datacenter handling internal routing, plus Cloudflare CDN in front handling global caching, DDoS, and edge delivery. The CDN handles the "last mile" from user to your infrastructure; the reverse proxy handles routing within your infrastructure.
Summary
🎯 What You Now Know
  • ✦ CDN = geographically distributed cache that serves content from the nearest edge PoP, not from a central origin
  • ✦ Anycast routing: same IP announced from 300+ locations — BGP directs users to nearest PoP automatically
  • ✦ Cache HIT: ~5ms from edge. Cache MISS: origin fetch (~150ms). Direct to origin: ~280ms for global users
  • ✦ Static assets with content hashes: cache forever (immutable). HTML: short TTL + stale-while-revalidate. APIs: s-maxage for CDN TTL
  • ✦ Cache invalidation: content-hashed URLs (best) + TTL + active API purge on deploy
  • ✦ Never cache user-specific content at edge — Cache-Control: private, no-store for authenticated responses
  • ✦ Origin Pull (lazy) vs Push (proactive pre-staging) — Pull for websites, Push for large media/game downloads
  • ✦ Edge compute (Cloudflare Workers, Lambda@Edge): auth, A/B testing, personalisation without origin latency
  • ✦ CDN also handles TLS termination, HTTP/3, DDoS absorption, image optimisation, and dynamic acceleration
