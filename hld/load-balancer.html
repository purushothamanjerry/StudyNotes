Load Balancer

The traffic director of distributed systems. Learn how load balancers distribute requests across servers, prevent overloading, enable zero-downtime deployments, and are the foundation of every scalable backend architecture.

Beginner → Intermediate
~35 min read
Production Ready
Interview Critical
Introduction

What is a Load Balancer?

A Load Balancer is a system that sits in front of your servers and distributes incoming network traffic across multiple backend servers. Think of it as a smart traffic controller that ensures no single server gets overwhelmed while others sit idle.

🏦 Simple Analogy — Bank Tellers

Imagine a bank with 5 tellers. A queue manager at the entrance directs each customer to the next available teller. If teller #2 is busy with a complex transaction, the manager routes the next customer to teller #3. Nobody waits while a teller is free, and no single teller handles everyone. That's exactly what a Load Balancer does for your servers.

When you have a website with many users visiting at the same time, one server can't handle all of them. So you add more servers. But now, who decides which user goes to which server?

That's the job of a Load Balancer. When a user's request comes in, the load balancer picks a server (using rules like "pick the least busy one") and forwards the request there. The user doesn't even know this happened — they just see your website load fast.

A Load Balancer operates at Layer 4 (Transport Layer) or Layer 7 (Application Layer) of the OSI model. Layer 4 LBs distribute traffic based on IP/TCP without inspecting content. Layer 7 LBs can read HTTP headers, cookies, and URLs to make intelligent routing decisions.

Modern LBs maintain a connection pool to backends, handle SSL termination to offload encryption from app servers, perform health checks with configurable intervals and thresholds, and support session persistence via sticky sessions or distributed session stores.

Problem Statement

Why is a Load Balancer Needed?

Without a load balancer, your entire backend architecture has several critical failure points:

💥
Single Point of Failure
If your one server crashes, 100% downtime. Netflix famously had this problem before moving to distributed architecture.
🔥
Server Overload
During traffic spikes (product launches, viral events), a single server gets crushed — CPU maxes out, memory fills up, users get 503 errors.
🛑
Zero-Downtime Deploys
You can't deploy new code without taking the server down. Load balancers enable rolling deployments — update one server at a time.
📊
Poor Resource Utilization
Some servers may be idle while others are overwhelmed. A load balancer distributes work evenly for maximum efficiency.
⚠️
Real Incident: In 2013, GitHub experienced an outage because a single database server became a bottleneck during a traffic spike from a Hacker News post. Without proper load balancing, the entire platform was unavailable for 90 minutes.
Architecture

Architecture & Request Flow

Here is how traffic flows through a typical load balancer setup in production:

// Load Balancer — Request Flow Diagram

Request Flow — Step by Step

1
Client sends HTTP request
A user's browser or mobile app sends a request to api.yourapp.com. DNS resolves to the load balancer's IP address, not your app servers directly.
2
Load Balancer receives the request
The LB accepts the TCP connection. If it's an L7 LB, it reads HTTP headers (Host, Cookie, URL path) to make routing decisions. SSL is terminated here.
3
Algorithm selects a backend server
Based on the configured algorithm (Round Robin, Least Connections, etc.), the LB picks a healthy backend. Server 3 is down, so it's excluded automatically.
4
Request forwarded to selected server
The LB forwards the request to the chosen backend. It adds headers like X-Forwarded-For (original client IP) and X-Real-IP so your app knows where the request came from.
5
Server processes and responds
The backend server handles the request, queries the database or cache, and sends the response back through the load balancer (or directly to client in DSR mode).
6
Response returned to client
The client receives the response. The entire round-trip typically adds < 1ms overhead — imperceptible to users.
Core Concepts

Load Balancing Algorithms

The algorithm decides which server gets the next request. Choosing the wrong one for your use case can cause hot spots and degraded performance.

// Comparing Load Balancing Algorithms
Algorithm How it works Best for Avoid when
Round Robin Requests cycle sequentially: S1 → S2 → S3 → S1... Identical servers, stateless apps Servers have different capacities
Weighted Round Robin Like Round Robin but with proportional distribution based on weights Mixed hardware (some servers more powerful) Real-time load matters more than capacity
Least Connections Routes to server with fewest active connections Long-lived connections (WebSockets, file uploads) Very short requests (adds tracking overhead)
IP Hash Same client IP always routes to same server (hash-based) Stateful apps without a shared session store Servers go down often (breaks sessions)
Random Picks a server randomly Simple stateless APIs with identical servers You need predictable distribution
Resource-Based Considers CPU/memory of each server in real-time Heavy compute tasks, ML inference Low-overhead APIs (monitoring cost)
Types

Types of Load Balancers

Operates at the TCP/UDP transport layer. It routes traffic based on IP addresses and port numbers without looking at the application data. Much faster since it doesn't inspect packet content.

When to use: Non-HTTP traffic (game servers, SMTP, FTP), extremely high throughput scenarios where L7 inspection overhead matters.

Examples: AWS NLB (Network Load Balancer), HAProxy in TCP mode

Operates at the HTTP/HTTPS application layer. It can read the full request — URL paths, headers, cookies, and body — to make intelligent routing decisions.

Capabilities: Route /api/* to API servers, /static/* to CDN, /checkout to dedicated payment servers. Handle SSL termination, compression, and caching.

Examples: Nginx, HAProxy, AWS ALB (Application Load Balancer), Traefik, Envoy

Routes traffic across multiple geographic data centers using DNS. When a user in India requests api.yourapp.com, DNS resolves to the Mumbai datacenter instead of US-East.

How it works: Uses GeoDNS to return different IP addresses based on the requestor's location. TTLs are set very low (10-60 seconds) for fast failover.

Examples: AWS Route 53 Latency Routing, Cloudflare Load Balancing, Google Cloud Global LB

Hardware LBs (F5 BigIP, Citrix ADC): Physical appliances. Extremely fast (10M+ requests/sec), but expensive ($50k-$1M), hard to scale, and inflexible. Used in large enterprises and telcos.

Software LBs (Nginx, HAProxy): Run on commodity servers. Flexible, cheap, cloud-native, scriptable. Handle millions of req/sec on modern hardware. Used by 99% of companies today.

Configuration & Code

Real-World Configurations

Nginx Load Balancer Config

NGINX nginx.conf 
# ─── Upstream block: defines your server pool ───
upstream app_backend {
    # Algorithm: least_conn routes to server with fewest active connections
    least_conn;

    # Healthy servers with weights
    server 10.0.1.1:3000 weight=5;
    server 10.0.1.2:3000 weight=3;
    server 10.0.1.3:3000 weight=2;

    # Backup server (only used if all above are down)
    server 10.0.1.4:3000 backup;

    # Health check: mark server down after 3 fails in 30s
    # Retry after 30s wait
    server 10.0.1.1:3000 max_fails=3 fail_timeout=30s;

    # Keepalive: maintain persistent connections to upstreams
    keepalive 32;
}

server {
    listen 443 ssl http2;
    server_name api.yourapp.com;

    # SSL termination at the load balancer
    ssl_certificate     /etc/ssl/certs/yourapp.crt;
    ssl_certificate_key /etc/ssl/private/yourapp.key;
    ssl_protocols       TLSv1.2 TLSv1.3;

    # Rate limiting (100 req/s per IP)
    limit_req_zone $binary_remote_addr zone=api:10m rate=100r/s;

    location /api/ {
        limit_req zone=api burst=20 nodelay;

        # Pass request to upstream pool
        proxy_pass         http://app_backend;
        proxy_http_version 1.1;

        # Pass client info to backend
        proxy_set_header X-Real-IP        $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Host              $host;

        # Keepalive for upstream connections
        proxy_set_header Connection "";

        # Timeouts
        proxy_connect_timeout 5s;
        proxy_send_timeout    60s;
        proxy_read_timeout    60s;
    }

    # Separate pool for static assets → CDN/storage
    location /static/ {
        proxy_pass http://static_servers;
        expires    30d;
        add_header Cache-Control "public, max-age=2592000";
    }
}

Node.js App — Trust Proxy & Real IP

JAVASCRIPT app.js (Express) 
const express = require('express');
const app = express();

// CRITICAL: Tell Express to trust the load balancer's X-Forwarded-For header
// Without this, req.ip returns the LB's IP, not the real client IP
app.set('trust proxy', 1);  // trust first proxy hop

// Health check endpoint (used by load balancer)
app.get('/health', (req, res) => {
  // Check DB connectivity, memory, etc.
  const healthy = checkDbConnection() && process.memoryUsage().heapUsed < 512 * 1024 * 1024;
  
  if (healthy) {
    res.status(200).json({ 
      status: 'ok', 
      uptime: process.uptime(),
      serverId: process.env.HOSTNAME
    });
  } else {
    // 503 causes LB to stop routing to this instance
    res.status(503).json({ status: 'degraded' });
  }
});

app.get('/api/users', (req, res) => {
  // req.ip now correctly shows the real client IP
  console.log(`Request from: ${req.ip}`);
  console.log(`Via servers: ${req.headers['x-forwarded-for']}`);
  
  res.json({ message: 'Hello!', server: process.env.HOSTNAME });
});
Industry Usage

How Top Companies Use Load Balancers

🎬 Netflix AWS ALB + Envoy
Netflix uses Envoy service mesh for inter-service load balancing and AWS ALBs for edge traffic. Their "Chaos Monkey" randomly kills servers to ensure the load balancer fails over correctly. They handle 200M+ streaming sessions simultaneously.
🚗 Uber NGINX + Custom
Uber built RING POP, a custom application-layer load balancer for their Node.js microservices. They use consistent hashing so trip data for a specific city always routes to the same cluster of servers for locality.
🔍 Google Maglev
Google built Maglev, their own software L4 LB handling millions of connections per second. Maglev uses consistent hashing with connection tracking tables so existing connections aren't disrupted when servers are added/removed.
📘 Facebook Katran
Facebook's Katran is an eBPF-based L4 LB that runs in the kernel for ultra-low latency. It handles ~billions of requests per day across Facebook, Instagram, and WhatsApp infrastructure.
🛒 Amazon AWS ALB/NLB/GWLB
Amazon uses ALBs for HTTP/HTTPS, NLBs for TCP/UDP gaming, and Gateway LBs for security appliances. They invented target groups and weighted routing for canary deployments (send 5% to new version).
☁️ Cloudflare Unimog
Cloudflare's Unimog load balancer handles traffic across 270+ global PoPs. It uses XDP (eXpress Data Path) for kernel bypass, achieving sub-100µs routing decisions for 70M+ HTTP requests per second.
Tradeoffs

Advantages, Disadvantages & Tradeoffs

✅ Advantages
  • Horizontal scaling — add more servers without downtime
  • High availability — automatic failover when servers die
  • Zero-downtime deployments via rolling updates
  • SSL termination offloads crypto work from app servers
  • DDoS mitigation at the entry point
  • Centralized logging and monitoring point
  • A/B testing and canary deployments via weighted routing
❌ Disadvantages
  • Single point of failure (if not deployed in HA mode)
  • Added network hop increases latency by 1-5ms
  • Session stickiness problems with stateful apps
  • Load balancer itself can become the bottleneck
  • Complexity in debugging (which server handled request?)
  • Additional cost for managed LB services
  • WebSocket sticky sessions need special config

Key Tradeoffs to Discuss in Interviews

Stateless apps (RESTful APIs) work perfectly with round-robin. Any server can handle any request. Stateful apps (shopping carts stored in server memory) break because different requests go to different servers.

Solution: Move state out to a shared store (Redis). This is the real reason why "store sessions in Redis" is a fundamental best practice.

L4 LBs are 3-5x faster (no HTTP parsing), but they can't make content-aware decisions. L7 LBs can route based on URL, headers, cookies — enabling microservice routing and A/B testing — but at higher CPU cost per connection.

Modern answer: Use L7 LBs for web APIs (the intelligence is worth it). Use L4 LBs for non-HTTP services or ultra-high throughput scenarios.

SSL Termination at LB: LB decrypts traffic, internal network runs plain HTTP. Faster (no per-server SSL), easier certificate management. Risk: internal traffic unencrypted.

SSL Passthrough: LB forwards encrypted TCP connections directly. Backend handles decryption. Secure end-to-end but LB can't inspect headers for routing. Can't do cookie-based routing.

SSL Re-encryption: LB terminates SSL, then re-encrypts to backend. Best security, highest cost.

Industry standard: Terminate at LB + use mTLS or private network for backend communication.

Production Concepts

Health Checks — The Heartbeat of Load Balancing

Health checks are what allow load balancers to automatically exclude failed servers. Without them, the LB would continue sending requests to a dead server.

📡
TCP Health Check
LB tries to open a TCP connection to the server's port. If the port accepts the connection → healthy. Doesn't check if the app is functioning, just that something is listening.
🌐
HTTP Health Check
LB sends GET /health and expects HTTP 200. Your app can check DB connections, memory limits, dependencies. Returns 503 to signal "don't send me traffic."
🩺
Deep Health Check
Custom endpoint that checks cache connectivity, queue depth, downstream service availability. Returns detailed JSON status for monitoring dashboards.
JAVASCRIPT healthcheck.js — Production deep health check 
const router = require('express').Router();

router.get('/health', async (req, res) => {
  const checks = {};
  let overallHealthy = true;

  // 1. Database check
  try {
    await db.query('SELECT 1');
    checks.database = { status: 'ok', latency: '2ms' };
  } catch (err) {
    checks.database = { status: 'error', message: err.message };
    overallHealthy = false;
  }

  // 2. Redis cache check
  try {
    await redis.ping();
    checks.cache = { status: 'ok' };
  } catch (err) {
    checks.cache = { status: 'degraded' };
    // Cache failure = degraded, not unhealthy (app still works)
  }

  // 3. Memory check (if too high, signal LB to stop sending)
  const mem = process.memoryUsage();
  const heapUsedMB = Math.round(mem.heapUsed / 1024 / 1024);
  checks.memory = { heapUsedMB, status: heapUsedMB > 800 ? 'warning' : 'ok' };
  if (heapUsedMB > 950) overallHealthy = false;

  const statusCode = overallHealthy ? 200 : 503;

  res.status(statusCode).json({
    status:    overallHealthy ? 'healthy' : 'unhealthy',
    timestamp: new Date().toISOString(),
    uptime:    process.uptime(),
    checks,
    // Include which server responded (useful for debugging)
    server:    process.env.HOSTNAME,
    version:   process.env.APP_VERSION
  });
});
💡
Senior Engineer Tip: Design your health check to be honest but nuanced. Return 503 for hard failures (DB down), but a "degraded" 200 for soft failures (cache miss). This prevents the LB from routing all traffic away during minor blips, causing a thundering herd on the remaining healthy servers.
Scaling

Scaling Strategies with Load Balancers

Making the Load Balancer Itself Highly Available

Ironically, a single load balancer is itself a SPOF. Production setups use multiple LBs in an Active-Passive or Active-Active configuration:

// HA Load Balancer Setup — Active-Passive with VIP
1
Primary LB owns the Virtual IP (VIP)
Both LBs have their own IPs, but only the primary "owns" the VIP (e.g. 10.0.0.1). DNS points to this VIP. VRRP/Keepalived protocol manages ownership.
2
Heartbeat monitoring between LBs
Both LBs send heartbeat packets to each other every 1 second. If the primary misses 3 consecutive heartbeats, the standby assumes the primary is dead.
3
Automatic VIP failover
The standby broadcasts a gratuitous ARP to claim the VIP. All routers update their ARP tables. Failover completes in ~2 seconds. Users experience a brief blip.
Engineering Wisdom

Common Mistakes & Best Practices

🚨 Beginner Mistakes

When running behind a load balancer, req.ip returns the LB's private IP, not the client's real IP. This breaks IP-based rate limiting — everyone looks like they're coming from the same IP.

Fix: app.set('trust proxy', 1) in Express. In Nginx: use $http_x_forwarded_for for rate limiting, not $remote_addr.

Using express-session with the default MemoryStore means user sessions are stored on a specific server. The next request may go to a different server → user gets logged out.

Fix: Use connect-redis or connect-mongo to store sessions in a shared external store. All servers share the same session data.

A health check that just returns {"status":"ok"} without actually checking the database, cache, or dependencies is useless. The LB will keep routing traffic to a server that's connected but broken.

Fix: Genuinely check DB connectivity (SELECT 1), Redis PING, and memory usage. Return 503 when truly unhealthy. Check with a timeout so a slow health check doesn't hang the LB.

If your health check timeout is 1 second but your app sometimes takes 2-3 seconds to respond under load, the LB marks healthy servers as "down" and creates a cascading failure.

Fix: Set timeouts conservatively: check inter 5000 fall 3 rise 2 means: check every 5 seconds, mark down after 3 failures, mark up after 2 successes.

✅ Senior Engineer Best Practices

🔁
Connection Draining
When removing a server from rotation (deployment), enable connection draining. The LB stops sending new requests but lets existing connections finish (e.g. 30s timeout). AWS ALB calls this "deregistration delay."
📊
Slow Start
When a new server joins, ramp up traffic gradually. Nginx's slow_start=30s parameter increases the server's weight from 0 to full over 30 seconds, avoiding JVM cold-start overload.
🔐
mTLS Internally
Don't trust your internal network. Use mutual TLS between the LB and backend servers. Service meshes like Istio do this automatically for every service-to-service call.
📈
Monitor LB Metrics
Track: active connections, requests/sec, error rates (4xx/5xx), backend response time p50/p95/p99, health check pass rates. Alert when any backend shows elevated error rate.
⚠️
Production Warning: Never set max_fails=0 in Nginx — this disables passive health checks entirely. Always set max_fails ≥ 2 with a reasonable fail_timeout to prevent a single slow response from incorrectly marking a server as down.
Security

Security Considerations

🛡️
DDoS Mitigation
Load balancers can absorb and distribute DDoS attacks. Configure connection limits per IP, request rate limits, and SYN cookie protection to handle SYN flood attacks at the LB level.
🔒
IP Whitelisting
Restrict admin endpoints to specific IP ranges at the LB level. E.g., only allow /admin/* from office VPN CIDR blocks. Cheaper than building auth into every microservice.
🔑
WAF Integration
Modern L7 LBs (AWS ALB + WAF, Cloudflare) can inspect requests for SQL injection, XSS patterns, and block malicious payloads before they reach your application code.
📋
Access Logs
Centralize LB access logs (all requests, response codes, latency) to Elasticsearch or Splunk. The LB is the single choke point — you get 100% of traffic visibility in one place.
🚨
Critical Security: Never expose backend server IPs directly to the internet. All traffic MUST route through the load balancer. Use security groups/firewall rules to allow inbound traffic on app ports ONLY from the load balancer's private IP, not from 0.0.0.0/0.
Interview Preparation

Common Interview Questions

What is the difference between a Load Balancer and a Reverse Proxy?
A Reverse Proxy is a server that sits in front of web servers and forwards client requests to them. It handles one or more servers. A Load Balancer is a type of reverse proxy that specifically distributes traffic across a pool of servers. All load balancers are reverse proxies, but not all reverse proxies are load balancers. Nginx can function as both simultaneously — it can proxy and load balance at the same time.
How would you handle WebSocket connections with a load balancer?
WebSockets need sticky sessions (connection persistence) because the connection is long-lived. Configure Nginx with ip_hash or HAProxy with cookie-based persistence. The better solution is to offload WebSocket state to a pub/sub system (Redis Pub/Sub, Kafka) so any server can handle any connection. Also ensure the LB has long proxy_read_timeout values (minutes/hours, not seconds) since WebSocket connections are persistent.
Explain how you would do a zero-downtime deployment with a load balancer.
Rolling deployment strategy: (1) Remove Server 1 from the LB pool (enable connection draining, wait for existing connections to finish). (2) Deploy new version to Server 1. (3) Run smoke tests. (4) Add Server 1 back to the pool. (5) Repeat for Server 2, Server 3, etc. AWS uses this with target group deregistration delay (default 300s). The key is: never remove more servers than you can handle traffic without.
What is the C10K problem and how do load balancers address it?
C10K is the challenge of handling 10,000 concurrent connections on a single server. Traditional thread-per-connection models fail because each thread uses ~8MB stack memory — 10k threads = 80GB RAM. Load balancers solve this by: (1) Distributing connections across many servers, (2) Using event-loop architectures (Nginx's epoll) to handle thousands of connections with minimal threads, (3) Enabling horizontal scaling — you can handle C10M by adding more servers behind the LB.
How does Consistent Hashing work in load balancing?
Consistent hashing maps both servers and requests to points on a circular hash ring. Each request routes to the nearest server clockwise on the ring. Why it's better than regular hash: When you add/remove a server with normal hashing (key % N), almost ALL keys remap to different servers, invalidating cache. With consistent hashing, only (1/N) of keys need to move — minimizing cache misses. Used by Uber, Memcached, Cassandra, and Dynamo.
Design a load balancer for a system handling 1 million requests per second.
At 1M RPS: (1) Use an L4 LB for maximum throughput (Maglev-style or Katran with eBPF). (2) Use ECMP (Equal Cost Multi-Path) routing so multiple LB instances share the VIP. (3) Implement consistent hashing for connection affinity. (4) Use DPDK (Data Plane Development Kit) to bypass the Linux kernel for packet processing. (5) Deploy in multiple AZs with GeoDNS for geographic distribution. (6) Each LB cluster should handle 2x your peak traffic for redundancy. Cloudflare and Google use exactly this architecture.
Production Ops

Production Challenges & Debugging

Scenario: One backend server goes down. The LB routes its traffic to the remaining servers. Those servers are now overloaded, so they start failing health checks. The LB removes more servers. Cascade failure.

Solutions: Circuit breaker pattern (stop sending traffic once error rate exceeds threshold), connection pooling with max limits, autoscaling triggered before LB removes servers, exponential backoff on health check retries.

With Least Connections algorithm, if one backend has a memory leak and all requests slow down, it accumulates the most connections and becomes a black hole — the LB keeps sending more because it has "least connections finishing per second."

Solutions: Use Least Response Time algorithm instead of Least Connections. Set hard proxy_read_timeout values. Monitor p95 latency per backend, not just error rates.

When debugging issues behind a load balancer, you often need to know which server handled a specific request. Solutions: (1) Return X-Server-ID response header. (2) Include server hostname in structured log output. (3) Use distributed tracing (OpenTelemetry, Jaeger) where the trace ID ties together all logs from the same request across all servers.

Summary
🎯 What You Now Know
  • ✦ Load balancers distribute traffic across multiple servers to prevent overload and enable high availability
  • ✦ L4 LBs work at TCP/IP level (fast, no content inspection); L7 LBs work at HTTP level (slower but intelligent routing)
  • ✦ Algorithms: Round Robin (simple), Least Connections (variable requests), Weighted (mixed hardware), IP Hash (session stickiness)
  • ✦ Health checks are what make auto-failover work — design them to be genuinely honest
  • ✦ Session state must be externalized to Redis/DB to work with stateless server pools
  • ✦ The LB itself needs HA (Active-Passive with VIP via VRRP/Keepalived) or you haven't eliminated SPOFs
  • ✦ Connection draining + slow start = safe zero-downtime deployments
