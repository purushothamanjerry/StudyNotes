What are Access Modifiers?
Definition: Access modifiers are keywords that control the visibility and accessibility of classes, methods, constructors, and variables. They are a key part of Encapsulation — hiding internal details and exposing only what is necessary.

Java has 4 access modifiers from most restrictive to least restrictive:

private  →  default  →  protected  →  public

(most restrictive)                          (least restrictive)

3
private
Accessible only within the same class. Not even subclasses or same-package classes can access it. 
public class BankAccount {

    private double balance;     // only this class can access
    private String pin;         // completely hidden

    // Public method to access private field safely
    public double getBalance() {
        return balance;          // ✅ same class can access
    }

    public void deposit(double amount) {
        if (amount > 0) balance += amount;  // validated access
    }

    private void validate() { }  // helper method — hidden
}

BankAccount acc = new BankAccount();
// acc.balance = 1000;  ❌ compile error — private!
// acc.pin = "1234";    ❌ compile error — private!
acc.deposit(1000);      // ✅ through public method
acc.getBalance();        // ✅ through public getter

When to use private:

  • All instance variables (fields) — core of encapsulation
  • Helper methods that are internal implementation details
  • Anything that should never be directly accessed from outside
4
default (package-private)
No keyword written = default access. Accessible within the same package only. Also called "package-private". 
// File: com/company/Animal.java
package com.company;

class Animal {          // default class — no public keyword
    String name;        // default field
    void eat() { }      // default method
}

// File: com/company/Dog.java (SAME package)
package com.company;

class Dog {
    void test() {
        Animal a = new Animal();
        a.name = "Cat";  // ✅ same package — accessible
        a.eat();           // ✅ same package — accessible
    }
}

// File: com/other/Test.java (DIFFERENT package)
package com.other;
// import com.company.Animal; ❌ cannot import — default access!
Use case: Default access is useful for classes that are implementation details of a package — they should only be used within that package, not exposed as public API.
5
protected
Accessible within same package AND subclasses (even in different packages). Specifically designed for inheritance. 
// Package: animals
public class Animal {
    protected String name;       // accessible to subclasses
    protected void breathe() {    // accessible to subclasses
        System.out.println("Breathing...");
    }
}

// Package: pets (different package!)
public class Dog extends Animal {
    public void show() {
        name = "Bruno";   // ✅ subclass can access protected
        breathe();          // ✅ subclass can access protected
    }
}

// Different package, NOT a subclass
public class Test {
    void run() {
        Animal a = new Animal();
        // a.name = "x";   ❌ different package, not subclass
        // a.breathe();    ❌ different package, not subclass
    }
}
⚠️ Protected tricky rule:
A subclass in a different package can access protected members through inheritance (via this or directly), but NOT through a reference to a parent object. 
class Dog extends Animal {
    void test() {
        breathe();          // ✅ via inheritance
        Animal a = new Animal();
        // a.breathe();    ❌ via object reference — NOT allowed!
    }
}
6
public
Accessible from everywhere — any class in any package. No restrictions at all. 
public class MathUtils {           // accessible from anywhere
    public static final double PI = 3.14159;  // public constant

    public static int square(int n) {   // public method
        return n * n;
    }
}

// Any class anywhere can use it
import com.utils.MathUtils;
int s = MathUtils.square(5);       // ✅
double p = MathUtils.PI;           // ✅

When to use public:

  • Class declarations (only one public class per file)
  • API methods that should be accessible to all users
  • Constants (public static final)
  • The main() method

private → 1 place (same class)
default → 2 places (+ same package)
protected → 3 places (+ subclasses)
public → 4 places (everywhere)

Class-level Access Modifiers
Top-level classes (not nested) can only be public or default.
private and protected are NOT allowed for top-level classes.
ModifierTop-level classNested classMembers (fields/methods)
public✅ Yes✅ Yes✅ Yes
default✅ Yes✅ Yes✅ Yes
protected❌ No✅ Yes✅ Yes
private❌ No✅ Yes✅ Yes
⚠️ One public class per file rule:
A Java file can have multiple classes, but only ONE can be public, and the file name must match that public class name. Other classes in the same file must have default access. 
// File: Animal.java
public class Animal { }   // ✅ public — file name must be Animal.java
class Helper { }          // ✅ default — allowed in same file
// public class Other { } ❌ only ONE public class per file!

Best Practices
💡 Principle of Least Privilege — use the most restrictive modifier possible!
Start with private and only open up access when genuinely needed.
WhatRecommended ModifierReason
Instance variables (fields)privateEncapsulation — expose via getters/setters
Helper / utility methodsprivateInternal implementation detail
API methods (public API)publicShould be accessible to all consumers
Methods meant for subclassesprotectedInheritance hook — subclasses can override
Package-internal classesdefaultNot part of public API
Constantspublic static finalShared, immutable, accessible everywhere
// ✅ Good design — encapsulated
public class Employee {
    private String name;    // private field
    private int salary;     // private field

    public String getName() { return name; }    // public getter
    public void setSalary(int s) {              // public setter with validation
        if (s > 0) salary = s;
    }
    private void calculateTax() { }   // private helper
}

Top Interview Questions & Model Answers
Q1. What are the 4 access modifiers in Java? Explain each.
private — accessible only within the same class. Most restrictive.
default (no keyword) — accessible within the same package only.
protected — accessible within the same package AND in subclasses (even in different packages).
public — accessible from anywhere with no restrictions.
Q2. What is the difference between protected and default?
Both are accessible within the same package. The difference is that protected is also accessible in subclasses in different packages through inheritance. Default is strictly package-private — not accessible outside the package even by subclasses.
Q3. Can a top-level class be private or protected?
No. Top-level classes (not nested) can only be public or default. Using private or protected on a top-level class causes a compile error. However, nested (inner) classes can use all 4 access modifiers.
Q4. Why are instance variables usually declared private?
Declaring instance variables private is the core of encapsulation. It prevents direct, uncontrolled access from outside the class. Instead, access is provided through public getter and setter methods, where we can add validation logic. This keeps the internal state of an object safe and consistent.
Q5. Can a private method be overridden?
No. Private methods are not visible to subclasses, so they cannot be overridden. If a subclass defines a method with the same name and signature, it is a completely new method (method hiding), not overriding. This is because overriding requires the method to be inherited first.
Q6. What is the default access modifier?
Default access (also called package-private) is when no access modifier keyword is written. The member or class is accessible to all classes within the same package but not to classes in different packages. It is not a keyword — you simply omit the modifier.
Q7. Can you reduce the access modifier when overriding a method?
No. When overriding a method in a subclass, you can only keep the same or increase the access level — never reduce it. For example, if the parent has a public method, the overriding method cannot be protected or private. This is enforced by the compiler to maintain the Liskov Substitution Principle.
