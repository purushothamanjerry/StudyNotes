What are Generics? Why needed?
Generics allow you to write type-safe, reusable code that works with any data type. Instead of hardcoding a type, you use a type parameter (like T) that gets replaced by an actual type when used. Introduced in Java 5.

3 Benefits of Generics

  • Type Safety — compiler catches type errors at compile time, not runtime
  • Code Reuse — write one class/method that works with any type
  • Eliminates Casting — no need for explicit type casts when retrieving elements

Common Type Parameter Names (conventions)

  • T — Type (general)
  • E — Element (used in collections)
  • K — Key (used in Map)
  • V — Value (used in Map)
  • N — Number
  • R — Return type
  • S, U — Second, third type params

Where Generics Are Used

  • Classes: Box<T>
  • Interfaces: Comparable<T>
  • Methods: <T> T getValue()
  • Collections: List<String>
  • Functional interfaces: Function<T,R>

Before Generics — The Problem
// BEFORE generics (Java 1.4) — raw types
import java.util.ArrayList;

ArrayList list = new ArrayList();  // raw type — no type info!
list.add("Hello");
list.add(42);            // no error — anything goes!
list.add(new Dog());    // still no error

// Problems:
String s = (String) list.get(1);  // ❌ ClassCastException at RUNTIME!
// Need explicit cast — ugly and risky

// AFTER generics (Java 5+)
ArrayList<String> safeList = new ArrayList<>();
safeList.add("Hello");
// safeList.add(42); ❌ COMPILE ERROR — caught early!

String str = safeList.get(0);  // ✅ no cast needed!
Generics shift errors from RUNTIME to COMPILE TIME — much safer!

Generic Class
// Generic class — T is type parameter (placeholder)
public class Box<T> {
    private T value;   // T used as field type

    public Box(T value) {     // T used as constructor param
        this.value = value;
    }

    public T getValue() { return value; }  // T used as return type
    public void setValue(T value) { this.value = value; }

    @Override
    public String toString() { return "Box[" + value + "]"; }
}

// Using Box with different types
Box<String>  strBox  = new Box<>("Hello");   // T = String
Box<Integer> intBox  = new Box<>(42);         // T = Integer
Box<Dog>     dogBox  = new Box<>(new Dog()); // T = Dog

String  s = strBox.getValue();   // no cast needed!
Integer i = intBox.getValue();   // no cast needed!

System.out.println(strBox);  // Box[Hello]
System.out.println(intBox);  // Box[42]

// Diamond operator <> — type inferred from left side (Java 7)
Box<String> b = new Box<>("World");  // compiler infers String

Generic class with multiple type parameters

public class Pair<K, V> {
    private K key;
    private V value;

    public Pair(K key, V value) { this.key = key; this.value = value; }
    public K getKey()   { return key; }
    public V getValue() { return value; }
}

Pair<String, Integer> p = new Pair<>("age", 25);
System.out.println(p.getKey() + ": " + p.getValue());  // age: 25

Generic Method
A generic method declares its own type parameter before the return type: <T> returnType methodName(params). It can be in any class — even non-generic ones. 
public class Utils {

    // Generic method — <T> before return type
    public static <T> void printArray(T[] array) {
        for (T item : array) {
            System.out.print(item + " ");
        }
        System.out.println();
    }

    // Generic method with return type
    public static <T> T getFirst(List<T> list) {
        if (list.isEmpty()) return null;
        return list.get(0);
    }

    // Generic swap
    public static <T> void swap(T[] arr, int i, int j) {
        T temp = arr[i];
        arr[i] = arr[j];
        arr[j] = temp;
    }
}

// Works with ANY array type!
Integer[] ints    = {1, 2, 3};
String[]  strings = {"a", "b", "c"};
Double[]  doubles = {1.1, 2.2};

Utils.printArray(ints);     // 1 2 3
Utils.printArray(strings);  // a b c
Utils.printArray(doubles);  // 1.1 2.2

Bounded Type Parameters
Bounded types restrict what types can be used as type arguments. Use extends to set an upper bound. 
// Unbounded — T can be anything
public <T> void print(T item) { System.out.println(item); }

// Upper bound — T must be Number or its subclass
public <T extends Number> double sum(List<T> list) {
    double total = 0;
    for (T item : list) {
        total += item.doubleValue();  // Number method — allowed!
    }
    return total;
}

// Works with Integer, Double, Long (all extend Number)
List<Integer> ints    = Arrays.asList(1, 2, 3);
List<Double>  doubles = Arrays.asList(1.5, 2.5);
sum(ints);    // ✅ 6.0
sum(doubles); // ✅ 4.0
// sum(Arrays.asList("a","b")); ❌ compile error — String not a Number

// Bound with interface — T must implement Comparable
public <T extends Comparable<T>> T max(T a, T b) {
    return a.compareTo(b) > 0 ? a : b;
}
System.out.println(max(10, 20));        // 20
System.out.println(max("apple", "banana")); // banana

// Multiple bounds — extends class AND interface
public <T extends Number & Comparable<T>> T clamp(T val, T min, T max) {
    if (val.compareTo(min) < 0) return min;
    if (val.compareTo(max) > 0) return max;
    return val;
}

Wildcards — ?, ? extends, ? super
Wildcard ? represents an unknown type. Used in method parameters to increase flexibility. Three forms: unbounded, upper-bounded, lower-bounded.

1. Unbounded Wildcard — ?

// List<?> — list of any type, READ ONLY
public static void printList(List<?> list) {
    for (Object obj : list) {
        System.out.print(obj + " ");
    }
}
printList(List.of(1, 2, 3));          // ✅
printList(List.of("a", "b"));          // ✅
// list.add("x"); ❌ cannot add to wildcard list

2. Upper-Bounded Wildcard — ? extends T

// ? extends Number — Number or any subclass (Integer, Double...)
public static double sumList(List<? extends Number> list) {
    double sum = 0;
    for (Number n : list) sum += n.doubleValue();
    return sum;
}
sumList(List.of(1, 2, 3));         // ✅ Integer extends Number
sumList(List.of(1.5, 2.5));       // ✅ Double extends Number
// Good for READING — you know it's at least a Number

3. Lower-Bounded Wildcard — ? super T

// ? super Integer — Integer or any superclass (Number, Object)
public static void addNumbers(List<? super Integer> list) {
    list.add(1);
    list.add(2);
    list.add(3);
}
List<Number>  nums  = new ArrayList<>();
List<Object> objs  = new ArrayList<>();
addNumbers(nums);   // ✅ Number is super of Integer
addNumbers(objs);   // ✅ Object is super of Integer
// Good for WRITING — you're adding Integers
💡 PECS Rule — Producer Extends, Consumer Super:
If the collection produces (you read from it) → ? extends T
If the collection consumes (you write to it) → ? super T
If both → use explicit type parameter T
WildcardMeaningRead?Write?Use for
?Any typeAs Object❌ NoJust printing/counting
? extends TT or subtype✅ As T❌ NoReading/consuming
? super TT or supertypeAs Object✅ T typesWriting/producing

Multiple Type Parameters
// Generic class with 3 type params
public class Triple<A, B, C> {
    private A first;
    private B second;
    private C third;

    public Triple(A a, B b, C c) {
        this.first  = a;
        this.second = b;
        this.third  = c;
    }
    // getters...
}

Triple<String, Integer, Boolean> t = new Triple<>("Alice", 25, true);

// Generic method with multiple params
public static <K, V> void printMap(Map<K, V> map) {
    for (Map.Entry<K, V> entry : map.entrySet()) {
        System.out.println(entry.getKey() + " → " + entry.getValue());
    }
}
Map<String, Integer> ages = Map.of("Alice", 25, "Bob", 30);
printMap(ages);

Type Erasure
Type Erasure — Java generics are a compile-time feature only. The compiler checks types, then removes all generic type information from the bytecode. At runtime, there are no generics — just raw types. 
// What you write:
List<String> list = new ArrayList<String>();
list.add("hello");
String s = list.get(0);

// What the compiler turns it into (after erasure):
List list = new ArrayList();
list.add("hello");
String s = (String) list.get(0);  // cast inserted by compiler

// At runtime — generics don't exist
List<String> strList  = new ArrayList<>();
List<Integer> intList = new ArrayList<>();
System.out.println(strList.getClass() == intList.getClass()); // true!
// Both are just ArrayList at runtime

Consequences of Type Erasure

  • Cannot do new T() — type not known at runtime
  • Cannot do T[] arr = new T[10] — cannot create generic arrays
  • Cannot use instanceof with generic type: obj instanceof List<String> ❌
  • Cannot have static T field — T is instance-level, not class-level
  • List<String> and List<Integer> are same class at runtime
⚠️ Cannot create generic arrays:
T[] arr = new T[10]; → ❌ compile error
Workaround: T[] arr = (T[]) new Object[10]; or use ArrayList<T>

Top Interview Questions & Model Answers
Q1. What are Generics? What are the benefits?
Generics allow writing type-safe, reusable code that works with any data type using type parameters like T. Three key benefits: (1) Type safety — compiler catches ClassCastException at compile time, not runtime. (2) Code reuse — one class/method works with any type. (3) Eliminates casting — no explicit casts needed when retrieving elements from collections.
Q2. What is the difference between List<?>, List<? extends T>, and List<? super T>?
List<?> — list of unknown type, read-only (only as Object). List<? extends T> — list of T or any subtype — you can read as T but cannot add (producer). List<? super T> — list of T or any supertype — you can add T objects but read only as Object (consumer). Remember PECS: Producer Extends, Consumer Super.
Q3. What is type erasure?
Type erasure is the process by which the Java compiler removes all generic type information from bytecode. Generics are a compile-time feature only — after compilation, List<String> and List<Integer> are both just List at runtime. The compiler adds necessary casts automatically. This was done for backward compatibility with pre-Java 5 code that used raw types.
Q4. Can you use primitive types with generics?
No. Generics only work with reference types (objects), not primitives. You cannot write List<int> — you must use the wrapper class: List<Integer>. Java automatically autoboxes primitives to their wrapper types when adding to generic collections. This is because after type erasure, generics become Object, and primitives don't extend Object.
Q5. What is a bounded type parameter? Give an example.
A bounded type parameter restricts which types can be used as type arguments. <T extends Number> means T must be Number or a subclass (Integer, Double, etc.). This allows calling Number's methods on T inside the method. Use & for multiple bounds: <T extends Number & Comparable<T>> — class bound must come first.
Q6. What is the diamond operator <>?
The diamond operator (introduced Java 7) allows type inference on the right side of a generic declaration. Instead of List<String> list = new ArrayList<String>(), you can write List<String> list = new ArrayList<>() — the compiler infers the type from the left side. It reduces verbosity without losing type safety.
Q7. Can you create a generic array? Why not?
No. T[] arr = new T[10] is not allowed because of type erasure — at runtime, T is erased to Object, and arrays are reifiable (their type is known at runtime). Creating a generic array would allow heap pollution (mixing types). Workaround: use new Object[10] and cast, or prefer ArrayList<T> which handles this internally.
