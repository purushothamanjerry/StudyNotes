What is String in Java?
String is a class in Java (not a primitive type) in the java.lang package. It represents a sequence of characters. Strings are immutable — once created, their value cannot be changed. String is a final class — cannot be subclassed.
  • String is an object, not a primitive — stored on Heap
  • String literals are stored in the String Pool (special Heap area)
  • String is immutable — any modification creates a new String object
  • String implements Comparable<String>, Serializable, CharSequence
  • String is the most used class in Java

Creating Strings — Literal vs new
// Way 1: String literal — stored in String Pool
String s1 = "Hello";
String s2 = "Hello";  // reuses same "Hello" from pool

// Way 2: new keyword — always creates NEW object on Heap
String s3 = new String("Hello");  // new object, outside pool
String s4 = new String("Hello");  // another new object

// Way 3: from char array
char[] chars = {'H', 'e', 'l', 'l', 'o'};
String s5 = new String(chars);  // "Hello"

// Way 4: from StringBuilder/StringBuffer
StringBuilder sb = new StringBuilder("Hello");
String s6 = sb.toString();

// intern() — put string into pool from heap
String s7 = new String("World").intern();  // now in pool

Why String is Immutable
Immutable means once a String object is created, its character sequence CANNOT be changed. Any operation that appears to "modify" a String actually creates a NEW String object. 
String s = "Hello";
s = s + " World";    // "Hello" object still exists in pool!
                       // NEW "Hello World" object created
                       // s now points to new object

String original = "Java";
String upper    = original.toUpperCase();  // creates NEW "JAVA"
System.out.println(original);  // still "Java" — unchanged!
System.out.println(upper);     // "JAVA" — new object

Why immutability?

ReasonExplanation
String Pool safetyMultiple references share same object. If mutable, one ref changing it would affect all others.
SecurityStrings used for class names, network URLs, file paths, passwords. Mutable strings could be changed after security check.
Thread safetyImmutable objects are inherently thread-safe — no synchronization needed.
HashCode cachingString caches its hash code after first calculation. Immutability guarantees the hash never changes — safe for HashMap keys.

String Pool
String Pool (String Intern Pool) is a special area in the Heap where Java stores string literals. When a literal is created, JVM checks the pool first — if it exists, reuses it; if not, adds it. This saves memory. 
String a = "Java";      // "Java" added to pool
String b = "Java";      // "Java" already in pool → reuse same object
String c = new String("Java"); // new object on Heap — NOT in pool

System.out.println(a == b);          // true  — same pool object
System.out.println(a == c);          // false — different objects
System.out.println(a.equals(c));     // true  — same content

// intern() — forces string into pool
String d = c.intern();
System.out.println(a == d);          // true  — d now points to pool object

// Compile-time concatenation — also goes to pool!
String e = "Ja" + "va";  // compiler combines → "Java" in pool
System.out.println(a == e);  // true!

// Runtime concatenation — NOT in pool (new object)
String x = "Ja";
String f = x + "va";     // runtime concat → new heap object
System.out.println(a == f);  // false!
⚠️ Compile-time vs Runtime concatenation in pool:
"Ja" + "va" → compiler resolves to "Java" at compile time → goes to pool ✅
x + "va" (where x is a variable) → resolved at runtime → new Heap object, NOT in pool ❌

String Comparison
String s1 = "Hello";
String s2 = "Hello";
String s3 = new String("Hello");
String s4 = "hello";

// == compares REFERENCES (memory address)
System.out.println(s1 == s2);            // true  — same pool object
System.out.println(s1 == s3);            // false — different objects

// equals() compares CONTENT (case-sensitive)
System.out.println(s1.equals(s2));       // true
System.out.println(s1.equals(s3));       // true
System.out.println(s1.equals(s4));       // false — case sensitive!

// equalsIgnoreCase() — case-insensitive comparison
System.out.println(s1.equalsIgnoreCase(s4));  // true

// compareTo() — lexicographic comparison (like dictionary)
System.out.println("apple".compareTo("banana"));  // negative (a before b)
System.out.println("cat".compareTo("cat"));       // 0 (equal)
System.out.println("dog".compareTo("cat"));       // positive (d after c)

// NEVER compare String with == for content!
// Always use .equals()
💡 Null-safe comparison tip:
str.equals("Hello") — throws NullPointerException if str is null!
"Hello".equals(str) — safe! Literal on left, variable on right — never NPE.

StringBuilder
StringBuilder is a mutable sequence of characters. Unlike String, modifications happen in-place — no new objects created. Not thread-safe. Use in single-threaded code for performance. 
StringBuilder sb = new StringBuilder();
StringBuilder sb2 = new StringBuilder("Hello");
StringBuilder sb3 = new StringBuilder(50); // initial capacity 50

// append — most used method
sb.append("Hello");
sb.append(" ");
sb.append("World");
System.out.println(sb);  // Hello World

// Method chaining
sb.append("!").append("!!");  // Hello World!!!

// insert — add at specific position
sb.insert(5, ",");   // Hello, World!!!

// delete — remove characters
sb.delete(5, 6);    // Hello World!!!

// replace
sb.replace(6, 11, "Java");  // Hello Java!!!

// reverse
new StringBuilder("hello").reverse();  // "olleh"

// Other useful methods
sb.length();           // current length
sb.capacity();         // internal buffer size (default 16)
sb.charAt(0);         // 'H'
sb.indexOf("Java");   // position
sb.toString();         // convert to String

// Why StringBuilder is faster than String concatenation in loops
// BAD — creates 1000 String objects!
String result = "";
for (int i = 0; i < 1000; i++) result += i;

// GOOD — modifies same StringBuilder object
StringBuilder fast = new StringBuilder();
for (int i = 0; i < 1000; i++) fast.append(i);

StringBuffer
StringBuffer is identical to StringBuilder except all its methods are synchronized — making it thread-safe. Use in multi-threaded environments. Slower than StringBuilder due to synchronization overhead. 
StringBuffer sbuf = new StringBuffer("Hello");
sbuf.append(" World");   // synchronized — thread safe
sbuf.reverse();
sbuf.insert(0, ">> ");
System.out.println(sbuf.toString());

// Same API as StringBuilder — just synchronized

Top Interview Questions & Model Answers
Q1. Why is String immutable in Java?
String is immutable for four reasons: (1) String Pool safety — multiple references share the same object; mutability would cause one change to affect all. (2) Security — strings are used for class names, URLs, file paths, passwords — immutability prevents tampering after security checks. (3) Thread safety — immutable objects need no synchronization. (4) HashCode caching — String caches its hash code after first calculation, safe because value never changes — important for HashMap keys.
Q2. What is the String Pool?
The String Pool (intern pool) is a special area in the Heap where Java stores string literals. When a string literal is created, JVM checks the pool first — if the same value exists, it reuses that object (no new allocation). This saves memory since strings are very frequently used. String literals go to the pool automatically; strings created with new String() go to the regular Heap. The intern() method moves a heap string into the pool.
Q3. What is the difference between String, StringBuilder, and StringBuffer?
String is immutable — any modification creates a new object, making it slow for frequent changes. StringBuilder is mutable and fast, but not thread-safe — use in single-threaded code. StringBuffer is mutable and thread-safe (synchronized methods), but slower than StringBuilder due to synchronization overhead — use when multiple threads share the same object. For most cases, prefer StringBuilder over StringBuffer.
Q4. What is the output? String s = "a" + "b" + "c"; Is it in the String Pool?
Yes, "a" + "b" + "c" is a compile-time constant expression. The Java compiler evaluates it to "abc" at compile time and places it in the String Pool. So String s = "a"+"b"+"c" and String t = "abc" would have s == t as true. But if any part is a variable (String x = "a"; x + "bc"), it's a runtime operation and the result goes to the regular Heap.
Q5. How many String objects are created? String s = new String("Hello");
Up to 2 objects: (1) The literal "Hello" is created in the String Pool (if not already there). (2) The new String("Hello") creates a second object on the Heap. If "Hello" was already in the pool, only 1 new object (the Heap one) is created. This is why new String() should be avoided — it wastes memory.
Q6. What is the difference between == and equals() for Strings?
== compares references — checks if both variables point to the same memory location. For strings, this is true for pool literals but false for new String() objects with the same content. .equals() compares the actual character content — always use this for string comparison. The safe idiom is "literal".equals(variable) to avoid NullPointerException.
Q7. How to reverse a String in Java?
Easiest way: use StringBuilder's reverse() method — new StringBuilder(str).reverse().toString(). Alternatively, iterate from end to start using a loop or convert to char array and swap. String has no built-in reverse() method because strings are immutable and reversing would require a new object — hence StringBuilder is the standard approach.
