What is Encapsulation?
Definition: Encapsulation is the OOP principle of bundling data (fields) and the methods that operate on that data into a single unit (class), while hiding the internal state from direct outside access. It is achieved by making fields private and providing controlled access through public getter and setter methods.

Real World Analogy

💊 Medicine Capsule

The medicine (data) is hidden inside the capsule. You take the capsule (use the public interface) — you don't directly touch the medicine inside (private data). The capsule controls how the medicine is delivered.

🏧 ATM Machine

You interact with the ATM through a public interface (buttons, card slot). The internal mechanisms (cash counting, security) are completely hidden. You can't directly access the cash storage.

Encapsulation = Data Hiding + Controlled Access
It is the FIRST and most fundamental OOP principle. The other pillars (Inheritance, Polymorphism, Abstraction) build upon it.
2
How to Achieve Encapsulation

Two simple steps:

  1. Declare all instance variables as private
  2. Provide public getter and setter methods to read and write those variables
// Step 1: private fields — hidden from outside
public class Person {
    private String name;    // ← private
    private int    age;     // ← private
    private String email;   // ← private

    // Step 2: public getter — read access
    public String getName()  { return name;  }
    public int    getAge()   { return age;   }
    public String getEmail() { return email; }

    // Step 2: public setter — write access with validation
    public void setName(String name) {
        if (name != null && !name.isEmpty())
            this.name = name;
    }
    public void setAge(int age) {
        if (age > 0 && age < 150)
            this.age = age;
    }
    public void setEmail(String email) {
        if (email != null && email.contains("@"))
            this.email = email;
    }
}
3
Getters — Reading Data

A getter is a public method that returns the value of a private field. Naming convention: get + FieldName (for boolean: is + FieldName).

private String  name;
private int     age;
private boolean active;
private double  salary;

// Standard getters
public String  getName()   { return name;   }
public int     getAge()    { return age;    }
public double  getSalary() { return salary; }

// Boolean getter uses "is" prefix (not "get")
public boolean isActive()  { return active; }

// Getter can also do processing before returning
public String getFullInfo() {
    return name + " (Age: " + age + ")";
}
💡 Naming Convention:
String name → getter: getName()
int age → getter: getAge()
boolean active → getter: isActive() (NOT getActive())
This convention is used by frameworks like Spring, Hibernate, and Jackson for auto-discovery.
4
Setters — Writing Data with Validation

A setter is a public void method that sets the value of a private field. The KEY advantage over direct field access is validation.

private int    age;
private double salary;
private String email;

// Setter with validation — prevents invalid data!
public void setAge(int age) {
    if (age >= 0 && age <= 150) {
        this.age = age;
    } else {
        System.out.println("Invalid age!");
        // or throw new IllegalArgumentException("Invalid age");
    }
}

public void setSalary(double salary) {
    if (salary >= 0) {
        this.salary = salary;
    }
}

public void setEmail(String email) {
    if (email != null && email.contains("@") && email.contains(".")) {
        this.email = email.toLowerCase();  // can also transform
    }
}

// Usage
Person p = new Person();
p.setAge(25);      // ✅ valid
p.setAge(-5);     // ❌ rejected by validation
p.setAge(999);    // ❌ rejected by validation

Without Encapsulation — The Problem
// BAD — no encapsulation
public class BankAccount {
    public double balance;   // public field — DANGEROUS!
    public String pin;        // public field — DANGEROUS!
}

// Anyone can do this — no control!
BankAccount acc = new BankAccount();
acc.balance = -99999;  // ❌ negative balance — invalid!
acc.balance = 1000000; // ❌ set any amount directly!
acc.pin = "0000";      // ❌ exposed sensitive data!
Problems without encapsulation:
  • Invalid data can be set (negative age, negative balance)
  • Sensitive data is exposed (pin, password, salary)
  • No way to add validation later without breaking existing code
  • Internal implementation cannot change without affecting all users
6
With Encapsulation — The Solution
// GOOD — fully encapsulated
public class BankAccount {
    private double balance;    // hidden
    private String pin;         // hidden
    private String accountNo;   // hidden

    public BankAccount(String accountNo, String pin) {
        this.accountNo = accountNo;
        this.pin       = pin;
        this.balance   = 0.0;
    }

    // Controlled read access
    public double getBalance() { return balance; }

    // Controlled write — with validation
    public void deposit(double amount) {
        if (amount > 0) balance += amount;
        else System.out.println("Invalid deposit amount");
    }

    public void withdraw(double amount, String pin) {
        if (!this.pin.equals(pin)) {
            System.out.println("Wrong PIN!");
            return;
        }
        if (amount > 0 && amount <= balance) {
            balance -= amount;
        } else {
            System.out.println("Insufficient funds or invalid amount");
        }
    }
    // No setter for balance — it can only change via deposit/withdraw
    // No getter for pin — completely hidden!
}

// Usage — controlled and safe
BankAccount acc = new BankAccount("ACC001", "1234");
acc.deposit(1000);               // ✅ controlled
acc.withdraw(500, "1234");      // ✅ PIN verified
acc.withdraw(500, "0000");      // ❌ Wrong PIN
// acc.balance = 999999;         ❌ compile error — private!

Benefits of Encapsulation
BenefitExplanation
Data Hiding Internal state is hidden from outside. Users only see what they need to see.
Validation Setters can validate data before setting. Prevents invalid state (negative age, empty name).
Flexibility Internal implementation can change without affecting the outside. Change field type from int to long — no external code breaks.
Security Sensitive data (password, PIN, salary) can be protected — no getter, no setter, or PIN-verified access.
Maintainability Easier to debug — data can only change through controlled methods. You know exactly where data is being modified.
Read-only / Write-only Only getter = read-only field. Only setter = write-only field. Full control over access.
Loose coupling Classes interact through well-defined interfaces, not internal details. Easier to change one class without affecting others.

Read-only and Write-only Fields
public class Employee {
    private String id;          // read-only — only getter
    private String name;        // read-write — both getter and setter
    private String password;    // write-only — only setter, no getter
    private double salary;      // read-only via getter, write via method

    public Employee(String id, String name) {
        this.id   = id;
        this.name = name;
    }

    // Read-only — getter only, no setter
    public String getId() { return id; }

    // Read-write
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }

    // Write-only — setter only, no getter (security!)
    public void setPassword(String password) {
        this.password = hashPassword(password); // store hashed
    }

    // Controlled salary change — not a simple setter
    public double getSalary() { return salary; }
    public void giveRaise(double percent) {
        if (percent > 0 && percent <= 50)
            salary += salary * percent / 100;
    }

    private String hashPassword(String p) { return "HASHED:" + p; }
}

POJO and JavaBean

POJO — Plain Old Java Object

A simple class with no special restrictions — just fields and methods. No framework dependencies. The term emphasizes simplicity.

Rules: no extends (except Object), no annotations required, no interface required.

JavaBean

A POJO with additional conventions used by frameworks (Spring, Hibernate, JSP).

Rules: (1) private fields, (2) public getters/setters, (3) no-arg constructor, (4) implements Serializable.

// JavaBean — follows all conventions
import java.io.Serializable;

public class StudentBean implements Serializable {

    private static final long serialVersionUID = 1L;

    private String name;   // private fields
    private int    age;

    public StudentBean() { }  // no-arg constructor required!

    // Standard getters and setters
    public String getName()            { return name; }
    public void   setName(String name) { this.name = name; }
    public int    getAge()             { return age; }
    public void   setAge(int age)      { this.age = age; }
}
Why JavaBeans matter: Spring Framework uses JavaBean conventions to auto-inject dependencies and bind form data. Jackson (JSON library) uses getter/setter names to serialize/deserialize JSON. Hibernate uses them for ORM mapping. Following these conventions makes your code work seamlessly with these frameworks.

Top Interview Questions & Model Answers
Q1. What is encapsulation? How do you achieve it in Java?
Encapsulation is the OOP principle of hiding an object's internal state and requiring all interaction to occur through well-defined methods. It is achieved by: (1) declaring all instance variables as private and (2) providing public getter and setter methods for controlled access. This protects data integrity and hides implementation details.
Q2. What is the benefit of using getters and setters over public fields?
Getters and setters provide: (1) Validation — setters can reject invalid values (negative age, null name). (2) Control — make fields read-only (getter only) or write-only (setter only). (3) Flexibility — change internal implementation without breaking external code. (4) Security — sensitive data can be hidden (no getter for password). Direct public fields offer none of these benefits.
Q3. What is data hiding?
Data hiding means making class fields private so they cannot be directly accessed or modified from outside the class. It is the core of encapsulation. Outside code can only interact with the object through its public methods, which can include validation and logic. This prevents the object from being put into an inconsistent or invalid state.
Q4. What is a POJO? What is a JavaBean?
A POJO (Plain Old Java Object) is a simple Java class with no special restrictions — just fields and methods, with no framework dependencies. A JavaBean is a POJO that follows specific conventions: private fields, public getters/setters following naming convention (getX/setX, isX for boolean), a public no-arg constructor, and implements Serializable. Frameworks like Spring and Hibernate rely on JavaBean conventions.
Q5. Can you have a class with only getters (no setters)? What does that mean?
Yes. A class with only getters and no setters creates immutable objects — once created, their state cannot be changed. Fields are set via the constructor and cannot be modified afterward. This is a common design pattern for thread safety and defensive programming. String in Java is a classic example of an immutable class.
Q6. Is encapsulation the same as abstraction?
No, they are related but different. Encapsulation is about hiding DATA — wrapping fields and methods together and restricting direct access to fields. Abstraction is about hiding COMPLEXITY — showing only essential features and hiding implementation details (achieved through abstract classes and interfaces). Encapsulation is implemented at the class level with private fields; abstraction is implemented at the design level with abstract classes/interfaces.
Q7. Why should a boolean getter use "is" instead of "get"?
Java naming conventions (and the JavaBean specification) state that getters for boolean fields should be prefixed with is instead of get. Example: private boolean active → getter is isActive() not getActive(). Frameworks like Spring, Hibernate, and JSON libraries follow this convention for auto-discovery of boolean properties. Using get for boolean still works technically but violates convention.
