Step 1 — Compilation

What does javac do?

The javac compiler reads your .java file and produces a .class file containing bytecode.

# Source file
HelloWorld.java   ← plain text, human-readable Java code

# Compile command
javac HelloWorld.java

# Output
HelloWorld.class  ← bytecode (binary, not human-readable)

What is Bytecode?

  • Bytecode is an intermediate representation — not machine code, not source code
  • It is platform-neutral — not tied to any OS or CPU
  • Stored in .class files
  • Only a JVM can understand and execute bytecode
  • This is why Java is platform-independent — the same .class file runs on any OS

Java (Compiled + Interpreted)

.java → bytecode → JVM interprets/JIT compiles → runs

Platform-independent bytecode

C/C++ (Compiled only)

.c → machine code → runs directly on CPU

Platform-dependent — must recompile for each OS

Step 2 — Class Loading

The ClassLoader is a subsystem of JVM that loads .class files into memory when needed. Java loads classes dynamically — only when they are first used.

3 Built-in ClassLoaders

ClassLoaderWhat it loadsExample
Bootstrap ClassLoader Core Java classes — the foundation java.lang.String, java.lang.Object
Extension ClassLoader Extension libraries from jre/lib/ext Security extensions, XML parsers
Application ClassLoader Your application classes (classpath) HelloWorld.class, your own classes

3 Phases of Class Loading

  1. Loading — reads the .class file from disk and brings it into memory
  2. Linking — 3 sub-steps:
    • Verification — checks bytecode is valid and safe
    • Preparation — allocates memory for static variables, sets default values
    • Resolution — replaces symbolic references with actual memory references
  3. Initialization — runs static initializers and assigns actual values to static variables
💡 Key Point: Classes are loaded lazily — only when first referenced. This makes Java startup faster.

Step 3 — Bytecode Verification

Before executing bytecode, the JVM's Bytecode Verifier checks it for safety. This is a security feature.

What does it check?

  • Code doesn't violate access restrictions (private, protected)
  • No stack overflow or underflow
  • All variables are initialized before use
  • Method calls have correct number and type of arguments
  • No illegal type casts
Why this matters: When you download a Java app from the internet, you don't know if the bytecode was produced by a safe compiler. The verifier ensures even tampered or maliciously crafted bytecode cannot harm your system.

Step 4 — Execution (Interpreter + JIT)

The JVM has two ways to execute bytecode:

Interpreter

  • Reads and executes bytecode one instruction at a time
  • Simple and fast to start
  • Slow for repeated code — translates the same line every time it's encountered
  • Used for code that runs only once or rarely

JIT Compiler (Just-In-Time)

  • Monitors bytecode execution — finds "hot code" (code run many times)
  • Compiles hot code to native machine code once
  • Caches the result — subsequent calls use native code directly
  • Makes Java nearly as fast as C++ for long-running applications

How JIT Works

// This loop runs 10,000 times — JIT detects it as "hot"
for (int i = 0; i < 10000; i++) {
    calculate(i);   // ← JIT compiles calculate() to native code
}
// First few calls: Interpreter (slow)
// After JIT kicks in: native machine code (fast)
Interview line: "Java uses a hybrid approach — Interpreter for startup speed, JIT for runtime performance. This is why Java is slow to start but fast once warmed up."

JVM Memory Areas

JVM Memory Layout

Heap
All objects & arrays live here. Shared across all threads. GC manages this.
Method Area
Class metadata, static variables, method code. Shared across threads.
Stack
One per thread. Stores method call frames — local variables, return address.
PC Register
One per thread. Stores address of current instruction being executed.
Native Stack
For native (C/C++) method calls via JNI.

⚠️ Common Interview Trick:
"Where is a local variable stored?" → Stack
"Where is an object stored?" → Heap
"Where is a reference variable stored?" → Stack (the variable itself), but the object it points to is on the Heap

Garbage Collection

Garbage Collection (GC) is the process by which JVM automatically frees memory occupied by objects that are no longer reachable (referenced) by the program.

How does GC decide what to collect?

An object is eligible for GC when it has no live references pointing to it.

Dog d = new Dog();   // Dog object created on heap
d = null;             // reference removed → Dog object now eligible for GC
// GC will free the Dog object's memory at some point

Key GC Facts

  • GC runs automatically — you cannot force it (calling System.gc() is only a suggestion)
  • GC only collects objects on the Heap
  • The finalize() method (deprecated in Java 9+) was called before GC collected an object
  • Java has multiple GC algorithms: Serial GC, Parallel GC, G1 GC (default Java 9+), ZGC (Java 11+)

Generational GC — How Heap is Organized

GenerationWhat it containsGC frequency
Young Generation (Eden + Survivor)Newly created objectsVery frequent (Minor GC)
Old Generation (Tenured)Long-lived objects promoted from YoungLess frequent (Major GC)
Metaspace (Java 8+)Class metadata (replaced PermGen)Rare
Why Generational GC? Most objects die young (created and immediately discarded). By collecting short-lived objects frequently in a small Young Generation, GC is fast and efficient. Long-lived objects are promoted to Old Generation and collected less often.

Top Interview Questions & Model Answers
Q1. Explain how a Java program runs from source code to output.
You write source code in a .java file. The javac compiler compiles it into bytecode stored in a .class file. When you run the program, JVM's ClassLoader loads the .class file into memory. The Bytecode Verifier checks it for safety. The Execution Engine then runs it — the Interpreter executes bytecode line by line, while the JIT compiler detects hot code and compiles it to native machine code for better performance. Finally the output is produced.
Q2. What is the difference between Interpreter and JIT Compiler?
The Interpreter translates and executes bytecode one line at a time — simple but slow for repeated code. The JIT (Just-In-Time) Compiler monitors execution, detects frequently-called code ("hot code"), compiles it to native machine code once, and caches it for reuse. This makes Java fast for long-running applications. Java uses both — Interpreter for startup, JIT for performance.
Q3. What is the difference between Heap and Stack memory?
Heap stores all objects and arrays — it is shared across all threads and managed by Garbage Collector. Stack stores method call frames including local variables and references — each thread has its own stack, and it follows LIFO. When heap is full we get OutOfMemoryError; when stack is full we get StackOverflowError (usually from infinite recursion).
Q4. What is Garbage Collection? Can we force it?
Garbage Collection is JVM's automatic memory management — it identifies objects with no live references and frees their heap memory. We cannot force GC — calling System.gc() is only a hint to JVM, not a guarantee. GC runs in the background and is managed entirely by JVM.
Q5. What is the role of the ClassLoader?
ClassLoader is a subsystem of JVM responsible for loading .class files into memory when they are first needed. Java uses three built-in ClassLoaders: Bootstrap (core classes like java.lang), Extension (JRE extension libraries), and Application (your classpath classes). Classes are loaded lazily — only when first referenced.
Q6. What is StackOverflowError? When does it occur?
StackOverflowError occurs when the JVM Stack runs out of space. This typically happens with infinite recursion — a method that calls itself endlessly fills up the stack with frames until there is no space left. Each method call adds a new frame to the stack; infinite calls exhaust it.
