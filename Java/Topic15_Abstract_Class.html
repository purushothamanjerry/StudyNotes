What is an Abstract Class?
Definition: An abstract class is a class declared with the abstract keyword. It cannot be instantiated directly — you must extend it and provide implementations for all abstract methods. It provides a partial blueprint — some behavior is defined, some is left for subclasses.

Real World Analogy

Think of a Vehicle blueprint — it defines common properties (speed, fuel) and behaviors (start, stop). But you can't build a "Vehicle" — you build a Car, Bike, or Truck. The Vehicle is the abstract concept.

Key Idea

Abstract class = "Family template"

Has some shared code ready to use.

Forces subclasses to fill in missing pieces.

Represents an incomplete concept.

Syntax & Basic Example
// Abstract class — cannot be instantiated
abstract class Animal {

    // Instance variable — allowed
    String name;
    int age;

    // Abstract method — no body, MUST be overridden
    public abstract void makeSound();

    // Concrete method — has body, inherited as-is
    public void eat() {
        System.out.println(name + " is eating");
    }

    public void sleep() {
        System.out.println(name + " is sleeping");
    }
}

// Concrete subclass — MUST implement makeSound()
class Dog extends Animal {
    @Override
    public void makeSound() {
        System.out.println("Woof!");
    }
}

class Cat extends Animal {
    @Override
    public void makeSound() {
        System.out.println("Meow!");
    }
}

// Animal a = new Animal(); ❌ Cannot instantiate!
Animal dog = new Dog(); // ✅ parent ref → child object
dog.name = "Bruno";
dog.makeSound();  // Woof!
dog.eat();        // Bruno is eating (inherited)

Abstract Methods
Abstract method = a method declared with abstract keyword that has NO body. It ends with a semicolon. The subclass is FORCED to provide an implementation. 
// Abstract method — declaration only, no body
public abstract void makeSound();        // ← ends with ;
public abstract double calculateArea();  // ← no { } block
public abstract String getType();        // ← must be overridden

// ❌ These are NOT allowed for abstract methods:
// private abstract void method();   — private can't be overridden
// static abstract void method();    — static doesn't belong to object
// final abstract void method();     — final can't be overridden

Rules for Abstract Methods

  • Can only be in an abstract class or interface
  • Cannot be private — subclass must be able to override it
  • Cannot be static — abstract means it needs to be overridden by instance
  • Cannot be final — final prevents overriding
  • If subclass does NOT implement all abstract methods → subclass must also be abstract

Concrete Methods in Abstract Class

Abstract classes can have regular (concrete) methods with full implementation. Subclasses inherit these and can optionally override them.

abstract class Vehicle {
    private String brand;
    private int    speed;

    // Constructor
    public Vehicle(String brand, int speed) {
        this.brand = brand;
        this.speed = speed;
    }

    // Abstract — subclass must implement
    public abstract void fuelType();

    // Concrete — shared behavior
    public void start() {
        System.out.println(brand + " starting...");
    }

    public void stop() {
        System.out.println(brand + " stopping...");
    }

    public int getSpeed() { return speed; }
    public String getBrand() { return brand; }
}

class Car extends Vehicle {
    public Car(String brand) { super(brand, 120); }

    @Override
    public void fuelType() {
        System.out.println("Petrol / Diesel");
    }
}

class ElectricCar extends Vehicle {
    public ElectricCar(String brand) { super(brand, 150); }

    @Override
    public void fuelType() {
        System.out.println("Electric battery");
    }
}

Constructor in Abstract Class
Abstract classes CAN have constructors — even though you can't create objects directly. The constructor is called via super() when a subclass is instantiated. 
abstract class Shape {
    String color;

    // Abstract class constructor
    public Shape(String color) {
        this.color = color;
        System.out.println("Shape created with color: " + color);
    }

    public abstract double area();
}

class Circle extends Shape {
    double radius;

    public Circle(String color, double radius) {
        super(color);   // calls Shape constructor
        this.radius = radius;
    }

    @Override
    public double area() {
        return Math.PI * radius * radius;
    }
}

Circle c = new Circle("Red", 5.0);
// Output: Shape created with color: Red
System.out.println(c.area());  // 78.53...
💡 Why constructor in abstract class?
To initialize shared fields (like color, id) that all subclasses need. Instead of each subclass repeating the same initialization code, the parent constructor handles it once.

Real World Example — Shape
abstract class Shape {
    private String color;

    public Shape(String color) { this.color = color; }
    public String getColor()    { return color; }

    // Every shape MUST define its area and perimeter
    public abstract double area();
    public abstract double perimeter();

    // Shared method — same for all shapes
    public void describe() {
        System.out.println(getClass().getSimpleName() +
            " | Color: " + color +
            " | Area: " + String.format("%.2f", area()) +
            " | Perimeter: " + String.format("%.2f", perimeter()));
    }
}

class Circle extends Shape {
    private double r;
    public Circle(String color, double r) { super(color); this.r = r; }

    @Override public double area()      { return Math.PI * r * r; }
    @Override public double perimeter() { return 2 * Math.PI * r; }
}

class Rectangle extends Shape {
    private double l, w;
    public Rectangle(String color, double l, double w) {
        super(color); this.l = l; this.w = w;
    }
    @Override public double area()      { return l * w; }
    @Override public double perimeter() { return 2 * (l + w); }
}

// Polymorphic usage
Shape[] shapes = {
    new Circle("Red", 5),
    new Rectangle("Blue", 4, 6)
};
for (Shape s : shapes) s.describe();
// Circle  | Color: Red  | Area: 78.54 | Perimeter: 31.42
// Rectangle | Color: Blue | Area: 24.00 | Perimeter: 20.00

Template Method Pattern
Template Method Pattern — define the skeleton of an algorithm in an abstract class, letting subclasses fill in specific steps. The overall structure is fixed; only the details vary. 
abstract class DataProcessor {

    // Template method — defines the algorithm skeleton
    public final void process() {
        readData();    // step 1
        processData(); // step 2 — abstract, varies per subclass
        saveData();    // step 3
    }

    private void readData() {
        System.out.println("Reading data from source...");
    }

    // Abstract — each subclass implements differently
    protected abstract void processData();

    private void saveData() {
        System.out.println("Saving processed data...");
    }
}

class CSVProcessor extends DataProcessor {
    @Override
    protected void processData() {
        System.out.println("Processing CSV data...");
    }
}

class JSONProcessor extends DataProcessor {
    @Override
    protected void processData() {
        System.out.println("Processing JSON data...");
    }
}

new CSVProcessor().process();
// Reading data...  Processing CSV data...  Saving data...
new JSONProcessor().process();
// Reading data...  Processing JSON data...  Saving data...

Choose Abstract Class when:

  • Classes share common state (fields)
  • Need a constructor to initialize shared fields
  • Classes are closely related (IS-A)
  • Want to provide partial implementation

Choose Interface when:

  • Classes are unrelated but share behavior
  • Need multiple inheritance
  • Defining a pure contract/API
  • Want to use with lambdas (functional interface)

Top Interview Questions & Model Answers
Q1. What is an abstract class? When would you use it?
An abstract class is declared with the abstract keyword and cannot be instantiated directly. It can have both abstract methods (no body — must be overridden) and concrete methods (with body — inherited). Use it when you have closely related classes that share common state (fields) and some common behavior, but each must implement certain methods differently. Example: Shape has area() abstract but describe() concrete.
Q2. Can abstract class have a constructor?
Yes. Even though you cannot instantiate an abstract class directly, its constructor is called via super() from the subclass constructor. The purpose is to initialize shared fields that all subclasses need. This avoids duplicate initialization code in every subclass.
Q3. Can we have an abstract class with no abstract methods?
Yes, this is perfectly valid. A class can be declared abstract without having any abstract methods. The sole effect is that the class cannot be instantiated directly. This is useful when you want to prevent direct instantiation of a base class while still providing a fully implemented utility class.
Q4. What happens if a subclass doesn't implement all abstract methods?
If a subclass extends an abstract class but does not implement all abstract methods, the subclass itself must also be declared abstract. Otherwise, the compiler throws an error. The responsibility to implement the abstract methods is then passed down to the next concrete (non-abstract) subclass in the hierarchy.
Q5. What is the difference between abstract class and interface?
Abstract class supports constructors, instance variables with any modifier, and methods with any access level — used for closely related classes sharing state. Interface has only public static final variables and public methods — used to define a contract for unrelated classes. A class can extend only one abstract class but implement multiple interfaces. Since Java 8, interfaces can have default and static methods, making the gap smaller, but abstract classes are still preferred when shared state is needed.
Q6. Can an abstract method be private?
No. An abstract method cannot be private because private methods are not inherited or visible to subclasses — but the entire purpose of an abstract method is to be overridden by a subclass. This is a contradiction. Abstract methods must be at least package-private (default), or preferably protected or public, so subclasses can see and override them.
Q7. What is the Template Method pattern? How does abstract class enable it?
The Template Method pattern defines the skeleton of an algorithm in an abstract class with a final concrete method, while deferring some steps to abstract methods that subclasses implement. The abstract class controls the overall flow (which steps and in what order), and subclasses only fill in the specific steps. This ensures the algorithm structure is consistent while allowing variations in specific steps. Example: a DataProcessor with fixed read→process→save flow where only process() varies per subclass.
