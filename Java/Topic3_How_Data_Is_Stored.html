Two Types of Data in Java
Golden Rule: Java has exactly two kinds of data — Primitives (stored directly by value) and Reference types (stored as a pointer to an object on the heap).

Primitive Types (8 total)

Store the actual value directly in memory.

Examples: int, double, boolean, char

Live on the Stack (when local variables)

Fixed size — always same bytes regardless of OS

Reference Types

Store a memory address (reference) pointing to an object.

Examples: String, int[], Dog, any class

Object lives on Heap, reference lives on Stack

Default value is null (no object)

Stack Memory

Stack memory stores data related to method execution. It follows LIFO (Last In First Out) — when a method is called, a new frame is pushed; when it returns, the frame is popped.

What is stored on Stack?

  • Local variables (primitive types stored directly by value)
  • Reference variables (the address/pointer, not the object itself)
  • Method call frames (return address, parameters)
  • Partial results of expressions

Stack Characteristics

  • Each thread has its own Stack — thread safe by nature
  • Memory is automatically freed when method returns
  • Much faster than Heap (simple pointer increment)
  • Smaller in size than Heap
  • Full Stack → StackOverflowError (usually infinite recursion)
public void calculate() {
    int x = 10;          // x = 10 stored directly on Stack
    int y = 20;          // y = 20 stored directly on Stack
    int sum = x + y;    // sum = 30 stored on Stack
}   // method returns → all 3 variables popped from Stack automatically
4
Heap Memory

Heap memory is where all objects and arrays live. It is shared across all threads and managed by the Garbage Collector.

What is stored on Heap?

  • All objects created with new keyword
  • All arrays (even primitive arrays like int[])
  • Instance variables (fields of an object)
  • String pool (special area inside Heap)
  • Static variables (in Method Area, part of Heap in Java 8+)

Heap Characteristics

  • Shared across all threads — needs synchronization for thread safety
  • Managed by Garbage Collector — auto cleanup of unreferenced objects
  • Larger in size than Stack
  • Slower than Stack (complex allocation)
  • Full Heap → OutOfMemoryError
public void createDog() {
    Dog d = new Dog("Bruno");
    // 'd' reference → stored on STACK
    // Dog object (name="Bruno") → stored on HEAP
    // Dog's field 'name' → also on HEAP (inside the object)
}

String Pool — Special Memory Area

The String Pool (also called String Intern Pool) is a special area inside the Heap where Java stores string literals to avoid duplicate objects.

How String Pool works

String s1 = "Hello";   // "Hello" created in String Pool
String s2 = "Hello";   // reuses SAME "Hello" from pool — no new object!
String s3 = new String("Hello"); // forces NEW object on Heap (outside pool)

System.out.println(s1 == s2);       // true  — same pool reference
System.out.println(s1 == s3);       // false — different objects
System.out.println(s1.equals(s3));  // true  — same content
Key Rule:
== compares references (memory address)
.equals() compares content (actual characters)
Always use .equals() to compare Strings!

Why String is Immutable

Strings in Java are immutable — once created, their value cannot be changed. This is essential for the String Pool to work safely — if one reference modifies the string, it would affect all other references pointing to the same pool object.

String s = "Hello";
s = s + " World";   // Does NOT modify "Hello"
                      // Creates a NEW String "Hello World" on Heap
                      // s now points to the new object
                      // "Hello" still exists in pool (until GC)

Pass by Value vs Pass by Reference
Most asked tricky question: Java is ALWAYS pass by value — never pass by reference. But the "value" passed for objects is the reference (address).

Primitives — pass by value (copy)

public static void change(int x) {
    x = 100;   // changes LOCAL copy only
}

int a = 10;
change(a);
System.out.println(a);  // still 10 — original not changed

Objects — pass reference by value

public static void changeName(Dog d) {
    d.name = "Max";   // modifies the ACTUAL object on heap
}

Dog dog = new Dog("Bruno");
changeName(dog);
System.out.println(dog.name);  // Max — object was modified!
Why? When you pass an object, Java copies the reference (memory address). Both the original and parameter point to the same heap object — so modifying the object's fields affects the original. But reassigning the parameter (d = new Dog()) does NOT affect the original reference.

Wrapper Classes & Autoboxing

Every primitive has a corresponding Wrapper Class — an object version of the primitive. Needed when working with Collections (which only accept objects).

PrimitiveWrapper ClassExample
intIntegerInteger i = 42;
doubleDoubleDouble d = 3.14;
booleanBooleanBoolean b = true;
charCharacterCharacter c = 'A';
longLongLong l = 100L;
floatFloatFloat f = 1.5f;
byteByteByte by = 10;
shortShortShort s = 200;

Autoboxing & Unboxing

Autoboxing — automatic conversion from primitive to wrapper object.
Unboxing — automatic conversion from wrapper object to primitive.

// Autoboxing — primitive → object (automatic)
Integer i = 42;        // compiler converts to: Integer.valueOf(42)

// Unboxing — object → primitive (automatic)
int x = i;             // compiler converts to: i.intValue()

// Used in Collections — List only accepts objects
List<Integer> list = new ArrayList<>();
list.add(10);          // autoboxing: 10 → Integer(10)
int val = list.get(0); // unboxing: Integer(10) → 10
⚠️ Integer Cache Trap (common interview trick): 
Integer a = 127;
Integer b = 127;
System.out.println(a == b);   // true! (cached -128 to 127)

Integer c = 128;
Integer d = 128;
System.out.println(c == d);   // false! (outside cache range)
Java caches Integer objects from -128 to 127. Outside this range, new objects are created — so == returns false. Always use .equals() for Integer comparison!

Top Interview Questions & Model Answers
Q1. Where are local variables stored in Java?
Local variables are stored on the Stack. If the local variable is a primitive, its value is stored directly on the Stack. If it is a reference type, the reference (address) is stored on the Stack, but the actual object is stored on the Heap.
Q2. Where are instance variables stored?
Instance variables are stored on the Heap — inside the object they belong to. When you create an object with new, the object and all its instance variables are allocated on the Heap.
Q3. What is the String Pool? Why does Java have it?
The String Pool is a special area in the Heap where Java stores string literals. When you create a string literal like "Hello", Java checks the pool first — if it already exists, it reuses the same object instead of creating a new one. This saves memory since strings are used very frequently and are immutable.
Q4. Is Java pass by value or pass by reference?
Java is always pass by value. For primitives, a copy of the value is passed. For objects, a copy of the reference (memory address) is passed — so both the caller and the method point to the same object, meaning object fields can be modified, but reassigning the reference inside the method doesn't affect the original.
Q5. What is autoboxing? Give an example.
Autoboxing is the automatic conversion of a primitive type to its wrapper class by the Java compiler. For example, Integer i = 5 — the compiler automatically converts the int value 5 to an Integer object. The reverse (wrapper to primitive) is called unboxing.
Q6. What is the output? Integer a = 127; Integer b = 127; System.out.println(a == b);
true — because Java caches Integer objects for values between -128 and 127. Both a and b point to the same cached object. But if the value were 128 or above, == would return false because new objects would be created outside the cache range.
Q7. Why is String immutable in Java?
String is immutable for three reasons: (1) String Pool safety — multiple references share the same object; if one could change it, others would be affected. (2) Security — strings are used for class names, network connections, file paths — mutability would be a security risk. (3) Thread safety — immutable objects are inherently thread safe.
