What is an Exception?
Exception = an abnormal condition that disrupts the normal flow of a program. When something unexpected happens (divide by zero, null pointer, file not found), Java creates an Exception object and "throws" it. If not handled, the program terminates abruptly.

Without Exception Handling

Program crashes immediately on error

User sees ugly stack trace

Resources not released properly

No graceful recovery

With Exception Handling

Error is caught and handled gracefully

User sees meaningful message

Resources released in finally block

Program can continue or exit cleanly

Exception Hierarchy
Throwable
├── Error                                ← JVM errors — don't handle
│    ├── OutOfMemoryError
│    ├── StackOverflowError
│    └── VirtualMachineError
└── Exception                         ← Handle these!
      ├── IOException                   ← Checked
      ├── SQLException                 ← Checked
      ├── ClassNotFoundException       ← Checked
      └── RuntimeException             ← Unchecked
            ├── NullPointerException
            ├── ArrayIndexOutOfBoundsException
            ├── ArithmeticException
            ├── ClassCastException
            ├── NumberFormatException
            └── IllegalArgumentException
Key distinction:
Throwable — root of all exceptions
Error — serious JVM problems, don't catch
Exception — problems you should handle
RuntimeException — subclass of Exception, unchecked

4
try-catch Block
try — code that might throw an exception
catch — handles the exception if thrown
Code after the exception in try block is SKIPPED when exception occurs 
// Basic try-catch
try {
    int result = 10 / 0;              // ArithmeticException thrown here
    System.out.println("This skipped"); // NEVER runs!
} catch (ArithmeticException e) {
    System.out.println("Cannot divide by zero: " + e.getMessage());
}
System.out.println("Program continues..."); // runs after catch

// Exception object methods
try {
    String s = null;
    s.length();  // NullPointerException
} catch (NullPointerException e) {
    System.out.println(e.getMessage());    // null (or description)
    System.out.println(e.getClass());      // class java.lang.NullPointerException
    e.printStackTrace();                   // full stack trace to console
}
5
finally Block
finally block ALWAYS executes — whether an exception occurs or not, whether it is caught or not. Used for cleanup: closing files, database connections, releasing resources. 
try {
    System.out.println("try block");
    int x = 10 / 0;  // exception thrown
} catch (ArithmeticException e) {
    System.out.println("catch block");
} finally {
    System.out.println("finally block");  // ALWAYS runs!
}
// Output: try block → catch block → finally block

// Even when no exception
try {
    System.out.println("try — no exception");
} catch (Exception e) {
    System.out.println("catch — won't run");
} finally {
    System.out.println("finally — still runs!");
}
// Output: try — no exception → finally — still runs!

// Resource cleanup pattern
Connection conn = null;
try {
    conn = getConnection();
    // use connection
} catch (SQLException e) {
    e.printStackTrace();
} finally {
    if (conn != null) conn.close();  // always close!
}
⚠️ finally does NOT run in one case:
System.exit() terminates the JVM — finally is skipped.
Also: if the JVM crashes (power loss, etc.), finally may not run.

Multiple catch & Multi-catch (Java 7)
// Multiple catch blocks — most specific FIRST!
try {
    int[] arr = {1, 2, 3};
    System.out.println(arr[5]);   // ArrayIndexOutOfBoundsException
    int x = 10 / 0;               // ArithmeticException
} catch (ArrayIndexOutOfBoundsException e) {
    System.out.println("Array error: " + e.getMessage());
} catch (ArithmeticException e) {
    System.out.println("Math error: " + e.getMessage());
} catch (Exception e) {         // general catch — always LAST
    System.out.println("General error: " + e.getMessage());
}

// Multi-catch (Java 7) — handle multiple exceptions same way
try {
    // some code
} catch (IOException | SQLException e) {  // pipe | separates types
    System.out.println("IO or DB error: " + e.getMessage());
}
⚠️ Most specific exception MUST come before general!
Putting catch (Exception e) first makes all specific catches unreachable — compile error!
Order: specific → general (child before parent)
7
throw keyword
throw is used to explicitly throw an exception from your code. You use it when you want to signal that something went wrong based on your business logic. 
// throw an exception explicitly
public void setAge(int age) {
    if (age < 0 || age > 150) {
        throw new IllegalArgumentException("Invalid age: " + age);
    }
    this.age = age;
}

public void withdraw(double amount) {
    if (amount > balance) {
        throw new ArithmeticException("Insufficient funds");
    }
    balance -= amount;
}

// throw stops method execution immediately
public void validate(String name) {
    if (name == null) {
        throw new NullPointerException("Name cannot be null");
    }
    if (name.isEmpty()) {
        throw new IllegalArgumentException("Name cannot be empty");
    }
}
8
throws keyword
throws is used in the method signature to declare that this method might throw certain checked exceptions. It tells the caller: "you must handle these exceptions." 
// throws in method signature — declares possible exceptions
public void readFile(String path) throws IOException {
    FileReader fr = new FileReader(path);  // may throw IOException
    // not handled here — propagated to caller
}

// Caller MUST handle
public void main() {
    try {
        readFile("data.txt");
    } catch (IOException e) {
        System.out.println("File error: " + e.getMessage());
    }
}

// Multiple exceptions in throws
public void process() throws IOException, SQLException {
    // may throw either
}

Custom Exceptions
Create your own exception class by extending Exception (checked) or RuntimeException (unchecked). Use when built-in exceptions don't convey the business meaning. 
// Custom checked exception
public class InsufficientFundsException extends Exception {
    private double amount;

    public InsufficientFundsException(double amount) {
        super("Insufficient funds. Need: " + amount);
        this.amount = amount;
    }

    public double getAmount() { return amount; }
}

// Custom unchecked exception
public class InvalidAgeException extends RuntimeException {
    public InvalidAgeException(String message) {
        super(message);
    }
}

// Using custom exceptions
public class BankAccount {
    private double balance = 1000;

    public void withdraw(double amount) throws InsufficientFundsException {
        if (amount > balance) {
            throw new InsufficientFundsException(amount - balance);
        }
        balance -= amount;
    }
}

// Handle custom exception
try {
    account.withdraw(1500);
} catch (InsufficientFundsException e) {
    System.out.println(e.getMessage());  // Insufficient funds. Need: 500.0
    System.out.println("Short by: " + e.getAmount());
}

try-with-resources (Java 7)
try-with-resources automatically closes resources (files, connections, streams) when the try block finishes — whether normally or by exception. Resources must implement AutoCloseable. 
// Old way — manual finally close
BufferedReader br = null;
try {
    br = new BufferedReader(new FileReader("file.txt"));
    System.out.println(br.readLine());
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (br != null) br.close();  // tedious and error-prone
}

// New way — try-with-resources (cleaner!)
try (BufferedReader br = new BufferedReader(new FileReader("file.txt"))) {
    System.out.println(br.readLine());
} catch (IOException e) {
    e.printStackTrace();
}
// br.close() called AUTOMATICALLY — no finally needed!

// Multiple resources
try (Connection conn = getConn();
     Statement stmt = conn.createStatement()) {
    stmt.execute("SELECT * FROM users");
}
// Both stmt and conn closed automatically, in reverse order
💡 AutoCloseable:
Any class implementing AutoCloseable (or Closeable) can be used in try-with-resources. Custom classes can also implement it to get automatic cleanup.

Top Interview Questions & Model Answers
Q1. What is the difference between checked and unchecked exceptions?
Checked exceptions are checked at compile time — the compiler forces you to either handle them with try-catch or declare them with throws. They extend Exception (but not RuntimeException). Examples: IOException, SQLException. Unchecked exceptions extend RuntimeException — the compiler doesn't force handling. They usually indicate programming bugs. Examples: NullPointerException, ArrayIndexOutOfBoundsException.
Q2. What is the difference between throw and throws?
throw is used inside a method body to actually throw an exception instance: throw new IOException("msg"). throws is used in a method signature to declare that the method might throw certain checked exceptions: void readFile() throws IOException. throw actually creates and throws the exception; throws is just a declaration warning the caller.
Q3. Does finally always execute? When does it NOT run?
finally almost always executes — whether exception occurs or not, whether it is caught or not. The only case where finally does NOT run is when System.exit() is called inside try or catch (terminates the JVM), or if the JVM itself crashes (power failure, etc.).
Q4. What is the difference between Exception and Error?
Both extend Throwable. Exception represents conditions that a reasonable application might want to catch and recover from — like IOException, NullPointerException. Error represents serious problems that the JVM itself cannot recover from — like OutOfMemoryError, StackOverflowError. Errors are not meant to be caught by application code. Exceptions should be caught and handled.
Q5. What is try-with-resources? Why is it better than finally?
try-with-resources (Java 7) automatically closes resources declared in the try parentheses when the block exits — no explicit finally needed. Resources must implement AutoCloseable. It's better than finally because: it's cleaner (less code), it handles suppressed exceptions properly (if both try and close throw exceptions, the close exception is "suppressed" not lost), and it's less error-prone (no chance to forget closing).
Q6. How do you create a custom exception?
Extend Exception for a checked custom exception, or RuntimeException for an unchecked one. Provide constructors that call super(message) to pass the message to the parent. Optionally add extra fields for additional error details. Use your custom exception when built-in exceptions don't clearly express the business error — for example, InsufficientFundsException is clearer than ArithmeticException for a banking context.
Q7. What happens if an exception is thrown in a catch block?
If an exception is thrown inside a catch block, the finally block still runs (if present), and then the new exception propagates up the call stack. The original exception is replaced by the new one unless you use exception chaining — throw new NewException("msg", originalException) — which preserves the original cause and is best practice for wrapping exceptions.
