Section 01

Why Algorithms & Performance Matter

Before we dive into complexity, understand why every FAANG interviewer asks about it and why it costs companies millions.

🌐 Google Search

Google processes 8.5 billion searches/day. If their search was O(n²) instead of O(n log n), the servers would melt. The difference between algorithms is worth billions in hardware savings.

🛍️ Amazon

Amazon has 300 million products. O(n) search = 300M checks. O(log n) = just 28 comparisons. Same result — astronomically different speed.

🎬 Netflix

Netflix recommends movies to 238 million users. A naive O(n²) recommendation is impossible at scale. Efficient algorithms make real-time personalization feasible.

💳 Visa Payments

Visa processes 65,000 transactions/second. Fraud detection must complete in <100ms. Only highly optimized algorithms survive these constraints.

The Phone Book Analogy

📞
Finding a name in 1,000,000 entries.

Linear Search O(n): Start from page 1, check every page. Worst case: 1,000,000 checks.
Binary Search O(log n): Open to middle. Is the name earlier? Discard second half. Repeat. Worst case: just 20 checks.

Same problem. 50,000× faster. This is why complexity matters.

Section 02

Computer Execution Fundamentals

To truly understand complexity, you must understand how computers actually execute your code at the hardware level.

How the CPU Executes Your Java Program

Every Java line → bytecode → JIT compiled to machine instructions. Each instruction costs nanoseconds. Total instruction count = "cost" of the algorithm.

Fetch
CPU reads next instruction from memory. L1 cache hit ≈ 1ns. RAM access ≈ 100ns. Cache miss = massive penalty.
Decode
CPU decodes the binary instruction — what operation to perform: add, compare, load, store, jump...
Execute
ALU performs the operation. Integer add ≈ 1 cycle. Memory read ≈ 100–300 cycles. Division ≈ 20–90 cycles.
Write Back
Result stored in register or memory. Loop back — fetch next instruction. This cycle repeats billions of times per second.

Memory Hierarchy & Access Cost

Memory LevelSizeLatencyImpact on Complexity
CPU Register~1 KB<1 nsLoop counters, arithmetic — O(1)
L1 Cache32–64 KB~1 nsSequential array access — very fast
L2 Cache256 KB–1 MB~5 nsModerate working sets
L3 Cache4–32 MB~20 nsLarge arrays fit here
RAM (Heap)GBs~100 nsRandom access, LinkedList traversal
SSD / DiskTBs~100 µsDatabase I/O — avoid in tight loops
💡 Key Insight: Arrays are faster than LinkedLists for sequential access because arrays are contiguous in memory — the CPU prefetches the next elements into cache automatically. LinkedList nodes are scattered in heap, causing constant cache misses (100× slower).

What "One Operation" Actually Means

java
// These are all O(1) — single CPU operations:
int x = 5;               // 1 store
int y = x + 3;           // 1 load + 1 add + 1 store
int z = arr[i];          // 1 index calc + 1 load
boolean b = x > y;       // 1 compare

// These LOOK like one line but are NOT O(1):
Arrays.sort(arr);         // O(n log n) — do not be fooled!
str1.equals(str2);        // O(n) — compares char by char
new ArrayList<>(list);   // O(n) — copies all elements
str.substring(0, n);     // O(n) in Java — creates new String
Collections.sort(list);   // O(n log n) — TimSort
Section 03

What Is Time Complexity?

Time complexity is NOT how long your code takes in seconds. It's a mathematical description of how runtime grows as input size grows.

The Core Question

If I double my input size, how much more time do I need?

O(1)Input doubles → time same. Array[5] takes same time on 10 or 10M element array.
O(n)Input doubles → time doubles. Must inspect every element once.
O(n²)Input doubles → time quadruples. n=100: 10K ops. n=200: 40K ops.

Why Not Measure in Seconds?

⚡ Hardware varies

Your laptop runs the same code 5× faster than a budget server. Seconds are meaningless without knowing hardware.

🔀 Other processes interfere

OS scheduling, GC pauses, network I/O — all affect wall-clock time unpredictably.

📐 Math is universal

O(n log n) is always better than O(n²) regardless of language, hardware, or decade.

Big-O, Big-Ω, Big-Θ

NotationMeaningCommon Use
O (Big-O)Upper bound — worst caseInterviews, everyday analysis. Default choice.
Ω (Big-Omega)Lower bound — best caseTheoretical minimum guarantees.
Θ (Theta)Tight bound — exact rateWhen best and worst case match. Precise analysis.
Interview Reality: When asked "what's the time complexity?", always answer with Big-O worst case unless explicitly told otherwise.

Every Big-O Complexity — Deep Dive

Mathematical intuition, real-world analogies, Java examples, and CPU-level understanding for every complexity class.

O(1) — Constant Time

Regardless of input size, always the same number of steps. The holy grail of algorithm design.

📖
Analogy: Looking up a word when you know the exact page number. Jump to page 247 — doesn't matter if the book has 100 or 1,000,000 pages.
java — O(1) examples
// All O(1):
int val = arr[5];          // array index
map.get("key");            // HashMap lookup
map.put("key", val);       // HashMap insert
stack.push(item);           // Stack push
stack.pop();                // Stack pop
list.get(0);               // ArrayList by index
list.size();                // size field
int mid = (l + r) / 2;    // arithmetic

Why it's O(1)

Array index: CPU computes base + index × size — always exactly 2 ops. HashMap: hash(key) → bucket index → direct memory jump. Stack push/pop: pointer increment + write.

O(log n) — Logarithmic Time

Each step eliminates half the remaining input. Even for n = 1 billion, only ~30 steps needed.

📚
Analogy: Binary search in a sorted phone book. Open to middle — name in first half? Discard second half. 1,000,000 names → ~20 comparisons.
java — O(log n)
int binarySearch(int[] arr, int target) {
    int lo=0, hi=arr.length-1;
    while(lo<=hi) {
        int mid = lo+(hi-lo)/2;
        if(arr[mid]==target) return mid;
        if(arr[mid]<target) lo=mid+1;
        else hi=mid-1;
    }
    return -1;
}
// Also O(log n):
treeMap.get(key);   // BST traversal
treeSet.add(x);     // balanced tree insert

The Math

Halving n until n/2^k = 1 gives k = log₂(n) steps.

O(n) — Linear Time

Visits each element once. Double the input → double the time. Often the best possible when you must inspect all data.

🔍
Analogy: Reading a book — you must read every page. A 200-page book takes twice as long as a 100-page book.
java — O(n)
int findMax(int[] arr) {
    int max = arr[0];
    for(int num : arr)       // n iterations
        if(num > max) max=num; // O(1) work each
    return max;
}
// Also O(n): str.contains(), Arrays.fill(), LinkedList traversal
// Collections.frequency(), new ArrayList(list)
O(n log n) — Linearithmic Time

The "price" of comparison-based sorting. The sweet spot for most interview solutions.

🗂️
Analogy: Sorting a deck of cards by repeatedly splitting into halves, sorting each, then merging. Each card passes through O(log n) merge rounds.
java — O(n log n)
Arrays.sort(arr);          // Java's TimSort — O(n log n)
Collections.sort(list);    // O(n log n)

// Merge sort recurrence: T(n) = 2T(n/2) + O(n)
// log n levels × O(n) merge each = O(n log n)

// Adding n elements to TreeSet:
for(int x : arr) treeSet.add(x); // n × O(log n) = O(n log n)
O(n²) — Quadratic Time

Two nested loops of n. Acceptable for n ≤ 10,000 but dangerously slow beyond that. Classic brute force signature.

🤝
Analogy: Everyone at a party shaking hands with everyone else. 10 people = 45 handshakes. 1,000 people = 499,500 handshakes.
java — O(n²)
// Bubble sort — classic O(n²)
for(int i=0; i<n; i++)          // n
    for(int j=0; j<n-i-1; j++)  // ~n
        if(arr[j]>arr[j+1]) swap(j,j+1);

// All pairs — O(n²)
for(int i=0; i<n; i++)
    for(int j=i+1; j<n; j++)
        process(arr[i], arr[j]);
O(2ⁿ) — Exponential Time

Each extra element doubles the work. Only feasible for n ≤ 20–25.

🌳
Analogy: All subsets of a set. 3 items → 8 subsets. 20 items → 1,048,576 subsets. 30 items → 1 billion subsets.
java — O(2ⁿ)
// Fibonacci WITHOUT memo — O(2^n)
int fib(int n) {
    if(n<=1) return n;
    return fib(n-1) + fib(n-2); // 2 calls each time
}
// fib(40) = ~1 trillion calls!
O(n!) — Factorial Time

The worst practical complexity. 10! = 3.6 million. 15! = 1.3 trillion. Only feasible for n ≤ 10–12.

🎲
Analogy: Arrange n people in every possible order. 3 people = 6 orders. 10 people = 3,628,800 orders.
java — O(n!)
// Generate all permutations — O(n!)
void permute(int[] arr, int start) {
    if(start==arr.length){ results.add(arr.clone()); return; }
    for(int i=start; i<arr.length; i++) {
        swap(arr, start, i);
        permute(arr, start+1); // n × (n-1) × (n-2)... = n!
        swap(arr, start, i);
    }
}
Section 05

How To Calculate Time Complexity

A systematic method you can apply to any code. Master these five rules and complexity analysis becomes mechanical.

The 5 Golden Rules

Drop Constants
O(3n) = O(n). O(100) = O(1). Constants reflect hardware speed, not algorithmic growth rate.
Drop Lower-Order Terms
O(n² + n) = O(n²). At large n, lower terms become negligible. Keep only the dominant term.
Sequential Code — Add
O(n) then O(n) = O(n) + O(n) = O(2n) = O(n). Sequential blocks do NOT multiply.
Nested Code — Multiply
O(n) INSIDE O(n) = O(n) × O(n) = O(n²). Nesting always multiplies.
Take Dominant Term
After combining, keep the fastest-growing term. O(n² + n log n + n + 1) = O(n²).

Worked Examples

java — Two Sequential Loops
void twoLoops(int[] arr) {
    for(int x : arr) print(x);      // O(n)
    for(int x : arr) print(x*2);   // O(n)
}
// Total: O(n) + O(n) = O(2n) = O(n)   ← NOT O(n²)!
java — Nested Loops
void nested(int[] arr) {
    for(int i=0; i<arr.length; i++)      // n
        for(int j=0; j<arr.length; j++)   // n
            print(arr[i]+arr[j]);           // O(1)
}
// Total: n × n × O(1) = O(n²)
java — Mixed: Find the Dominant
void mixed(int[] arr) {
    int n = arr.length;
    int x = arr[0];                      // O(1)
    for(int v : arr) print(v);            // O(n)
    for(int i=0; i<n; i++)
        for(int j=0; j<n; j++) print(i+j); // O(n²)
    Arrays.sort(arr);                      // O(n log n)
}
// Sum: O(1) + O(n) + O(n²) + O(n log n) = O(n²)  ← dominant

Loop Analysis Deep Dive

Loops are the primary source of complexity. Master every pattern here.

Pattern 1: Simple Loop — O(n)

java
for(int i=0; i<n; i++) work(); // exactly n iterations → O(n)

Pattern 2: Halved Loop — Still O(n)

java
for(int i=0; i<n/2; i++) work(); // n/2 iterations → O(n/2) = O(n)

Pattern 3: Logarithmic Loop — O(log n)

java
for(int i=1; i<n; i*=2) work(); // i = 1,2,4,8... → log₂(n) iters → O(log n)
for(int i=n; i>1; i/=2) work(); // i = n,n/2,n/4... → O(log n)

Pattern 4: Standard Nested — O(n²)

java
for(int i=0; i<n; i++)
    for(int j=0; j<n; j++) work(); // n × n = O(n²)

Pattern 5: Triangular Loop — O(n²)

java
for(int i=0; i<n; i++)
    for(int j=i; j<n; j++) work(); // j starts at i!
// Iterations: n + (n-1) + ... + 1 = n(n+1)/2 ≈ n²/2 → O(n²)
// Still O(n²) — the 1/2 constant is dropped

Pattern 6: Nested Logarithmic — O(n log n)

java
for(int i=0; i<n; i++)          // n
    for(int j=1; j<n; j*=2) work(); // log n
// n × log n = O(n log n)

Pattern 7: Two Different Variables — O(m×n)

java
for(int i=0; i<m; i++)    // m iterations
    for(int j=0; j<n; j++) work(); // n iterations
// Total: O(m×n) — NOT O(n²) unless m==n!

Section 07

Recursion Analysis

Recursion is where most students stumble. We'll demystify it with tree visualizations and the Master Theorem.

Key Formula: Total work = (branching factor)^(depth) × (work per node). For divide-and-conquer, use the Master Theorem: T(n) = aT(n/b) + f(n)

Example 1: Linear Recursion — O(n)

java
int sum(int n) {
    if(n==0) return 0;
    return n + sum(n-1);
}
// Call chain: sum(5)→sum(4)→...→sum(0)
// Depth: n. Branching: 1. Total calls: n+1 → O(n)

Example 2: Binary Recursion — O(2ⁿ)

java
int fib(int n) {
    if(n<=1) return n;
    return fib(n-1) + fib(n-2);
}
/* fib(4) tree:
           fib(4)
          /      \
       fib(3)   fib(2)
       /    \   /    \
   fib(2) fib(1) fib(1) fib(0)
   /   \
fib(1) fib(0)
Branching=2, Depth≈n → O(2^n). fib(40)= ~trillion calls! */

Example 3: Merge Sort — O(n log n)

java
// T(n) = 2T(n/2) + O(n)
void mergeSort(int[] arr, int l, int r) {
    if(l>=r) return;
    int mid=(l+r)/2;
    mergeSort(arr,l,mid);    // T(n/2)
    mergeSort(arr,mid+1,r); // T(n/2)
    merge(arr,l,mid,r);      // O(n) merge work
}
/* log n levels × O(n) merge each = O(n log n) */

The Master Theorem

T(n) = aT(n/b) + f(n)ConditionResult
Case 1f(n) grows slower than n^log_b(a)T(n) = Θ(n^log_b(a))
Case 2f(n) grows at same rate as n^log_b(a)T(n) = Θ(n^log_b(a) · log n)
Case 3f(n) grows faster than n^log_b(a)T(n) = Θ(f(n))

Example: Merge sort: a=2, b=2, f(n)=n. n^log₂2 = n¹ = n. f(n)=n matches → Case 2 → T(n) = Θ(n log n). ✓

Section 08

Space Complexity

Space complexity measures the extra memory your algorithm uses. Often a trade-off against time.

📚 Stack Memory

  • Stores local variables & function call frames
  • Each recursive call adds a stack frame
  • Limited (~1–8 MB typical per thread)
  • StackOverflowError = recursion too deep
  • Recursion depth n → O(n) stack space

🗄️ Heap Memory

  • Stores all objects and arrays
  • Managed by JVM Garbage Collector
  • Much larger (configured via -Xmx)
  • new int[n] = O(n) heap space
  • HashMap with n entries = O(n)
java — Space Complexity Examples
// O(1) Space — only constant extra memory
int sum(int[] arr) {
    int total=0;           // one integer, always
    for(int x:arr) total+=x;
    return total;
}

// O(n) Space — heap allocation proportional to input
int[] doubled(int[] arr) {
    int[] res = new int[arr.length]; // n extra slots
    for(int i=0;i<arr.length;i++) res[i]=arr[i]*2;
    return res;
}

// O(n) Stack Space — n recursive call frames
int factorial(int n) {
    if(n==0) return 1;
    return n * factorial(n-1); // n stack frames!
}

// O(log n) Stack — binary search recursion depth
// O(n²) Heap — 2D DP table: new int[n][n]
In-place vs Auxiliary: In-place algorithms (quicksort, two-pointer) use O(1) extra space. Auxiliary-space algorithms (merge sort) use O(n) extra. FAANG interviews often ask for in-place solutions when possible.

Section 09

How To Identify Complexity In ANY Program

A complete, systematic methodology. Apply this to every piece of code.

Find all loops — note their iteration count
For every for/while: how many times does it run in terms of n? Simple: n. Halving: log n. Doubling increment: log n.
Check for nesting
Inner loop inside outer loop? Multiply. Is inner loop dependent on the outer variable? (triangular — still O(n²))
Identify recursive calls
How many calls per invocation (branching factor)? How deep (depth)? Apply branching^depth or Master Theorem.
Audit library/built-in calls
Arrays.sort()=O(n log n). str.equals()=O(n). map.get()=O(1). str.substring()=O(n). Never assume library calls are O(1)!
Add sequential, multiply nested
Combine all parts. Drop constants and lower-order terms. State the dominant term.

Section 13

Searching Algorithms

java — Linear Search O(n)
int linearSearch(int[] arr, int target) {
    for(int i=0; i<arr.length; i++) {
        if(arr[i]==target) return i; // Best: O(1)
    }
    return -1; // Worst: O(n). Works on UNSORTED arrays.
}
java — Binary Search O(log n)
int binarySearch(int[] arr, int target) {
    int lo=0, hi=arr.length-1;
    while(lo<=hi) {
        int mid = lo+(hi-lo)/2; // avoids integer overflow
        if(arr[mid]==target) return mid;
        else if(arr[mid]<target) lo=mid+1;
        else hi=mid-1;
    }
    return -1;
}
// REQUIRES sorted array. n=1,000,000 → max 20 iterations
Pro Tip — Binary Search on the Answer: Binary search applies to ANY monotonic condition, not just sorted arrays. In competitive programming: if you can check whether answer ≥ X in O(f(n)), you can find the optimal answer in O(f(n) × log(answer_range)).
Section 14

FAANG Interview Analysis

✅ What interviewers want

  • Proactively state time AND space complexity
  • Identify the bottleneck in your solution
  • Compare to brute force unprompted
  • Recognize when O(n²) is too slow
  • Know all DS complexities without pausing
  • Suggest optimizations before being asked

❌ Red flags they notice

  • Not mentioning complexity at all
  • Confusing O(n+m) with O(n×m)
  • Forgetting recursion stack space
  • Thinking Arrays.sort() is O(n)
  • Not knowing HashMap is O(1) amortized
  • Unable to explain why code is slow

The FAANG Script — Say This After Writing Code

"The time complexity is O(n log n) because I sort the array — O(n log n) — then do a single linear scan — O(n). Together: O(n log n) + O(n) = O(n log n). Space is O(1) auxiliary since I sort in-place and use only a few integers. This handles n up to 10⁶ comfortably."

Brute Force → Optimized: Two Sum

java — O(n²) → O(n)
// BRUTE FORCE: O(n²) — check every pair
int[] twoSumBrute(int[] nums, int target) {
    for(int i=0;i<nums.length;i++)
        for(int j=i+1;j<nums.length;j++)
            if(nums[i]+nums[j]==target)
                return new int[]{i,j};
    return null;
}  // O(n²) time, O(1) space

// OPTIMAL: O(n) — HashMap stores complements
int[] twoSumOptimal(int[] nums, int target) {
    Map<Integer,Integer> map = new HashMap<>();
    for(int i=0;i<nums.length;i++) {
        int comp = target-nums[i];
        if(map.containsKey(comp))
            return new int[]{map.get(comp), i}; // O(1)
        map.put(nums[i], i);                       // O(1)
    }
    return null;
}  // O(n) time, O(n) space — classic time-space tradeoff

⚡ The 10⁸ Rule

CPUs execute ~10⁸–10⁹ simple ops/second. Most CP problems: 1–2 second limits. Budget: ~2×10⁸ ops. Simple ops = array access, compare, arithmetic. Division and modulo count ~3–5×. Float slower than int.

Program 1: Find Duplicates

java
List<Integer> findDuplicates(int[] nums) {
    Set<Integer> seen = new HashSet<>();     // O(1)
    Set<Integer> dups = new HashSet<>();     // O(1)
    for(int num : nums)                         // n iterations
        if(!seen.add(num)) dups.add(num);       // O(1) avg each
    return new ArrayList<>(dups);             // O(d) ≤ O(n)
}
// TIME: O(n) — one pass, O(1) per step
// SPACE: O(n) — HashSets can hold up to n elements

Program 2: Group Anagrams

java
List<List<String>> groupAnagrams(String[] strs) {
    Map<String,List<String>> map = new HashMap<>();
    for(String s : strs) {                     // n strings
        char[] c = s.toCharArray();             // O(k)
        Arrays.sort(c);                         // O(k log k)
        String key = new String(c);           // O(k)
        map.computeIfAbsent(key, x -> new ArrayList<>()).add(s);
    }
    return new ArrayList<>(map.values());
}
// TIME: O(n × k log k) — n strings, each sorted in O(k log k)
// SPACE: O(n × k) — storing all characters in the map

Program 3: DFS Graph

java
void dfs(int node, boolean[] vis, List<List<Integer>> adj) {
    vis[node] = true;
    for(int nb : adj.get(node))
        if(!vis[nb]) dfs(nb, vis, adj);
}
// Each node visited once (V calls), each edge processed once (E total)
// TIME: O(V + E).  SPACE: O(V) for visited array + O(V) call stack

Program 4: LCS Dynamic Programming

java
int lcs(String s1, String s2) {
    int m=s1.length(), n=s2.length();
    int[][] dp = new int[m+1][n+1];            // O(m×n) space
    for(int i=1;i<=m;i++)                      // m iterations
        for(int j=1;j<=n;j++) {               // n iterations
            if(s1.charAt(i-1)==s2.charAt(j-1))
                dp[i][j]=dp[i-1][j-1]+1;
            else
                dp[i][j]=Math.max(dp[i-1][j],dp[i][j-1]);
        }
    return dp[m][n];
}
// TIME: O(m×n).  SPACE: O(m×n) — can optimize to O(min(m,n)) with rolling array
Section 18

Advanced Topics

Amortized Complexity

When occasional expensive operations are "amortized" across many cheap ones, the average cost can be lower than worst case.

ArrayList.add() — Amortized O(1)

When full, ArrayList copies all elements (O(n)), but doubles its capacity. Copy costs: 1+2+4+...+n/2+n ≈ 2n. Across n add() calls: 2n total copies / n calls = O(2) per call = amortized O(1).

Average vs Worst Case

AlgorithmAverageWorstWhen worst occurs
QuickSortO(n log n)O(n²)Pivot always min/max (sorted input)
HashMap getO(1)O(n)All keys hash to same bucket

Cache Locality

Why arrays beat LinkedLists in practice: Array elements are contiguous — CPU loads 64-byte cache lines. Accessing arr[0] loads arr[0..15] into L1 cache for free. LinkedList nodes are scattered — every next-pointer is a cache miss (~100× slower than L1 hit). This is why ArrayList almost always outperforms LinkedList despite the same Big-O.

JVM JIT Optimization

After ~10,000 executions, Java's JIT kicks in and can: eliminate array bounds checks in hot loops, inline small methods (removes call overhead), perform loop unrolling. This means O(n) Java code can approach C speed in production. In benchmarks: always warm up the JVM first.

Optimization Techniques

Memoization — O(2ⁿ) → O(n)

java — Fibonacci with memo
Map<Integer,Long> memo = new HashMap<>();
long fib(int n) {
    if(n<=1) return n;
    if(memo.containsKey(n)) return memo.get(n); // O(1) cache hit
    long res = fib(n-1) + fib(n-2);
    memo.put(n, res);
    return res;
}
// Each value computed once → O(n) time, O(n) space

Prefix Sums — O(n) query → O(1)

java
// Build prefix array once — O(n)
int[] prefix = new int[n+1];
for(int i=0;i<n;i++) prefix[i+1] = prefix[i]+arr[i];

// Any range sum query — O(1)!
int rangeSum = prefix[r+1] - prefix[l];
// For q queries: O(n) build + O(1)×q = O(n+q) vs O(n×q)

Two Pointers — O(n²) → O(n)

java — Two Sum sorted array
boolean twoSumSorted(int[] arr, int target) {
    int lo=0, hi=arr.length-1;
    while(lo<hi) {
        int sum = arr[lo]+arr[hi];
        if(sum==target) return true;
        else if(sum<target) lo++;
        else hi--;
    }
    return false;
} // O(n) — each pointer moves at most n times total

Sliding Window — O(n²) → O(n)

java — Max sum subarray length k
int maxSumWindow(int[] arr, int k) {
    int windowSum=0, maxSum=0;
    for(int i=0;i<k;i++) windowSum+=arr[i]; // initial window
    maxSum=windowSum;
    for(int i=k;i<arr.length;i++) {
        windowSum += arr[i]-arr[i-k];           // slide: +right, -left
        maxSum = Math.max(maxSum, windowSum);
    }
    return maxSum;
} // O(n) — single pass, never recomputing the full window sum
