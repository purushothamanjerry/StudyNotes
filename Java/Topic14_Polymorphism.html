What is Polymorphism?
Definition: Polymorphism means "many forms" (poly = many, morph = form). In Java, it means one entity (method or object) can behave differently in different situations. The same method name can produce different results depending on the object calling it.

Real World Analogy

🎵 Same Action, Different Result

When you press the "play" button on a music player, it plays music. On a video player, it plays video. Same button (same method name), different behavior based on context (object type).

🐾 Animal Sound

Call makeSound() on a Dog → "Woof!". Call it on a Cat → "Meow!". Call it on a Cow → "Moo!". Same method name, different behavior based on which animal object is used.

Two Types — Compile-time vs Runtime

⚡ Compile-time Polymorphism

Also called: Static polymorphism, Early binding

Achieved by: Method Overloading

Resolved: At compile time by compiler

How: Same name, different parameters in same class

Speed: Faster (resolved at compile time)

🔄 Runtime Polymorphism

Also called: Dynamic polymorphism, Late binding

Achieved by: Method Overriding

Resolved: At runtime by JVM

How: Parent reference → child object, overridden method

Speed: Slightly slower (resolved at runtime)

3
Compile-time Polymorphism (Method Overloading)

The compiler determines which method to call at compile time based on the method signature (name + parameters).

public class Calculator {

    // Same name "add" — 3 different forms
    public int add(int a, int b) {
        System.out.println("int version");
        return a + b;
    }

    public double add(double a, double b) {
        System.out.println("double version");
        return a + b;
    }

    public int add(int a, int b, int c) {
        System.out.println("3 params version");
        return a + b + c;
    }
}

Calculator calc = new Calculator();

// Compiler decides which add() to call at COMPILE TIME
calc.add(2, 3);         // → int version     (5)
calc.add(2.0, 3.0);    // → double version  (5.0)
calc.add(1, 2, 3);     // → 3 params version (6)
How compiler resolves overloaded methods:
1. Exact match → use it
2. Widening (int → long → float → double) → use widened version
3. Autoboxing (int → Integer) → use boxed version
4. Varargs → use varargs version
If still ambiguous → compile error!

Runtime Polymorphism (Method Overriding)
Key concept: A parent class reference can hold a child class object. When you call an overridden method, the JVM decides at runtime which version to call based on the actual object type, not the reference type. 
class Animal {
    public void sound() {
        System.out.println("Some animal sound");
    }
    public void eat() {
        System.out.println("Animal eating");
    }
}

class Dog extends Animal {
    @Override
    public void sound() { System.out.println("Woof!"); }

    public void fetch() { System.out.println("Fetching..."); }
}

class Cat extends Animal {
    @Override
    public void sound() { System.out.println("Meow!"); }
}

// Parent reference holding child objects — UPCASTING
Animal a1 = new Dog();  // Animal ref → Dog object
Animal a2 = new Cat();  // Animal ref → Cat object
Animal a3 = new Animal(); // Animal ref → Animal object

// JVM decides at RUNTIME which sound() to call
a1.sound();  // Woof!  ← Dog's version (actual object is Dog)
a2.sound();  // Meow!  ← Cat's version (actual object is Cat)
a3.sound();  // Some animal sound ← Animal's version

a1.eat();    // Animal eating ← inherited, not overridden

// Cannot call child-specific methods via parent reference!
// a1.fetch(); ❌ compile error — Animal ref doesn't know fetch()
⚠️ IMPORTANT rule:
Through a parent reference, you can ONLY call methods that are defined in the parent class. Even though a1 actually holds a Dog, you cannot call a1.fetch() because the Animal class doesn't have fetch(). To call Dog-specific methods, you must downcast.
5
Upcasting — Parent Reference, Child Object
Upcasting = assigning a child object to a parent reference.
It is automatic (implicit) — no cast operator needed. Always safe. 
// Implicit upcasting — automatic, always safe
Animal a = new Dog();   // ✅ Dog IS-A Animal — auto upcast

// Explicit upcasting also works (unnecessary but valid)
Animal a2 = (Animal) new Dog();  // same thing

// Real use case — polymorphic method parameter
public static void makeSound(Animal a) {
    a.sound();  // calls the right version at runtime!
}

makeSound(new Dog());    // passes Dog as Animal → Woof!
makeSound(new Cat());    // passes Cat as Animal → Meow!
makeSound(new Animal()); // passes Animal → Some sound

// Upcasting in arrays
Animal[] animals = { new Dog(), new Cat(), new Animal() };
for (Animal a : animals) {
    a.sound();  // Woof!, Meow!, Some animal sound
}
💡 Why upcasting is useful:
Write ONE method that works with ALL subtypes. No need for separate methods for Dog, Cat, etc. This is the power of polymorphism — write general code that works with specific types.
6
Downcasting — Getting Child Back
Downcasting = converting a parent reference back to a child type.
It is explicit — must use cast operator. Can fail at runtime if object is not actually that type! 
Animal a = new Dog();   // upcast — Dog stored as Animal

// Need to call Dog-specific method fetch()
// a.fetch(); ❌ compile error

// Downcast — explicitly tell compiler it's a Dog
Dog d = (Dog) a;          // ✅ works — actual object IS a Dog
d.fetch();                   // ✅ now can call Dog methods

// DANGEROUS — downcasting wrong type → ClassCastException!
Animal a2 = new Cat();
Dog d2 = (Dog) a2;        // ❌ ClassCastException at runtime!
                            // actual object is Cat, not Dog

// SAFE downcasting — always check with instanceof first!
Animal a3 = new Dog();
if (a3 instanceof Dog) {
    Dog d3 = (Dog) a3;      // ✅ safe — checked first
    d3.fetch();
}

// Java 16+ — pattern matching instanceof (cleaner!)
if (a3 instanceof Dog dog) {
    dog.fetch();             // ✅ no explicit cast needed
}
⚠️ Always use instanceof before downcasting!
Downcasting without instanceof check → risk of ClassCastException at runtime. Make it a habit: check first, then cast.
7
Dynamic Method Dispatch
Dynamic Method Dispatch is the mechanism by which Java resolves which overridden method to call at runtime. It is the backbone of runtime polymorphism. The JVM looks at the actual object type (not the reference type) to determine which method to execute. 
class Shape {
    public void draw() { System.out.println("Drawing shape"); }
    public void area() { System.out.println("Shape area"); }
}

class Circle extends Shape {
    @Override
    public void draw() { System.out.println("Drawing circle ○"); }
    @Override
    public void area() { System.out.println("π × r²"); }
}

class Rectangle extends Shape {
    @Override
    public void draw() { System.out.println("Drawing rectangle □"); }
    @Override
    public void area() { System.out.println("l × w"); }
}

// JVM DISPATCH TABLE — decides at runtime
Shape s;   // reference type: Shape

s = new Circle();      // actual type: Circle
s.draw();              // Drawing circle ○  ← Circle's version

s = new Rectangle(); // actual type: Rectangle
s.draw();              // Drawing rectangle □  ← Rectangle's version

s = new Shape();      // actual type: Shape
s.draw();              // Drawing shape  ← Shape's version
How JVM decides:
JVM maintains a virtual method table (vtable) for each class. When an overridden method is called on a parent reference, JVM looks up the actual object's vtable to find the correct method. This lookup happens at runtime — hence "runtime polymorphism".

Practical Use — Polymorphic Arrays & Methods
// REAL USE CASE 1 — Polymorphic array
Animal[] zoo = {
    new Dog(),
    new Cat(),
    new Dog(),
    new Cat()
};

// One loop, correct behavior for each animal
for (Animal a : zoo) {
    a.sound();   // Woof! Meow! Woof! Meow!
}

// REAL USE CASE 2 — Polymorphic method parameter
public static double calculateArea(Shape shape) {
    // Works for ANY shape — Circle, Rectangle, Triangle...
    shape.area();
    return 0; // simplified
}
calculateArea(new Circle());     // ✅ π × r²
calculateArea(new Rectangle()); // ✅ l × w

// REAL USE CASE 3 — Polymorphic return type
public static Animal createAnimal(String type) {
    if (type.equals("dog")) return new Dog();
    if (type.equals("cat")) return new Cat();
    return new Animal();
}
// Factory pattern — returns correct subtype as Animal reference

Top Interview Questions & Model Answers
Q1. What is polymorphism? What are its two types?
Polymorphism means "many forms" — one entity behaving differently in different situations. Two types: (1) Compile-time (Static) — achieved through method overloading, resolved by the compiler based on method signature. (2) Runtime (Dynamic) — achieved through method overriding, resolved by JVM at runtime based on actual object type. Runtime polymorphism is the more powerful and widely used form.
Q2. What is dynamic method dispatch?
Dynamic method dispatch is the mechanism by which Java resolves which overridden method to call at runtime. When an overridden method is called through a parent class reference, the JVM looks at the actual type of the object at runtime (not the reference type) to determine which implementation to execute. This is what makes runtime polymorphism work — the correct method is selected dynamically.
Q3. What is upcasting and downcasting?
Upcasting is assigning a child object to a parent reference — it is automatic (implicit) and always safe because a child IS-A parent. Downcasting is explicitly converting a parent reference back to a child type using a cast operator. Downcasting can throw ClassCastException at runtime if the object is not actually the target type, so always use instanceof before downcasting.
Q4. What is the output? 
Animal a = new Dog();
a.sound();
Woof! — Even though the reference type is Animal, the actual object is a Dog. At runtime, JVM's dynamic method dispatch calls Dog's overridden sound() method. This is runtime polymorphism in action.
Q5. Can you achieve polymorphism without inheritance?
Compile-time polymorphism (overloading) can be achieved without inheritance — just overload methods in the same class. Runtime polymorphism requires inheritance or interface implementation since it is based on method overriding across parent-child classes. So partial yes — overloading without inheritance, overriding needs inheritance.
Q6. What is the difference between static and dynamic binding?
Static binding (early binding) — method call is resolved at compile time. Used for static methods, private methods, final methods, and overloaded methods. The compiler knows exactly which method to call. Dynamic binding (late binding) — method call is resolved at runtime based on actual object type. Used for overridden instance methods. Allows polymorphic behavior.
Q7. Why can't you call child-specific methods through a parent reference?
Because the compiler only knows about the reference type (Animal), not the actual object type (Dog). The compiler checks the reference type's class definition for method availability at compile time. If Animal doesn't have fetch(), the compiler rejects it — it can't know at compile time that the reference will hold a Dog at runtime. To access child-specific methods, you must downcast to the child type first.
