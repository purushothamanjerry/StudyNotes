Abstract Class

🔑 Definition

An abstract class is a class declared with the abstract keyword. It cannot be instantiated directly — you must subclass it. It provides a partial blueprint for subclasses.

Key Idea: Abstract class = "I am a base class that partially implements behavior. My subclasses must fill in the rest."

📌 Characteristics

  • Declared with abstract keyword
  • Can have abstract methods (no body) and concrete methods (with body)
  • Can have constructors (called via super() from subclass)
  • Can have instance variables (state) with any access modifier
  • Can have static methods and static variables
  • A class can extend only one abstract class (single inheritance)
  • If a subclass doesn't implement all abstract methods → subclass must also be abstract

💻 Syntax & Example

abstract class Animal {

    // Instance variable (state)
    private String name;
    private int age;

    // Constructor
    public Animal(String name, int age) {
        this.name = name;
        this.age = age;
    }

    // Abstract method — subclass MUST implement
    public abstract void makeSound();

    // Concrete method — shared behavior
    public void eat() {
        System.out.println(name + " is eating");
    }

    // Static method
    public static void breathe() {
        System.out.println("All animals breathe");
    }

    // Getter
    public String getName() { return name; }
}

// Concrete subclass
class Dog extends Animal {

    public Dog(String name, int age) {
        super(name, age); // calls Animal constructor
    }

    @Override
    public void makeSound() {
        System.out.println("Woof!");
    }
}

// Animal a = new Animal(); ❌ Cannot instantiate
Animal dog = new Dog("Bruno", 3); // ✅
dog.makeSound(); // Woof!
dog.eat();       // Bruno is eating

📐 Abstract Method Rules

  • Declared with abstract keyword, no body — ends with ;
  • Can only exist inside an abstract class
  • Cannot be private, static, or final
  • Must be overridden in the first non-abstract subclass
⚠️ Common Mistake: You can have an abstract class with zero abstract methods — just to prevent direct instantiation. This is valid!

Interface

🔑 Definition

An interface defines a contract — a set of method signatures that implementing classes must fulfill. It represents the capability or behavior a class promises to provide.

Key Idea: Interface = "Any class that implements me MUST provide these behaviors. I don't care how — just do it."

📌 Characteristics (Before Java 8)

  • All methods were implicitly public abstract
  • All variables were implicitly public static final (constants)
  • No constructors allowed
  • No instance variables
  • A class can implement multiple interfaces
  • An interface can extend multiple interfaces

💻 Basic Syntax

interface Flyable {

    // Constant — public static final (implicitly)
    int MAX_ALTITUDE = 10000; // same as: public static final int MAX_ALTITUDE = 10000;

    // Abstract method — public abstract (implicitly)
    void fly();         // same as: public abstract void fly();
    void land();
}

interface Swimmable {
    void swim();
}

// Implementing MULTIPLE interfaces
class Duck implements Flyable, Swimmable {

    @Override
    public void fly()  { System.out.println("Duck flying"); }

    @Override
    public void land() { System.out.println("Duck landing"); }

    @Override
    public void swim() { System.out.println("Duck swimming"); }
}

🚀 Java 8+ Interface Features

1. Default Methods

Provide a default implementation in the interface itself. Implementing classes can use it or override it.

interface Vehicle {
    void start();

    // Default method — has body
    default void stop() {
        System.out.println("Vehicle stopped");
    }
}

class Car implements Vehicle {
    public void start() { System.out.println("Car started"); }
    // stop() inherited from interface — no need to override
}

2. Static Methods

Utility methods on the interface. NOT inherited by implementing classes — called via interface name.

interface MathUtils {
    static int square(int n) { return n * n; }
}

// Call via interface name only:
int result = MathUtils.square(5); // ✅
// myObject.square(5); ❌ Not accessible on instance

3. Private Methods (Java 9+)

Helper methods to avoid code duplication inside default methods.

interface Logger {
    default void logInfo(String msg)  { log("INFO", msg);  }
    default void logError(String msg) { log("ERROR", msg); }

    private void log(String level, String msg) {
        System.out.println("[" + level + "] " + msg);
    }
}

4. Functional Interface (Java 8)

An interface with exactly one abstract method. Can be used with lambdas.

@FunctionalInterface
interface Calculator {
    int calculate(int a, int b);
}

// Lambda usage
Calculator add = (a, b) -> a + b;
Calculator mul = (a, b) -> a * b;
System.out.println(add.calculate(3, 4)); // 7
Built-in Functional Interfaces (java.util.function):
Predicate<T> — takes T, returns boolean  |  Function<T,R> — takes T, returns R  |  Consumer<T> — takes T, returns void  |  Supplier<T> — takes nothing, returns T

🔗 Interface Extending Multiple Interfaces

interface A { void methodA(); }
interface B { void methodB(); }

interface C extends A, B {  // ✅ Multiple extension allowed
    void methodC();
}

class MyClass implements C {
    public void methodA() {}
    public void methodB() {}
    public void methodC() {}
}

When to Use Which?

✅ Use Abstract Class when…

  • Classes are closely related (share "is-a" relationship)
  • You want to share common state (instance variables)
  • You want to provide partial implementation
  • Subclasses need access to non-public members
  • You need a constructor to initialize shared fields
  • You want to provide template method pattern

Real Examples: HttpServlet, AbstractList, AbstractMap

✅ Use Interface when…

  • Classes are unrelated but share a behavior
  • You need multiple inheritance of type
  • You want to define a pure contract/API
  • Future implementors are unknown
  • You want to use lambda expressions (functional interface)
  • You want loose coupling in design

Real Examples: Runnable, Comparable, Serializable, List

🧠 Decision Rule (Simple)

Ask yourself:
→ Do the classes share state (fields) and are tightly related? → Abstract Class
→ Do you need multiple inheritance or are classes unrelated? → Interface
→ Modern Java (8+): prefer interfaces unless you need constructors or instance variables.

⚠️ Diamond Problem with Default Methods:
If two interfaces have the same default method, the implementing class must override it to resolve the conflict. 
interface A { default void greet() { System.out.println("Hello from A"); } }
interface B { default void greet() { System.out.println("Hello from B"); } }

class C implements A, B {
    @Override
    public void greet() {
        A.super.greet(); // explicitly call A's version
    }
}

Top Interview Questions & Model Answers
Q1. Can abstract class have a constructor? Why?
✅ Yes. Even though you can't instantiate an abstract class directly, its constructor is called by the subclass via super(). This allows the abstract class to initialize its own fields.
Q2. Can we have an abstract class with no abstract methods?
✅ Yes. This is valid — you'd do it to prevent direct instantiation while still providing a reusable concrete class.
Q3. Why can't we instantiate an interface?
Because interfaces have no constructor and no instance state. They define behavior contracts, not objects.
Q4. Can an abstract class implement an interface?
✅ Yes. An abstract class can implement an interface without implementing all its methods — the concrete subclass must implement the remaining methods.
Q5. What is a marker interface? Give examples.
A marker interface has no methods — it just "marks" a class to signal some property to the JVM or framework. Examples: Serializable, Cloneable, Remote.
Q6. Why did Java 8 add default methods to interfaces?
To allow adding new methods to existing interfaces without breaking all the classes that already implement them (backward compatibility). Used heavily in the Collections API — e.g., List.sort(), Collection.stream().
Q7. Difference between abstract method and default method?
Abstract method has no body — must be overridden. Default method has a body — optional to override. Abstract method forces contract; default method provides a fallback implementation.
Q8. Can interface variables be changed?
❌ No. Interface variables are implicitly public static final — they are constants. You cannot reassign them.

Tricky Edge Cases 🧩

1. Abstract class extending abstract class

abstract class A { abstract void show(); }
abstract class B extends A { } // ✅ B doesn't have to implement show()
class C extends B {
    void show() { System.out.println("C"); } // ✅ C must implement
}

2. Interface reference to abstract class object

interface Shape { double area(); }
abstract class Polygon implements Shape { }
class Square extends Polygon {
    public double area() { return 4.0; }
}

Shape s = new Square(); // ✅ Interface reference → concrete object

3. Static interface method — NOT inherited

interface I { static void hello() { System.out.println("Hi"); } }
class MyClass implements I { }

I.hello();           // ✅ Correct
// MyClass.hello(); ❌ Compile error — not inherited

4. Can you override a default method and make it abstract?

interface A { default void greet() { System.out.println("Hi"); } }

// ✅ Yes — an abstract class can re-declare it as abstract
abstract class B implements A {
    public abstract void greet(); // Forces subclasses to implement
}
💡 Quick Memory Trick:
Abstract Class = "Family template" — shares genes (fields) and some behaviors (concrete methods) with children.
Interface = "Job contract" — anyone can sign it (implement it) regardless of their background (class hierarchy).
