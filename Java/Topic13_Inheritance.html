What is Inheritance?
Definition: Inheritance is the OOP mechanism where a child class (subclass) acquires the properties and behaviors of a parent class (superclass). It promotes code reusability and establishes an IS-A relationship.

Real World Analogy

A child inherits traits from parents — eye color, height. But the child also has their own unique traits.

Similarly, a subclass inherits fields and methods from the parent class, and can also add its own.

Key Terms

  • Parent / Super / Base class — gives properties
  • Child / Sub / Derived class — receives properties
  • extends — keyword used for inheritance
  • IS-A — relationship type (Dog IS-A Animal)

Why Inheritance?

  • Code Reusability — write once in parent, use in all children
  • Method Overriding — customize parent behavior in child
  • Polymorphism — parent reference can hold child object
  • Extensibility — add new features without modifying existing code

Syntax — extends Keyword
// Parent class (Superclass)
class Animal {
    String name;
    int    age;

    public void eat() {
        System.out.println(name + " is eating");
    }

    public void breathe() {
        System.out.println("Breathing...");
    }
}

// Child class (Subclass) — inherits from Animal
class Dog extends Animal {
    String breed;  // Dog's own field

    public void bark() {   // Dog's own method
        System.out.println(name + " says: Woof!");
    }
}

// Dog inherits name, age, eat(), breathe() from Animal
Dog d = new Dog();
d.name  = "Bruno";
d.breed = "Labrador";
d.eat();     // inherited from Animal — Bruno is eating
d.breathe(); // inherited from Animal
d.bark();    // Dog's own — Bruno says: Woof!

Types of Inheritance
✅ Single

One child extends one parent.
A → B

✅ Multilevel

Chain of inheritance.
A → B → C

✅ Hierarchical

Multiple children, one parent.
A → B, A → C

❌ Multiple

One child, multiple parents.
NOT supported with classes!

⚠️ Hybrid

Combination of types.
Only via interfaces in Java

// Single Inheritance — A extends B
class Animal { }
class Dog extends Animal { }  // ✅

// Multilevel Inheritance — A → B → C
class Animal { }
class Dog    extends Animal { }
class Puppy  extends Dog    { }  // ✅ Puppy gets Animal + Dog

// Hierarchical — multiple children, one parent
class Animal { }
class Dog extends Animal { }   // ✅
class Cat extends Animal { }   // ✅

// Multiple Inheritance — NOT allowed with classes
// class Dog extends Animal, Pet { } ❌ compile error!

// Multiple Inheritance — allowed with interfaces
interface Animal { }
interface Pet    { }
class Dog implements Animal, Pet { }  // ✅ OK with interfaces
⚠️ Why no multiple inheritance with classes in Java?
The Diamond Problem — if class C extends both A and B, and both A and B have the same method, which one does C inherit? This ambiguity is why Java doesn't allow multiple class inheritance. Interfaces solve this because they either have no implementation (before Java 8) or the implementing class must override the conflicting method.

Method Overriding
Definition: When a subclass provides its own implementation of a method that is already defined in the parent class. Same method name, same parameters, same return type (or covariant). Resolved at runtime (dynamic polymorphism). 
class Animal {
    public void sound() {
        System.out.println("Some animal sound");
    }

    public void sleep() {
        System.out.println("Sleeping...");
    }
}

class Dog extends Animal {
    @Override
    public void sound() {   // Override — same signature
        System.out.println("Woof!");
    }
    // sleep() NOT overridden — inherited as-is
}

class Cat extends Animal {
    @Override
    public void sound() {
        System.out.println("Meow!");
    }
}

// Runtime polymorphism — parent ref, child object
Animal a1 = new Dog();
Animal a2 = new Cat();
a1.sound();  // Woof!  (Dog's version called at runtime)
a2.sound();  // Meow!  (Cat's version called at runtime)

super Keyword in Inheritance
class Animal {
    String type = "Animal";

    Animal(String type) {
        this.type = type;
        System.out.println("Animal constructor: " + type);
    }

    public void describe() {
        System.out.println("I am an animal");
    }
}

class Dog extends Animal {
    String type = "Dog";  // shadows Animal's type

    // super() — call parent constructor
    Dog(String breed) {
        super("Canine");   // MUST be first line
        System.out.println("Dog constructor: " + breed);
    }

    @Override
    public void describe() {
        super.describe();           // call parent method
        System.out.println("I am also a Dog");
    }

    public void showTypes() {
        System.out.println(super.type);  // Canine (parent's)
        System.out.println(this.type);   // Dog (own)
    }
}

Dog d = new Dog("Labrador");
// Output:
// Animal constructor: Canine
// Dog constructor: Labrador

d.describe();
// Output:
// I am an animal
// I am also a Dog

IS-A vs HAS-A Relationship

IS-A — Inheritance

Use extends or implements

"Dog IS-A Animal" → Dog extends Animal

"Car IS-A Vehicle" → Car extends Vehicle

Child IS a type of Parent

HAS-A — Composition

One class has a reference to another

"Car HAS-A Engine" → Car contains Engine object

"Student HAS-A Address" → Student has Address field

One class uses another class

// IS-A — inheritance
class Animal { void eat() {} }
class Dog extends Animal { }  // Dog IS-A Animal

// HAS-A — composition
class Engine {
    void start() { System.out.println("Engine started"); }
}

class Car {
    private Engine engine = new Engine();  // Car HAS-A Engine

    void drive() {
        engine.start();  // uses Engine
        System.out.println("Driving...");
    }
}

// Test IS-A with instanceof
Dog d = new Dog();
System.out.println(d instanceof Dog);     // true
System.out.println(d instanceof Animal);  // true — IS-A!
💡 When to use which:
IS-A (Inheritance) — when child truly IS a type of parent. Dog IS-A Animal ✅. Car IS-A Engine ❌ (wrong!)
HAS-A (Composition) — when one object uses another. Car HAS-A Engine ✅. Prefer composition over inheritance when in doubt — it's more flexible!

// final class — cannot be extended
final class MathUtils {
    public static int square(int n) { return n * n; }
}
// class MyMath extends MathUtils { } ❌ compile error!

// final method — cannot be overridden
class Animal {
    public final void breathe() {
        System.out.println("Breathing...");
    }
    public void sound() { }  // can be overridden
}

class Dog extends Animal {
    // @Override breathe() ❌ compile error — final!
    @Override
    public void sound() { System.out.println("Woof!"); }  // ✅
}
Why is String final? String is final to ensure immutability and security. If String could be subclassed, someone could create a malicious String subclass that behaves differently — this would be a security risk since strings are used everywhere (file paths, network connections, class loading).

Multilevel Inheritance — Constructor Chain
class Animal {
    Animal() { System.out.println("Animal constructor"); }
    void eat()   { System.out.println("Eating"); }
}

class Dog extends Animal {
    Dog() {
        super();  // auto-added by Java if not written
        System.out.println("Dog constructor");
    }
    void bark()  { System.out.println("Barking"); }
}

class Puppy extends Dog {
    Puppy() {
        super();  // calls Dog() which calls Animal()
        System.out.println("Puppy constructor");
    }
    void play()  { System.out.println("Playing"); }
}

Puppy p = new Puppy();
// Output (parent constructors first!):
// Animal constructor
// Dog constructor
// Puppy constructor

p.eat();   // ✅ inherited from Animal
p.bark();  // ✅ inherited from Dog
p.play();  // ✅ own method
Constructor execution order in inheritance:
Always top to bottom — parent constructor runs first, then child. This ensures the parent is fully initialized before the child adds its own state.

Top Interview Questions & Model Answers
Q1. What is inheritance? What are its types in Java?
Inheritance is the mechanism where a child class acquires properties and behaviors of a parent class using the extends keyword. Types supported in Java: Single (A→B), Multilevel (A→B→C), Hierarchical (A→B, A→C). Multiple inheritance with classes is NOT supported due to the Diamond Problem — but it is supported through interfaces.
Q2. Why doesn't Java support multiple inheritance with classes?
Java avoids multiple class inheritance to prevent the Diamond Problem. If class C extends both A and B, and both have the same method, there is ambiguity about which version C should inherit. Java solves this by allowing multiple inheritance only through interfaces, where the implementing class must explicitly override any conflicting default methods.
Q3. What is method overriding? What are its rules?
Method overriding is when a subclass provides its own implementation of a method already defined in the parent class. Rules: same method name, same parameters, same or covariant return type, cannot reduce access modifier (public can't become protected), cannot override private/static/final methods, and it is resolved at runtime (dynamic polymorphism). The @Override annotation is recommended for compile-time verification.
Q4. What is the difference between method overloading and overriding?
Overloading is in the same class with different parameters — resolved at compile time (static polymorphism). Overriding is in parent-child classes with identical signatures — resolved at runtime (dynamic polymorphism). Overloading can have different return types; overriding must have same or covariant return type. You cannot override private or static methods.
Q5. What is the IS-A and HAS-A relationship?
IS-A is the inheritance relationship — Dog IS-A Animal (Dog extends Animal). It means the child is a specialized type of parent. HAS-A is the composition relationship — Car HAS-A Engine (Car contains an Engine object as a field). Composition is generally preferred over inheritance when the relationship is not truly IS-A, as it provides more flexibility and looser coupling.
Q6. Can you override a static method in Java?
No. Static methods cannot be overridden — they can only be hidden (method hiding). When you define a static method with the same signature in a subclass, it creates a new method that hides the parent's. The key difference: overriding is resolved at runtime based on object type; method hiding is resolved at compile time based on reference type. @Override annotation on a static method causes a compile error.
Q7. What is the order of constructor execution in inheritance?
Parent constructors always execute before child constructors. When you create a Puppy (which extends Dog which extends Animal), the order is: Animal() → Dog() → Puppy(). Java automatically inserts super() as the first line of every constructor if you don't write it. This top-down order ensures the parent is fully initialized before the child begins its own initialization.
