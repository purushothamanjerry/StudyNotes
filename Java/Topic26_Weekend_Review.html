Quick Revision — Foundation & Basics (T1–T8)

🔵 T1 — What is Java?

  • High-level, Object-Oriented, Platform-Independent language
  • Created by James Gosling at Sun Microsystems, 1995
  • WORA — Write Once, Run Anywhere via bytecode + JVM
  • JDK ⊃ JRE ⊃ JVM — JDK for development, JRE to run, JVM executes bytecode

🔵 T2 — How Java Runs

  • .java → javac → .class (bytecode) → JVM → machine code
  • JVM: ClassLoader → Bytecode Verifier → Interpreter + JIT Compiler
  • Heap: objects | Stack: local vars, references (per thread)
  • GC automatically frees unreferenced heap objects

🟢 T3 — How Data is Stored

  • Primitives (int, double, boolean) → stored on Stack directly
  • Objects → Heap; reference variable → Stack
  • String Pool: literals reused; new String() → new heap object
  • Java is always pass by value (for objects: copy of reference)
  • Integer cache: -128 to 127 cached → == true; above → false

🟢 T4 — Variables & Data Types

  • Local (stack, no default), Instance (heap, has default), Static (method area, shared)
  • Widening (auto, safe): byte→short→int→long→float→double
  • Narrowing (manual cast, may lose data): (int) 9.99 → 9 (truncates!)
  • var (Java 10) — local type inference only

🟢 T5 — Operators

  • 1 + 2 + "3" → "33" | "1" + 2 + 3 → "123" (left to right)
  • && short-circuits (skips right if left false) | & evaluates both
  • Post: x++ uses then increments | Pre: ++x increments then uses
  • Ternary: condition ? valueIfTrue : valueIfFalse

🟢 T6 — Conditional Statements

  • else-if: first match runs, rest skipped even if true
  • switch: missing break → fall-through to next cases
  • switch does NOT support: long, float, double, boolean
  • Java 14+ switch expression: uses ->, no fall-through, no break needed

🟢 T7 — Loops

  • while: condition checked first → may run 0 times
  • do-while: body runs first → always at least 1 time
  • for-each: no index, no modification, no reverse
  • break exits loop | continue skips current iteration
  • Labeled break outer → exits outer loop from inner

🟢 T8 — Methods

  • Overloading: same name, different params, same class, compile-time
  • Cannot overload by return type alone
  • Varargs: int... nums — must be last param, only one per method
  • Recursion: needs base case or → StackOverflowError
  • Static method cannot access instance variables

Quick Revision — OOP (T9–T16)

🟣 T9 — Class & Object

  • Class = blueprint | Object = instance (each has own state)
  • new: allocates heap + calls constructor + returns reference
  • Dog d2 = d1 → copies reference (both point to same object!)
  • Override toString() — called by println automatically
  • Override equals() + hashCode() together always!

🟣 T10 — Access Modifiers

  • Top-level class: only public or default (NOT private/protected)
  • Cannot REDUCE access when overriding

🟣 T11 — Constructor

  • No return type (not even void) | same name as class
  • Java provides default constructor only if you write NO constructor
  • this() — same class constructor | super() — parent constructor
  • Both must be FIRST statement | Cannot use both in same constructor
  • Private constructor → Singleton pattern

🟣 T12 — Encapsulation

  • Private fields + public getters/setters = encapsulation
  • Boolean getter uses is prefix: isActive()
  • Read-only: getter only | Write-only: setter only
  • JavaBean: private fields + getters/setters + no-arg constructor + Serializable

🟣 T13 — Inheritance

  • Java supports: Single, Multilevel, Hierarchical — NOT multiple class inheritance
  • Diamond Problem → why no multiple class inheritance
  • Constructor order: parent first, then child (always!)
  • Private fields NOT directly inherited | Constructors NOT inherited
  • Cannot override: private, static (hidden), final methods

🟣 T14 — Polymorphism

  • Compile-time: method overloading, resolved by compiler
  • Runtime: method overriding, resolved by JVM via dynamic dispatch
  • Upcasting: automatic, safe | Downcasting: explicit, can throw ClassCastException
  • Always use instanceof before downcasting!
  • Parent reference → only parent class methods accessible

🟣 T15 — Abstract Class

  • Cannot instantiate | Can have constructor (called via super())
  • Abstract method: no body, must override in concrete subclass
  • Cannot: abstract + final, abstract + static, abstract + private
  • Zero abstract methods still valid (to prevent instantiation)

🟣 T16 — Interface

  • Variables: implicitly public static final — constants only
  • Java 8: default methods + static methods (NOT inherited!)
  • Java 9: private methods (helpers for default methods)
  • Functional interface: exactly ONE abstract method → can use lambda
  • Marker interface: no methods (Serializable, Cloneable)
  • Diamond conflict: implementing class MUST override conflicting default method

Quick Revision — Core Java (T17–T21)

🟡 T17 — Non-Access Modifiers

  • static: belongs to class, shared, no this
  • final: variable=constant, method=no override, class=no extend
  • final reference: reference fixed, but object CAN be modified
  • synchronized: one thread at a time (methods/blocks only)
  • volatile: visibility only, NOT atomicity (count++ still unsafe!)
  • transient: skip during serialization → default value after deserialization

🟡 T18 — String

  • Immutable, final class, stored in String Pool
  • ==: references | .equals(): content — always use equals()!
  • Compile-time concat in pool | Runtime concat → new heap object
  • new String("Hello") → up to 2 objects created
  • StringBuilder: mutable, fast, NOT thread-safe (single thread)
  • StringBuffer: mutable, synchronized, thread-safe (slower)

🟡 T19 — Arrays

  • arr.length — field, NOT method (no parentheses!)
  • Valid indices: 0 to length-1 → else ArrayIndexOutOfBoundsException
  • Arrays.sort() modifies original | Arrays.binarySearch() needs sorted array
  • arr2 = arr1 copies reference, not array | Use Arrays.copyOf()
  • Array passed to method → changes affect original!

🟡 T20 — Exception Handling

  • Checked: compiler forces handle/declare (IOException, SQLException)
  • Unchecked: RuntimeException subclasses (NPE, AIOOBE, CCE)
  • Error: JVM-level (OutOfMemoryError, StackOverflowError) — don't catch
  • finally: ALWAYS runs except on System.exit()
  • throw: inside method body | throws: method signature
  • try-with-resources: auto-closes AutoCloseable resources
  • Specific catch before general (child before parent)!

🟡 T21 — Generics

  • Type safety at compile time, eliminates casting
  • Cannot use primitives (List<int> ❌ → use List<Integer>)
  • <T extends Number>: T must be Number or subclass
  • ? extends T: read (producer) | ? super T: write (consumer) — PECS rule
  • Type erasure: generics removed at runtime — List<String> = List at runtime
  • Cannot create generic arrays: new T[10] ❌

  • ConcurrentModificationException: don't modify collection during for-each → use Iterator.remove() or removeIf()
  • Comparable: natural order (compareTo) | Comparator: custom order (compare)

🟠 T23 — Java 8+ Features

  • Lambda: (a, b) -> a + b — implements functional interface
  • Stream: Source → filter/map/sorted → collect/forEach/count (lazy pipeline)
  • Optional: avoids null — orElse(), orElseThrow(), map(), ifPresent()
  • Method ref: String::toUpperCase, System.out::println, ArrayList::new
  • LocalDate, LocalTime, LocalDateTime — immutable, thread-safe date/time
  • Records (Java 16): auto-generates constructor, getters, equals, hashCode, toString

🟠 T24 — Multithreading

  • start() → new thread | run() → same thread (no new thread!)
  • Thread states: NEW → RUNNABLE → RUNNING → BLOCKED/WAITING → TERMINATED
  • Race condition: use synchronized or AtomicInteger
  • Deadlock: threads waiting for each other's locks → fix with lock ordering
  • volatile: visibility | synchronized: visibility + atomicity
  • ExecutorService: thread pool, reuse threads, submit Callable → Future
  • wait() releases lock | sleep() does NOT release lock

🔴 T25 — JDBC

  • Steps: Load driver → Connection → Statement → Execute → ResultSet → Close
  • Statement: static SQL, SQL injection risk | PreparedStatement: parameterized (?) → safe + fast
  • executeQuery() → ResultSet | executeUpdate() → int (rows affected)
  • Transaction: setAutoCommit(false) → commit() or rollback()
  • Connection Pool (HikariCP): reuse connections, much faster than creating new each time
  • Always close in try-with-resources to avoid resource leaks!

🧠 Top Tricky Interview Questions — Must Know!
Q: What is the output? System.out.println(1 + 2 + "3"); and System.out.println("1" + 2 + 3);
"33" and "123" — evaluated left to right; once a String is encountered, + becomes concatenation.
Q: Is Java pass by value or pass by reference?
Always pass by value. For objects, the value passed is the copy of the reference — so field changes in method are visible, but reassigning the reference is not.
Q: What is the output? Integer a = 127; Integer b = 127; System.out.println(a == b);
true — Java caches Integer objects from -128 to 127. Same cached object. For 128+, new objects are created → false.
Q: Can abstract class have constructor? Can you have abstract class with no abstract methods?
Yes to both. Constructor called via super(). Zero abstract methods is valid — just prevents instantiation.
Q: Can you override a static method?
No — static methods can be hidden (method hiding), not overridden. Resolved at compile time by reference type, not runtime object type.
Q: What is the difference between == and equals() for String?
== compares references. .equals() compares content. Always use .equals() for String comparison. Use "literal".equals(variable) to avoid NPE.
Q: What is the difference between final variable, finally block, and finalize() method?
final variable: constant, cannot reassign. finally block: always executes after try-catch (except System.exit). finalize(): deprecated method called before GC (avoid using).
Q: Why must you override hashCode() when you override equals()?
Contract: if a.equals(b) → a.hashCode() == b.hashCode(). HashMap/HashSet use hashCode first to find bucket, then equals to find exact entry. Breaking this causes collections to malfunction.
Q: What is the difference between Comparable and Comparator?
Comparable (compareTo): natural order built into the class itself — one default sort. Comparator (compare): external, multiple sort orders without modifying the class.
Q: What is the difference between ArrayList and LinkedList?
ArrayList: O(1) random access, O(n) insert/delete at middle. LinkedList: O(n) random access, O(1) insert/delete at middle. Use ArrayList for reads, LinkedList for frequent middle modifications.
Q: What is the difference between wait() and sleep()?
wait() is an Object method: releases the lock and waits for notify(). Must be inside synchronized block. sleep() is a Thread method: pauses but does NOT release locks. wait() for thread communication; sleep() for pausing.
Q: What is type erasure in Generics?
Java removes all generic type info from bytecode at compile time. List<String> and List<Integer> are both just List at runtime. Can't create generic arrays or use instanceof with generic types.

💡 Interview Tips & Strategy

How to Answer Technical Questions

  • Definition first — give a clear one-line definition
  • Then explain how — implementation or mechanism
  • Give an example — concrete code or real-world analogy
  • Mention when to use — use cases and limitations
  • Common mistakes — shows depth of knowledge

Most Frequently Asked Topics (prepare deeply)

  • ✅ OOP Pillars with examples
  • ✅ Abstract class vs Interface (with Java 8+ changes)
  • ✅ HashMap internal working (hashCode, equals, collision)
  • ✅ String immutability + String Pool
  • ✅ Checked vs Unchecked exceptions
  • ✅ synchronized vs volatile
  • ✅ Java 8 features: Lambda, Stream, Optional
  • ✅ ArrayList vs LinkedList
  • ✅ this vs super
  • ✅ Method overloading vs overriding

Common Mistakes to Avoid

  • Never say "Java is pass by reference" — it's always pass by value
  • Don't confuse method overloading (same class) with overriding (parent-child)
  • Don't forget: interface static methods are NOT inherited
  • Don't forget: if you override equals(), you MUST override hashCode()
  • Don't say "interface is 100% abstract" — Java 8+ has default/static methods
  • Don't call t.run() thinking it creates a new thread — always t.start()
