Overview — 7 Non-Access Modifiers
Non-access modifiers control the behavior of classes, methods, and variables — NOT their visibility. They don't say who can access something; they say how it behaves.
static

Belongs to class, not object. Shared across all instances.

final

Cannot be changed/overridden/extended. Immutability.

abstract

No implementation. Must be overridden. Partial blueprint.

synchronized

Only one thread at a time. Thread safety.

volatile

Always read from main memory. Visibility in threads.

transient

Skip during serialization. Don't save this field.

2
static
static means the member belongs to the class, not to any specific object. Only ONE copy exists regardless of how many objects are created.

static variable

class Counter {
    static int count = 0;   // ONE copy shared by ALL objects
    int id;                   // each object has its OWN id

    Counter() {
        count++;
        id = count;
    }
}
Counter c1 = new Counter(); // count=1, c1.id=1
Counter c2 = new Counter(); // count=2, c2.id=2
System.out.println(Counter.count); // 2 — access via class name

static method

class MathUtils {
    public static int square(int n) { return n * n; }
    public static int cube(int n)   { return n * n * n; }
}
// No object needed!
System.out.println(MathUtils.square(4));  // 16
System.out.println(MathUtils.cube(3));    // 27

static block

class Config {
    static String dbUrl;

    // Static block — runs ONCE when class is first loaded
    static {
        System.out.println("Static block executing...");
        dbUrl = "jdbc:mysql://localhost:3306/mydb";
    }

    public static String getUrl() { return dbUrl; }
}
// Static block runs before any object creation or method call
System.out.println(Config.getUrl());
// Output: Static block executing... → jdbc:mysql://...

static nested class

class Outer {
    static int x = 10;

    static class Inner {       // static nested class
        void show() { System.out.println(x); }  // can access static members
    }
}
Outer.Inner obj = new Outer.Inner();  // no Outer object needed
obj.show();  // 10
Where usedEffect
VariableOne copy shared across all instances. Stored in Method Area.
MethodBelongs to class. Can't access instance members or this.
BlockRuns once when class is loaded. Used for initialization.
Nested classInner class that doesn't need outer class instance.
Import (import static)Import static members directly: import static Math.PI
3
final
final means cannot be changed. Applied to variables (value), methods (override), or classes (extend).

final variable — constant

final int MAX = 100;
// MAX = 200; ❌ compile error — cannot reassign

public static final double PI = 3.14159;  // class constant

// final reference — reference cannot change, BUT object CAN be modified
final StringBuilder sb = new StringBuilder("Hello");
sb.append(" World");  // ✅ object modified — OK
// sb = new StringBuilder(); ❌ reference cannot point to new object

// Blank final — must be initialized in constructor
class Person {
    final String name;   // blank final
    Person(String name) { this.name = name; }  // must init here
}

final method — cannot override

class Animal {
    public final void breathe() {
        System.out.println("Breathing...");
    }
}
class Dog extends Animal {
    // public void breathe() { } ❌ compile error — cannot override final
}

final class — cannot extend

final class ImmutablePoint {
    final int x, y;
    ImmutablePoint(int x, int y) { this.x = x; this.y = y; }
}
// class SubPoint extends ImmutablePoint { } ❌ compile error
// String, Integer, Double — all final classes in Java
⚠️ Tricky: final reference vs final object
final List<String> list = new ArrayList<>();
list.add("hello"); ✅ — object can be modified
list = new ArrayList<>(); ❌ — reference cannot change

abstract
Applied to classes (cannot instantiate, partial blueprint) and methods (no body, must be overridden). Already covered deeply in Topic 15 — quick summary here.
Applied toMeaning
abstract classCannot instantiate. Can have both abstract and concrete methods.
abstract methodNo body. Subclass MUST override it. Only inside abstract class.
⚠️ Cannot combine:
abstract final → ❌ final prevents overriding, abstract requires it
abstract static → ❌ static belongs to class, abstract needs object context
abstract private → ❌ private can't be seen by subclass to override

synchronized
synchronized ensures that only one thread at a time can execute a method or block. Used for thread safety when multiple threads access shared data. 
class Counter {
    private int count = 0;

    // Without synchronized — RACE CONDITION possible!
    public void increment() {
        count++;   // NOT atomic — read, increment, write (3 steps)
    }

    // With synchronized — THREAD SAFE
    public synchronized void safeIncrement() {
        count++;   // only ONE thread runs this at a time
    }

    // Synchronized block — more fine-grained control
    public void blockIncrement() {
        synchronized(this) {
            count++;
        }
    }

    public int getCount() { return count; }
}

// Static synchronized — locks on class object
public static synchronized void staticMethod() {
    // only one thread runs this across ALL instances
}
How it works: Every Java object has an intrinsic lock (monitor). When a thread calls a synchronized method, it acquires the lock. Other threads trying to call any synchronized method on the same object must wait until the lock is released.
⚠️ synchronized can only be applied to methods and blocks — NOT to variables or classes.

volatile
volatile ensures that a variable is always read from and written to main memory, not from a thread's local CPU cache. Solves visibility issues in multi-threaded programs. 
class SharedFlag {
    // Without volatile — thread may read stale value from CPU cache!
    private boolean running = true;

    // With volatile — always reads from main memory
    private volatile boolean flag = true;

    public void stop()    { flag = false; }  // Thread 1 writes
    public boolean isOn() { return flag; }  // Thread 2 reads — always fresh!
}

// Classic volatile use case — stop flag
class Worker implements Runnable {
    private volatile boolean active = true;

    public void run() {
        while (active) {
            // do work...
        }
    }

    public void stop() { active = false; }
}

volatile vs synchronized

volatile — visibility only, no atomicity

Lightweight, no locking

For simple flags/booleans

synchronized

Both visibility AND atomicity

Heavier, uses locking

For compound operations (count++)

⚠️ volatile does NOT make compound operations atomic!
volatile int count; count++; is still NOT thread-safe!
count++ is read + increment + write — 3 operations. Use synchronized or AtomicInteger instead.
7
transient
transient tells Java to skip this field during serialization. The field is NOT saved when the object is written to a file or network stream. 
import java.io.Serializable;

class User implements Serializable {
    String  username;         // ✅ saved during serialization
    String  email;            // ✅ saved
    transient String password; // ❌ SKIPPED — not saved for security
    transient int    sessionId; // ❌ SKIPPED — not meaningful after restart
}

// After deserialization:
// username → "alice"    ✅ restored
// email    → "a@b.com"  ✅ restored
// password → null       ← transient, default value
// sessionId → 0         ← transient, default value

When to use transient:

  • Security — passwords, PINs, API keys should not be saved
  • Calculated fields — can be recalculated, no need to store
  • Session data — temporary data not meaningful after restart
  • Non-serializable fields — if a field's type doesn't implement Serializable
8
native
native means the method is implemented in native code (C or C++) outside Java. Used via JNI (Java Native Interface). The method has no body in Java — the implementation is in a native library. 
class NativeExample {
    // Declared in Java — implemented in C/C++
    public native void nativeMethod();
    public native int  nativeAdd(int a, int b);

    static {
        System.loadLibrary("nativelib");  // load native library
    }
}

// Real examples in Java itself:
// Object.hashCode() → native
// System.currentTimeMillis() → native
// Thread.sleep() → native
// Math.sqrt() → native (delegates to CPU instruction)
When to use native: Performance-critical operations, hardware access, using existing C/C++ libraries, system-level operations (file I/O, networking at low level).

💡 Quick memory rule for invalid combinations:
If two modifiers contradict each other's purpose, they cannot be combined.
abstract + final → want to override vs. prevent override → contradiction ❌
abstract + private → need visibility to override vs. hidden → contradiction ❌
abstract + static → instance override vs. class member → contradiction ❌

Top Interview Questions & Model Answers
Q1. What is the difference between access modifiers and non-access modifiers?
Access modifiers (public, private, protected, default) control visibility — who can access a member. Non-access modifiers (static, final, abstract, synchronized, volatile, transient, native) control behavior — how a member works. For example, static says it belongs to the class; final says it can't be changed; synchronized says only one thread can access it at a time.
Q2. What is a static block? When does it run?
A static block is a block of code prefixed with static { }. It runs exactly once when the class is first loaded into memory by the ClassLoader — before any object creation or static method call. It is used to initialize static variables, load configuration, or perform one-time setup tasks like loading a database driver or a native library.
Q3. What is the difference between final variable and effectively final?
A final variable is explicitly declared with the final keyword — it cannot be reassigned. An effectively final variable (Java 8+) is NOT declared final but is never reassigned after initialization. Lambda expressions and anonymous classes can access effectively final local variables — the compiler treats them as final even without the keyword.
Q4. What is the difference between volatile and synchronized?
volatile ensures visibility — changes to the variable are immediately visible to all threads by always reading/writing from main memory. But it does NOT ensure atomicity for compound operations. synchronized ensures both visibility AND atomicity — only one thread executes the block at a time, and changes are flushed to main memory when the lock is released. Use volatile for simple flags; use synchronized for compound operations like increment.
Q5. What is transient? When would you use it?
The transient modifier tells the serialization mechanism to skip this field — it is not saved when the object is serialized to a file or network stream. Use it for: security-sensitive fields (passwords, keys), fields that can be recalculated, session-specific data, or fields whose type doesn't implement Serializable. After deserialization, transient fields have their default values (null, 0, false).
Q6. Why can't abstract and final be used together?
They are fundamentally contradictory. abstract means the method or class is incomplete and must be overridden/extended by a subclass. final means the method or class cannot be overridden/extended. Combining them would mean "you must override this but you cannot override this" — the compiler correctly rejects this contradiction.
Q7. Can you use static with a local variable inside a method?
No. Local variables inside methods cannot be declared static. The static modifier is only valid for class-level members (fields, methods, nested classes, blocks). Local variables are created on the stack each time the method is called and destroyed when it returns — they have no class-level lifetime, which is what static implies.
