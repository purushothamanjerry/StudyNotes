What is Multithreading?
Thread = the smallest unit of execution within a process. Multithreading = running multiple threads concurrently within the same program to improve performance and responsiveness.

Process vs Thread

Process — independent program with own memory space

Thread — lightweight unit inside a process, shares memory

Threads of same process share Heap, static data, code

Each thread has its own Stack and PC Register

Why Multithreading?

  • Better CPU utilization
  • Responsive UI (UI thread + background tasks)
  • Parallel processing of large data
  • Handle multiple client requests (servers)

Creating Threads — 3 Ways

Way 1 — Extend Thread class

class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("Thread: " + Thread.currentThread().getName());
    }
}

MyThread t = new MyThread();
t.start();  // ✅ starts new thread — calls run() in new thread
// t.run(); ❌ runs in SAME thread — NOT a new thread!

Way 2 — Implement Runnable interface (preferred)

class MyTask implements Runnable {
    @Override
    public void run() {
        System.out.println("Runnable running in: " + Thread.currentThread().getName());
    }
}

Thread t = new Thread(new MyTask());
t.start();

// With lambda (Java 8) — most concise!
Thread t2 = new Thread(() -> System.out.println("Lambda thread!"));
t2.start();

Way 3 — Implement Callable (returns result)

import java.util.concurrent.*;

Callable<Integer> task = () -> {
    Thread.sleep(1000);
    return 42;   // can return value and throw checked exceptions
};

ExecutorService executor = Executors.newSingleThreadExecutor();
Future<Integer>  future   = executor.submit(task);
Integer         result   = future.get();  // blocks until done
System.out.println(result);   // 42
executor.shutdown();
FeatureThreadRunnableCallable
Return value❌ No❌ No✅ Yes (via Future)
Checked exception❌ No❌ No✅ Yes
Extend other class?❌ No (already extends)✅ Yes✅ Yes

⚠️ Always call start() not run()!
t.start() → JVM creates a new thread and calls run() in it ✅
t.run() → runs in the CURRENT thread — no new thread created! ❌

Thread Methods
Thread t = new Thread(() -> System.out.println("Hello"), "MyThread");

// Start — begins execution in new thread
t.start();

// getName / setName
System.out.println(t.getName());   // "MyThread"
t.setName("WorkerThread");

// sleep — pause current thread (throws InterruptedException)
try {
    Thread.sleep(1000);  // sleep 1 second
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
}

// join — wait for thread to finish
t.start();
t.join();  // current thread waits until t completes
System.out.println("t is done");

// join with timeout
t.join(2000);  // wait max 2 seconds

// priority (1=MIN, 5=NORMAL, 10=MAX)
t.setPriority(Thread.MAX_PRIORITY);  // hint to scheduler
t.getPriority();

// isAlive — check if thread is running
System.out.println(t.isAlive());  // true if between start() and end

// Daemon thread — background thread, dies when main thread ends
t.setDaemon(true);   // must call BEFORE start()
t.start();

// currentThread — get reference to running thread
Thread me = Thread.currentThread();
System.out.println(me.getName());  // "main" (in main thread)

// yield — hint to scheduler to let others run
Thread.yield();  // no guarantee

// interrupt — signal thread to stop
t.interrupt();  // sets interrupt flag — thread must check it

Synchronization — Race Condition Fix
Race condition — when multiple threads access shared data simultaneously and the outcome depends on thread scheduling. synchronized prevents this by allowing only one thread at a time. 
// Race condition — WITHOUT synchronization
class Counter {
    int count = 0;
    void increment() { count++; }  // NOT atomic! read-add-write
}
// 2 threads each calling increment() 1000 times
// Expected: 2000, Actual: could be 1500-1999 (race condition!)

// FIX 1 — synchronized method
class SafeCounter {
    int count = 0;
    synchronized void increment() {
        count++;  // only ONE thread at a time
    }
    synchronized int getCount() { return count; }
}

// FIX 2 — synchronized block (finer control)
class SafeCounter2 {
    int count = 0;
    void increment() {
        synchronized(this) {  // lock on 'this'
            count++;
        }
    }
}

// wait() and notify() — thread communication
class SharedResource {
    boolean ready = false;

    synchronized void produce() throws InterruptedException {
        ready = true;
        notify();  // wake up waiting thread
    }

    synchronized void consume() throws InterruptedException {
        while (!ready) {
            wait();  // release lock and wait
        }
        System.out.println("Consumed!");
    }
}
How synchronized works:
Every Java object has an intrinsic lock (monitor). When a thread enters a synchronized method/block, it acquires the lock. Other threads trying to enter any synchronized method on the SAME object must wait. When the thread exits, it releases the lock.
⚠️ wait() and notify() must be called inside synchronized block!
Otherwise → IllegalMonitorStateException at runtime.

volatile keyword
class StopFlag {
    // Without volatile — thread may read stale value from CPU cache!
    private volatile boolean running = true;

    public void run() {
        while (running) {
            // do work
        }
        System.out.println("Stopped!");
    }

    public void stop() {
        running = false;  // guaranteed visible to all threads
    }
}

// volatile ensures visibility — NOT atomicity
volatile int count = 0;
count++;  // STILL NOT thread-safe! (read-increment-write)
// Use AtomicInteger for atomic increment

Deadlock
Deadlock = two or more threads are blocked forever, each waiting for a lock held by the other. The program freezes. 
// Classic deadlock example
Object lockA = new Object();
Object lockB = new Object();

// Thread 1: acquires lockA, then tries lockB
Thread t1 = new Thread(() -> {
    synchronized(lockA) {
        System.out.println("T1: got lockA");
        synchronized(lockB) {  // waiting for lockB
            System.out.println("T1: got lockB");
        }
    }
});

// Thread 2: acquires lockB, then tries lockA
Thread t2 = new Thread(() -> {
    synchronized(lockB) {
        System.out.println("T2: got lockB");
        synchronized(lockA) {  // waiting for lockA ← DEADLOCK!
            System.out.println("T2: got lockA");
        }
    }
});
// T1 holds lockA, waits for lockB
// T2 holds lockB, waits for lockA → DEADLOCK!

// FIX — always acquire locks in the SAME order
// Both threads: acquire lockA first, then lockB → no deadlock

How to prevent deadlock

  • Lock ordering — always acquire locks in the same order across all threads
  • Lock timeout — use tryLock(timeout) from ReentrantLock
  • Avoid nested locks — minimize synchronized blocks
  • Use higher-level concurrency utilities — ExecutorService, Concurrent collections

ExecutorService (java.util.concurrent)
ExecutorService = a higher-level API for managing thread pools. Instead of creating threads manually, submit tasks to a pool — threads are reused, managed efficiently. 
import java.util.concurrent.*;

// Fixed thread pool — reuse N threads
ExecutorService pool = Executors.newFixedThreadPool(4);

// Submit Runnable tasks
for (int i = 0; i < 10; i++) {
    final int taskId = i;
    pool.submit(() -> {
        System.out.println("Task " + taskId + " on " + Thread.currentThread().getName());
    });
}

// Submit Callable — get result via Future
Future<Integer> future = pool.submit(() -> {
    Thread.sleep(500);
    return 42;
});
System.out.println("Result: " + future.get());  // blocks until done
future.isDone();    // check without blocking
future.cancel(true); // try to cancel

// Always shutdown!
pool.shutdown();               // no new tasks, finish current
pool.shutdownNow();            // interrupt running tasks
pool.awaitTermination(5, TimeUnit.SECONDS);

// Other executor types
Executors.newSingleThreadExecutor();  // 1 thread, sequential
Executors.newCachedThreadPool();       // grows as needed, reuses idle
Executors.newScheduledThreadPool(2);  // schedule tasks with delay

// ScheduledExecutorService
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(1);
scheduler.schedule(() -> System.out.println("Delayed!"), 3, TimeUnit.SECONDS);
scheduler.scheduleAtFixedRate(() -> System.out.println("Every 2s"), 0, 2, TimeUnit.SECONDS);

Atomic Classes (java.util.concurrent.atomic)
Atomic classes provide lock-free thread-safe operations on single variables. Faster than synchronized for simple operations. 
import java.util.concurrent.atomic.*;

// AtomicInteger — thread-safe int operations
AtomicInteger counter = new AtomicInteger(0);

counter.incrementAndGet();  // ++counter (atomic)
counter.getAndIncrement();  // counter++ (atomic)
counter.addAndGet(5);       // counter += 5 (atomic)
counter.get();              // read current value
counter.set(10);           // set value
counter.compareAndSet(10, 20); // if value==10, set to 20 (CAS)

// Other atomic types
AtomicLong      atomicLong    = new AtomicLong(0);
AtomicBoolean   atomicBool    = new AtomicBoolean(false);
AtomicReference<String> ref = new AtomicReference<>("Hello");

// Safe counter using AtomicInteger — NO synchronized needed
AtomicInteger safe = new AtomicInteger(0);
// 2 threads each increment 1000 times → always exactly 2000!
💡 AtomicInteger vs synchronized:
AtomicInteger — lock-free, uses CAS (Compare-And-Swap) CPU instruction. Faster for single variables.
synchronized — uses OS-level locking. Better when protecting multiple variables together.

BlockingQueue<String> queue = new LinkedBlockingQueue<>(10);

        queue.put("item1");  // blocks if full
        queue.put("item2");
    } catch (InterruptedException e) { }

        System.out.println("Consumed: " + item);

Top Interview Questions & Model Answers
Q1. What is the difference between Thread and Runnable?
Thread is a class; extending it ties your class to Thread, preventing extension of other classes. Runnable is a functional interface — implementing it allows your class to extend another class too. Runnable also separates the task (what to do) from the thread execution mechanism (how to run), following the Single Responsibility Principle. Runnable can also be used with ExecutorService, not just Thread.
Q2. What is the difference between start() and run()?
start() creates a new OS thread and then calls run() in that new thread — true multithreading. run() called directly just executes the run() method in the calling thread — no new thread is created. This is the most common multithreading mistake: calling run() instead of start().
Q3. What is a race condition? How do you fix it?
A race condition occurs when multiple threads access and modify shared data concurrently, and the final result depends on the timing/order of execution. Fix using: (1) synchronized methods or blocks — only one thread executes at a time. (2) AtomicInteger/AtomicLong for simple counter operations. (3) ReentrantLock for more control. (4) Immutable objects — can't be modified, always thread-safe.
Q4. What is deadlock? How do you prevent it?
Deadlock is when two or more threads are blocked forever, each waiting for a lock held by the other. Prevention strategies: (1) Lock ordering — always acquire locks in the same order across threads. (2) Lock timeout — use tryLock(timeout) instead of synchronized. (3) Avoid nested locks where possible. (4) Use high-level concurrency utilities (ExecutorService) instead of raw synchronized blocks.
Q5. What is the difference between synchronized and volatile?
volatile ensures visibility — changes to a variable are immediately visible to all threads by reading from main memory, not CPU cache. It does NOT ensure atomicity for compound operations. synchronized ensures both visibility AND atomicity — only one thread executes the block at a time. Use volatile for simple flags (boolean, read/write without compound operations); use synchronized for compound operations like increment or check-then-act sequences.
Q6. What is ExecutorService? Why is it better than creating threads directly?
ExecutorService is a higher-level thread management API. Benefits over manual threads: (1) Thread pooling — reuses threads instead of creating/destroying per task, saving overhead. (2) Task queue — manages work queue automatically. (3) Return values — Callable + Future allow getting results from threads. (4) Better control — shutdown, awaitTermination, scheduled execution. (5) Prevents thread explosion — fixed pool size limits concurrent threads.
Q7. What is the difference between wait() and sleep()?
wait() is an Object method called on a synchronized object — it releases the lock and waits until notify() is called. It must be called inside a synchronized block. sleep() is a Thread static method — it pauses the current thread for a specified time but does NOT release any locks it holds. wait() is for inter-thread communication; sleep() is for pausing execution.
